# v0.20 final security review — 2026-07-21

Task 19 of the v0.20 "Final Sweep" release. Re-runs the v0.18.0 per-user-
isolation + input-trust lens (`docs/security/2026-07-20-isolation-audit.md`)
over the five NEW surfaces this release added: `/metrics`,
`/api/internal/backup-complete`, outbound webhook dispatch, the account
import route, and the sync-jobs admin panel. The v0.18.0 audit already
covered the full pre-existing surface (101/101 clean); this review does not
repeat that ground, only what's new since.

**Result: no gap found.** All five surfaces below are `✅ confirmed` with
in-file evidence. Nothing was changed as a result of this review — the report
below is a verification pass, not a remediation.

## Methodology

Same as the v0.18.0 audit: for each surface, read the handler end to end,
trace every id that reaches a `WHERE`/`.values()`/fetch-target back to its
source, and confirm that source is `session.user.id` / `requireUser()` /
`requireOwner()` / a constant-time shared-secret compare — never a
client-supplied `userId` field. Confirmed via full reads plus targeted greps
(reproduced below so the claims are checkable, not just asserted).

## 1. `/api/metrics` — Prometheus scrape endpoint (Task 6)

**File:** `src/app/api/metrics/route.ts`

**Token gate**, checked before any query runs:

```
39:  const token = process.env.METRICS_TOKEN;
40:  if (!token) {
41:    return new NextResponse(null, { status: 404 });   // unset → 404, don't reveal the endpoint exists
...
44:  const provided = req.headers.get("authorization")?.replace(/^Bearer /, "");
47:  if (!provided || !secretsMatch(provided, token)) {
48:    return NextResponse.json({ error: "unauthorized" }, { status: 401 });
```

**Timing-safe compare** — `secretsMatch()` hashes both sides with SHA-256
before `timingSafeEqual`, so lengths always match and the comparison itself
is constant-time (same pattern as `/api/internal/backup-complete` and
`/api/cron`'s `CRON_SECRET`, both already audited clean in v0.18.0):

```
src/app/api/metrics/route.ts:1:   import { createHash, timingSafeEqual } from "node:crypto";
src/app/api/metrics/route.ts:8-11: function secretsMatch(provided, expected) {
                                      const a = createHash("sha256").update(provided).digest();
                                      const b = createHash("sha256").update(expected).digest();
                                      return timingSafeEqual(a, b);
                                    }
```

**No per-user data leak.** `grep -n "userId" src/app/api/metrics/route.ts`
returns zero hits. The six emitted series
(`recover_sync_staleness_seconds`, `recover_sync_jobs_pending`,
`recover_sync_jobs_running`, `recover_sync_jobs_failed_total`,
`recover_backup_age_seconds`, `recover_push_subscriptions`) are all
instance-wide aggregates sourced from `getOpsSnapshot()`
(`src/lib/ops-metrics.ts`), which itself has zero `userId` references
(`grep -n "userId" src/lib/ops-metrics.ts` → no hits) — every query in it is
a bare `count()`/`findFirst()` with no `userId` filter and no `userId` (or
any other per-user identifier) in the returned shape:

```ts
export interface OpsSnapshot {
  lastSyncAgeS: number | null;
  jobsPending: number;
  jobsRunning: number;
  jobsFailed: number;
  backupAgeS: number | null;
  pushSubscriptions: number;
}
```

None of these six fields is labelled by, or derivable to, a specific user —
whoever holds `METRICS_TOKEN` learns instance-wide health signals only
(sync staleness across all users, job queue depth, backup freshness, total
push-subscription count), never which user's sync is stale or whose job
failed. ✅ confirmed — token-gated, timing-safe, no per-user leak.

## 2. `/api/internal/backup-complete` (Task 6)

**File:** `src/app/api/internal/backup-complete/route.ts`

Shared-secret gated (`BACKUP_NOTIFY_SECRET`), same `secretsMatch()`
timing-safe helper as `/metrics`, checked before any write:

```
26:  const secret = process.env.BACKUP_NOTIFY_SECRET;
27:  const provided = req.headers.get("authorization")?.replace(/^Bearer /, "");
28:  if (!secret || !provided || !secretsMatch(provided, secret)) {
29:    return NextResponse.json({ error: "unauthorized" }, { status: 401 });
```

An unauthenticated caller who guesses/brute-forces this would get exactly
one capability: writing the current epoch-seconds timestamp into
`app_config.backup_last_success_at` and triggering a `backup_completed`
webhook broadcast (a harmless, idempotent, non-secret operation — worst case
is a false "backup just ran" freshness signal, not data disclosure or
corruption). `secret` unset → the `!secret` check fails closed. ✅ confirmed
— nothing an unauthenticated caller could meaningfully abuse.

## 3. Webhook dispatch (Task 7)

**File:** `src/lib/webhooks/dispatch.ts`

**Per-user dispatch never crosses `userId`.** `dispatchWebhook(userId, ...)`
queries strictly `eq(schema.webhookSubscriptions.userId, userId)` before
ever building a delivery:

```
166: export async function dispatchWebhook(
167:   userId: string,
...
172:   const subs = await db.query.webhookSubscriptions.findMany({
173:     where: and(
174:       eq(schema.webhookSubscriptions.userId, userId),
175:       eq(schema.webhookSubscriptions.active, true)
176:     ),
```

Confirmed by an existing regression test,
`src/lib/webhooks/dispatch.test.ts:194` — `"never dispatches to another
user's subscriptions"` — inserts subscriptions for `USER_A` and `USER_B`,
calls `dispatchWebhook(USER_A, ...)`, and asserts only `USER_A`'s URL was
ever fetched.

Both call sites pass an authenticated id, never a request-body value:
`src/lib/metrics.ts:193/202` calls `dispatchWebhook(userId, ...)` where
`userId` is the loop variable driving the scheduler's own per-user metrics
computation (not client input).

The one intentionally unscoped function, `broadcastWebhook()`, is documented
in-file as deliberately instance-wide — it exists only for the single
`backup_completed` event (an instance-level fact, not per-user data) and is
called from exactly one place, the shared-secret-gated
`/api/internal/backup-complete` route. Not a cross-user leak: there is no
per-user data in a "the backup ran" notification.

**SSRF trust model — explicit, not an oversight.** The dispatch file's own
header comment states the model directly:

```
src/lib/webhooks/dispatch.ts:6-14:
 * Trust model: the target URL is chosen by the subscription's own owner —
 * an invited user, trusted the same as everywhere else in this app (v0.18
 * threat model: self-hosted, invite-only, ~10 friends, single owner, behind
 * a Cloudflare tunnel). No SSRF / private-IP blocking is applied here on
 * purpose — a self-hoster pointing this at a Home-Assistant instance on
 * their own LAN is the primary use case, not an attack to defend against.
 * This is a deliberate, documented trust-boundary decision, not an
 * oversight.
```

This review confirms that framing is accurate given the rest of the app's
threat model (v0.18.0's audit and this app's `join/[code]/actions.ts`
confirm accounts are invite-only, never open signup — see the v0.18.0
isolation audit's route-handler table). A user who can create a webhook
subscription is, by construction, an invited/trusted operator of this
specific instance, not an anonymous internet caller — so "an operator can
point a webhook at an internal address" is an accepted, documented risk for
a self-hosted invite-only tool, not an open-signup SSRF hole. **Recorded
here as confirmed-accepted, not re-litigated** — no change proposed.

**Fetch timeout confirmed.** Every attempt uses a fresh
`AbortSignal.timeout(fetchTimeoutMs)` (default `FETCH_TIMEOUT_MS = 8_000`),
added in Task 7 per the in-file comment at `dispatch.ts:53-62`, so a target
that accepts a TCP connection but never responds cannot hang the scheduler's
sequential tick loop. ✅ confirmed on all three points (userId scoping,
SSRF model, timeout).

## 4. Import route (Task 10) — most important check

**File:** `src/app/api/import-account/route.ts`

**Confirmed: writes ONLY to `session.user.id`, never a caller-supplied
target.** The session is resolved first and gates the whole handler:

```
68:  const session = await auth.api.getSession({ headers: await headers() });
69:  if (!session)
70:    return NextResponse.json({ error: "unauthorized" }, { status: 401 });
```

The only call into the write path passes `session.user.id` directly — the
uploaded `body` (the untrusted, caller-controlled part of the request) is
passed as the _data_ argument, never as the _target-user_ argument:

```
107:    await importUserData(db, session.user.id, body);
```

`grep -n "importUserData(" src` (repo-wide, excluding tests) returns exactly
one call site — this one — confirming there is no second, less-audited path
into the import writer. `looksLikeUserExport()` (the body shape-check just
above) validates `body.version` and that the known array fields are arrays;
it does not read or trust any `userId`/`ownerId`/`targetUserId`-shaped field
because **no such field exists in `UserExport`** — the export/import schema
has no user-identity field at all, so there is structurally nothing in the
request body an attacker could set to redirect the import. This matches the
route's own header comment:

```
53-57:  * Restore a previously-exported account (see `/api/export`) into the
        * caller's own account. Session-gated; always imports into
        * `session.user.id` — there is no request field for a target user, so a
        * caller can never import into anyone else's account.
```

**Confirmed at the function level too.** `importUserData(db, userId, data)`
in `src/lib/export/import-user.ts` takes `userId` as an explicit second
parameter (never reads an id off `data`), and the existing test
`src/lib/export/import-user.test.ts` (`"imports an export for a fresh user
in FK order, remapping ids and skipping secret tables"`) asserts every
imported row across every table (`wellnessDaily`, `dailyMetrics`,
`coachMemories`, `chatThreads`, `activities`, `races`, `trainingPlans`,
`weekPlans`, `apiTokens`, `connections`, `webhookSubscriptions`,
`llmSettings`) lands under `TARGET_USER`, even though the export fixture
was built from a _different_ `SOURCE_USER` — i.e., the import path
deliberately re-keys every row to the resolved target id rather than
trusting any id embedded in the export payload. This is the strongest
possible proof against a cross-user import: the test literally exercises
"export from user A, import as user B" and asserts everything lands on B.

✅ confirmed clean — this is the single most important check in the release
and it holds: there is no code path, no request field, and no function
signature by which a caller could redirect an import at another user's
account.

## 5. Sync-jobs admin panel (Task 8)

**Files:** `src/app/admin/page.tsx`, `src/app/admin/actions.ts`

**Page-level gate** (blocks rendering the panel at all for non-owners):

```
src/app/admin/page.tsx:11-12:
  const user = await requireUser();
  if (user.role !== "owner") redirect("/");
```

**Action-level gate** (defense in depth — every mutating action re-checks,
independent of the page guard, since server actions are independently
invocable):

```
src/app/admin/actions.ts:16-22:
 async function requireOwner() {
   const user = await requireUser();
   if (user.role !== "owner") {
     throw new Error("Owner access required.");
   }
   return user;
 }
```

`retrySyncJob` (line 70) and `kickUserSync` (line 113) both call
`await requireOwner()` as their first line, before touching the DB. Both
are _intentionally_ cross-user by design (an owner retrying/kicking another
user's sync job is the entire point of the panel) — the isolation property
that matters here is owner-only, not per-user, and that's what's enforced.

Confirmed by existing regression tests in
`src/app/admin/__tests__/sync-jobs.test.ts`: both
`"retrySyncJob resets a failed test-* job to pending, owner only"` (line 73)
and `"kickUserSync bumps a due-later job to now for the target user, owner
only"` (line 138) assert a `MEMBER`-role caller gets `rejects.toThrow(/owner
access required/i)` **and** that the rejected call left the target row
untouched, before asserting the `OWNER`-role call succeeds. ✅ confirmed —
owner-only, not just any authenticated user, with a test proving the
rejection is a true no-op.
