# Accessibility sweep — 2026-07 (v0.20 Track 3, Task 13)

This is a **check-and-close** pass, not a retrofit. Tracks 1 and 2 of v0.20
(12 tasks) are already merged onto this branch, and v0.19 shipped with an
"accessibility as-you-go" commitment — so the expectation coming in was that
most surfaces would already be correct, with a handful of real gaps to find
and fix. That's roughly what happened: the app is meaningfully more
accessible than a typical unaudited app (aria-labels, `role="radiogroup"`,
`aria-pressed`, `role="status"` live regions, and a base-ui `Collapsible`
that already gets closed-panel tab-order right are all over the existing
code), and the gaps found were small and targeted, not systemic.

## Testing approach: axe (automated) + manual reading, not one or the other

The brief left this as a judgment call. Decision: **do both**, because they
catch different things and neither alone would have caught everything found
here.

- `vitest-axe` (0.1.0) was added as a devDependency and wired into 3 test
  files covering the brief's named highest-traffic surfaces: the dashboard
  hero (`hero-readiness.axe.test.tsx`), the journal form
  (`journal-form.axe.test.tsx`), and the shared `Collapsible` component used
  by every settings accordion, the journal's numbered steps, and the coach
  composer's history/quick-context sections (`ui/collapsible.test.tsx`, new
  `describe("accessibility")` block). Axe is good at objectively verifying
  "is this rendered DOM free of known WCAG rule violations" and it directly
  confirmed the ScoreRing fix (below) actually works, not just reads
  correctly.
- Manual reading was still necessary and caught real bugs axe's automated
  ruleset cannot: axe checks DOM structure, not color math against a
  hand-picked design-token opacity (it doesn't know `text-white/30` on a
  `#0a0a0a` background fails 4.5:1 — no DOM violation is triggered by that,
  it's just two CSS custom colors), and it can't tell you a `focus:` state
  was silently deleted (`outline-none` with zero replacement doesn't violate
  any DOM rule either — the element is still focusable, axe has no
  opinion on whether the _visual_ indicator exists). Both of the two most
  important fixes below (contrast, missing focus rings) were found by
  reading component code and globals.css, not by axe.

**vitest-axe gotcha (worth recording for future use):** the published
`vitest-axe@0.1.0` package's `vitest-axe/extend-expect` entry point is a
broken/empty build artifact for this Vitest version — importing it for its
runtime side effect registers nothing, and `expect(...).toHaveNoViolations()`
throws `Invalid Chai property`. Confirmed by inspecting
`node_modules/vitest-axe/dist/extend-expect.js` (empty) vs
`node_modules/vitest-axe/dist/matchers.js` (correct, working
implementation). Worked around by registering the real matcher manually
(`expect.extend(matchers)` from `vitest-axe/matchers`, done per test file)
and adding a small local type-augmentation file,
`src/test/vitest-axe.d.ts`, since the package's own `.d.ts` augments the old
`Vi.Assertion` namespace that Vitest 4's `@vitest/expect` no longer merges
into. Both workarounds are commented in place. This is a real deficiency in
an unmaintained package, not a project bug — flagging in case a future task
touches this again and wonders why the "normal" `vitest-axe/extend-expect`
import doesn't work.

## Checklist and findings, by surface

### Navigation — `bottom-nav.tsx` / `sidebar-nav.tsx` / `app-shell.tsx`

**PASS, no changes.** Both are `<nav>` landmarks built from `next/link`
(renders a real `<a>`), accessible name comes from visible text next to each
icon (not icon-only). The two navs never coexist in the accessibility tree —
`BottomNav` is `lg:hidden`, `SidebarNav` is `hidden lg:flex`; Tailwind's
`hidden` is `display:none`, which removes the inactive one from the a11y
tree entirely (not just visually), so there's no duplicate-landmark
confusion for screen reader users at any viewport. Active-page state is
communicated via `pathname` comparison but only changes visual styling —
this is a pre-existing minor gap (no `aria-current="page"`), noted below as
deferred since it wasn't flagged as broken by axe and is a "nice to have"
rather than a WCAG failure (the page's own `<h1>`/title still communicates
location).

### `score-ring.tsx` — the brief's flagged "already good, verify it" component

**Verified correct on the parts that were already there, found and fixed one
real gap.**

- `role="img"` + calibrating-aware `aria-label` (e.g. `"Readiness: 72 out of
100"` or `"Readiness: calibrating"`) reads correctly aloud — confirmed
  correct, no change.
- **Gap found:** the decorative SVG (progress ring) and the visually
  redundant center-text/label `<div>` were both plain children of the
  `role="img"` container with no `aria-hidden`. Per ARIA-in-HTML, `role="img"`
  is supposed to present its subtree as a single unit, and modern
  Chrome/Firefox do prune descendant text nodes from the accessibility tree
  correctly — but relying on that implicitly is a known, documented footgun
  (inconsistent across browser/AT combinations, notably older Safari/VoiceOver
  behavior). Without explicit `aria-hidden`, a `centerText` span like `"72"`
  and the `label` span (`"Readiness"`) sitting inside the `role="img"`
  container risk being announced a second time as plain text alongside the
  `aria-label`, on top of the visible label already rendering separately for
  the small-ring variant. **Fixed:** added `aria-hidden` to the `<svg>` and
  to the center-text wrapper `<div>`, so the only accessible content in that
  container is the single `aria-label`, guaranteed, not implementation- or
  browser-dependent.
  - Before: `<svg viewBox=... className=...>` / `<div className="absolute inset-0 ...">`
  - After: `<svg aria-hidden viewBox=... className=...>` / `<div aria-hidden className="absolute inset-0 ...">`
- Verified via the new `hero-readiness.axe.test.tsx` (ScoreRing is exercised
  through `HeroReadiness`) — 0 axe violations before and after, since axe's
  ruleset doesn't specifically check this redundancy pattern; the fix is a
  genuine defense-in-depth correctness fix, not a regression-test-driven one.

### `hero-readiness.tsx` — verified it doesn't break ScoreRing's semantics

**Confirmed correct on wrapping** — renders `<ScoreRing .../>` directly with
no extra `role`/`aria-*` overrides that would conflict with or duplicate its
internal accessibility contract.
**Contrast gap found and fixed** (see "Contrast" section below for the math):
the "Recovery" / "Sleep" / "Strain" stat labels (9px, uppercase) and the
"Load computed from your sessions" caption (10px) were `text-white/30` and
`text-white/40` respectively against the app's near-black background —
computed contrast ~2.6:1 and ~3.8:1, both below the 4.5:1 WCAG AA floor for
normal-size text. Bumped to `text-white/50` (computed ~5.3:1, passes).

### Focus order — v0.19 collapsible sections

**Verified correct, now with a regression test.** Read
`node_modules/@base-ui/react/collapsible/panel/useCollapsiblePanel.js`
directly: the panel's `hidden` prop is computed as `!open && !mounted` —
`mounted` stays `true` only while a close animation is actively running, and
flips to `false` once it finishes, at which point `hidden` becomes `true`.
The `hidden` HTML attribute removes an element from both the accessibility
tree and the tab order, so a fully-closed accordion section's content is
correctly unreachable by keyboard — confirmed with a new test in
`ui/collapsible.test.tsx` (`"closed panel content is not reachable by
keyboard tab order"`) that asserts the panel's content sits inside a `[hidden]`
ancestor when closed. This pattern is shared by every settings accordion,
the journal form's 3 numbered steps, and the coach composer's history/
quick-context sections, so one test covers all of them. No fix needed.

### Focus ring — visible indicator on interactive elements

Checked every `outline-none` usage in `src/components`/`src/app` (10 call
sites, excluding shadcn's own `ui/button.tsx`/`ui/input.tsx`/`ui/tabs.tsx`
which correctly pair `outline-none` with `focus-visible:ring-3
focus-visible:ring-ring/50`). **3 of the 10 had zero replacement focus
treatment** (fixed, see above: chat-interface.tsx input, journal-form.tsx
notes textarea, debrief-form.tsx notes textarea). The remaining 7 pair
`outline-none` with `focus:border-white/30` or `focus:border-emerald-400/50`
— a real, if visually subtle, focus indicator. `border-white/30` on this
near-black background computes to roughly the same low contrast as the
`text-white/30` failures elsewhere (~2.6:1), which is below the 3:1 floor
WCAG 1.4.11 asks for on UI-component boundaries — but 1.4.11's application to
focus-indicator-specifically vs. general component contrast is genuinely
disputed among accessibility practitioners (2.4.11 Focus Appearance, the
stricter and more literally on-point criterion, is AAA not AA). Given that
ambiguity and that this pattern appears in several unrelated files
(`day-actions.tsx`, `races-section.tsx`), fixing it site-wide would be a
larger, more speculative change than this task's "small, targeted fix"
mandate — **documented here as a deferred, lower-confidence finding** rather
than fixed. The unambiguous bug (a focus state that doesn't exist at all)
was fixed everywhere it was found.

## Contrast — methodology

WCAG AA requires 4.5:1 for normal text, 3:1 for large text (≥24px, or ≥19px
bold). This app is dark-only (no light theme), background is effectively
`#0a0a0a`. For a `text-white/N` opacity value blended over that background,
relative luminance was computed via the standard WCAG formula (sRGB → linear
per channel, then `0.2126R + 0.7152G + 0.0722B`, contrast = `(L1+0.05)/
(L2+0.05)`). Worked example for `white/30`: blended color ≈ `rgb(83,83,83)`,
contrast ≈ **2.6:1** (fails). `white/50`: blended ≈ `rgb(132,132,132)`,
contrast ≈ **5.3:1** (passes). `white/40` sits at ≈ **3.8:1** — passes for
large/bold text only, fails for the 9-10px labels it was actually used on
in this app. This is why every fix above lands on `/50`, not `/40`: it's the
first standard Tailwind opacity step that clears AA for small text against
this specific background.

**Scope note:** a repo-wide grep found ~21 more `text-white/⩽30` usages
outside the surfaces this task's brief named as highest-traffic (nav,
journal, settings, coach composer, dashboard hero). Most are `placeholder:`
text paired with a real associated `<label>` (a common, lower-severity case
— WCAG 1.4.3 technically applies to placeholder text too, but the visible
label already carries the accessible name and the primary instruction, so
this was judged lower priority and left alone) or single decorative
characters (an em-dash placeholder for an empty stat value). A few — the
login page's footer/labels, `import-form.tsx`'s "Max 5 MB" hint — are real
small-text-at-low-contrast instances outside the brief's named surfaces,
left unfixed and noted here rather than expanded into scope, consistent with
this being a sweep of named surfaces, not a full design-token migration.
**Recommended follow-up** (not done here, would be a real if modest scope
increase): define a semantic low-contrast-safe token (e.g. a
`--muted-foreground-subtle` CSS variable pinned at `/50` or above) and
migrate remaining `text-white/⩽40` call sites to it, so future components
can't reintroduce this by picking an arbitrary opacity value.
