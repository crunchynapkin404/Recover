# v0.10 — Honest Load: Design

**Date:** 2026-07-18
**Status:** Approved

## Goal

Recover stops borrowing its training-load math. Today `ctl`/`atl` are
written only by the intervals.icu sync, so a manual-only athlete gets a
hero "Recovery 60" and "Strain 0.0" invented from zero data
(`latest?.atl ?? 0`). This release computes per-activity load and
CTL/ATL/TSB natively from the athlete's own sessions — for every source —
and propagates the `calibrating` treatment readiness already has through
`ScoreRing`, `StrainBudget`, and the narrative. Done when: a
no-integration athlete who logs workouts sees Recovery and Strain computed
from their own sessions — or `calibrating` — and the honesty-debt section
of `docs/ROADMAP.md` is empty.

## Decisions

| Decision            | Choice                                                                  |
| ------------------- | ----------------------------------------------------------------------- |
| Load unit           | TSS-like throughout (intervals.icu `load` is already TSS-like)          |
| Per-activity ladder | provider `load` → power TSS → HR TSS → duration estimate → none         |
| CTL/ATL             | Standard EMAs (42d / 7d) over daily load sums, seeded at 0              |
| Precedence          | Provider (intervals.icu wellness) ctl/atl win; native fills gaps        |
| Storage             | Effective ctl/atl + source label on `daily_metrics` (new columns)       |
| Calibrating rule    | < `MIN_LOAD_DAYS` (7) activity days in trailing 42 → no strain/recovery |
| Athlete thresholds  | `body_prefs.max_hr` / `body_prefs.ftp_watts`, editable in Body card     |
| Strava              | Included in load sums (local analytics, not an AI context — see below)  |

## Architecture

### Pure engine — `src/lib/training-load.ts`

No db, no I/O — same shape as `readiness.ts`. Constants exported for tests.

- `activityLoad(activity, athlete)` → `{ load, source } | null`, first match
  wins:
  1. **provider** — `activity.load` when present (intervals.icu TSS, CSV
     `load` column).
  2. **power** — needs `avgPower` + athlete FTP:
     `hours × (avgPower/ftp)² × 100`. Average power proxies NP — documented
     under-estimate for spiky rides; honest direction of error.
  3. **hr** — needs `avgHr` + athlete `maxHr` + resting HR: heart-rate
     reserve fraction `hrr = (avgHr − restingHr)/(maxHr − restingHr)`
     (clamped 0–1), intensity factor `hrr / 0.85` (LTHR ≈ 85% of HRR),
     capped at 1.15; `hours × IF² × 100`.
  4. **duration** — `hours × 40`: an unlabeled hour counts as easy zone-2.
     Deliberate under-estimate; fabricating intensity upward is the exact
     defect class this release removes.
  5. No duration → `null` (the activity contributes nothing).
- `dailyLoadSeries(activities, athlete)` → `Map<ymd, { load, sources }>`.
  Cross-provider dedup: two activities from _different_ providers starting
  within 30 minutes with durations within 10% are the same workout synced
  twice — the row with a provider `load` wins (ties: `intervals_icu` >
  `manual` > `strava`).
- `emaSeries(days, timeConstant)` — `v = prev + (load − prev)/tc`, seeded 0
  from the athlete's first activity day. CTL tc = 42, ATL tc = 7.
- `nativeLoadMetrics(activities, athlete, upToDate)` → per-date
  `{ ctl, atl }` plus `activityDays42` (distinct activity days in the
  trailing 42) for the calibrating rule.

Resting HR for the HR rung comes from the trailing 60-day wellness mean
(min 7 values), computed by the caller — the engine takes plain numbers.

### Pipeline — `computeDailyMetrics` (`src/lib/metrics.ts`)

Already recomputes every day from `sinceDate`. Gains one step: fetch the
user's activities once, build the native series, and per day resolve

```
ctl = wellness.ctl ?? native.ctl   (atl likewise; source: "provider" | "computed")
```

The effective values feed `computeReadiness` (form component now works for
manual athletes) and are stored on `daily_metrics` — new nullable columns
`ctl`, `atl`, `load_source`. Days with neither remain null. Migration 0013,
purely additive.

### Dashboard — `src/app/page.tsx`

Recovery/Strain switch from `wellness.latest?.atl ?? 0` to the latest
`daily_metrics` row's effective values:

- Values present and ≥ `MIN_LOAD_DAYS` of history → rings as today, plus a
  small "computed from your sessions" caption when `load_source =
"computed"`.
- Otherwise → `ScoreRing` gains a `calibrating` prop (renders "—", muted
  track, honest aria label); `StrainBudget` renders its calibrating state
  instead of a full-width empty budget; the narrative's strain-budget
  sentence drops out (readiness handles its own calibrating line already).
- The Training Status tile's fabricated "Optimal load intensity" caption
  becomes the real CTL (`CTL 45 · computed`) or nothing.

### "This Week" rings — `WeeklySummary`

`ringOuter={0.7} ringInner={0.8}` is replaced by real fractions:

- **Volume ring**: week volume ÷ planned volume (open week plan's workout
  minutes) — fallback: trailing 28-day weekly average.
- **Load ring**: week load ÷ current block `targetLoad` — fallback:
  trailing 28-day weekly average load.
- No plan and no history → rings hidden (component accepts `null`), never
  a made-up fraction.

### Settings — Body card

`body_prefs` gains nullable `max_hr` and `ftp_watts`; the Body card gets
the two inputs (validated: maxHr 100–230, ftp 50–600, both optional).
Without them the engine simply can't use rungs 2–3 — it degrades to the
duration estimate, honestly labelled.

## Strava firewall — read side only

The Nov-2024 agreement bars Strava API data from AI contexts (coach/MCP)
and from being pushed elsewhere — Recover's queries there stay
`ne(provider,'strava')`. Local, user-facing load arithmetic on the user's
own dashboard is not an AI context; Strava activities therefore count
toward daily load (they cost real recovery capacity), same as they already
count in the dashboard's weekly volume card. Nothing new leaves the
database.

## Testing

- **Engine unit tests** (`training-load.test.ts`): each ladder rung, the
  precedence order, clamps, dedup (same/different provider, boundary
  cases), EMA correctness against hand-computed values, calibrating count.
- **Metrics tests**: effective-value resolution (provider wins, native
  fills, neither → null) — pure-resolution helper tested without db.
- **Component tests**: ScoreRing calibrating render + aria, StrainBudget
  calibrating state, WeeklySummary hidden-rings state (follow
  `body-battery.test.tsx` / `vitals-grid.test.tsx` patterns).
- Existing suite stays green; no test may depend on external env.

  provider-owned.
