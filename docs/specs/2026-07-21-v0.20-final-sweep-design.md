# v0.20 — Final Sweep: Design

**Date:** 2026-07-21
**Status:** In review

## Goal

Close out the current roadmap in one release. v0.20 is a **sweep, not a
feature launch**: it finishes the cross-cutting polish, the operations track,
and the 1.0-hardening items that were deferred out of earlier releases, so the
roadmap can be declared complete and a fresh one started on clean ground.
Nothing net-new in user-facing scope — everything here either finishes a
half-done item, closes a backlog line, or makes what exists more trustable.

Done when: every un-parked open box on the roadmap through v0.19 is either
checked or explicitly deferred to the new roadmap, the full suite is green, and
`v0.20.0` is tagged.

## Why this is the shape it is

The roadmap's remaining open items span four buckets: cross-cutting polish, a
whole social subsystem (v0.16 Stronger Together), an operations track (v0.17
Good Self-Hosted Citizen), and the 1.0-hardening remainder (v0.18). Three
decisions set this spec's boundary:

- **Social is a feature, not a sweep item.** Stronger Together (per-pair
  sharing, group view, coach seat, digest, shareable cards) is a real
  subsystem deserving its own brainstorm and spec. It moves to the **new
  roadmap**, not here.
- **The parked items stay parked.** Cycle-Aware Readiness (no instance
  generates cycle data — phase-aware baselines against synthetic cycles would
  be fabrication with extra steps) and Fitbit/Google-direct (conditional on
  demand that hasn't appeared) remain deferred. Building them now would
  violate the project's own honesty and YAGNI principles.
- **Usefulness is the filter, not completeness.** Auth hardening is scoped to
  a **session-management UI only** — no 2FA, no passkeys. The deployment model
  is the deciding lens, same as v0.18's: self-hosted, invite-only, no public
  signup, ~10 friends, behind a Cloudflare tunnel, single owner. v0.18.0
  already shipped login rate-limiting + an auth/token audit log. Against that
  model, 2FA and passkeys are real work for a threat the tunnel and
  invite-only gate already blunt; the ability to see and revoke your own
  active sessions is the piece that earns its place.

What remains — polish, ops, and the rest of hardening — ships as **one
`v0.20.0` release, one SDD branch, three ordered tracks.** Track order is
**Polish → Ops → Hardening**, deliberately: Polish and Ops build or extend
surfaces; Hardening runs **last** so the accessibility sweep, performance
pass, API/MCP freeze, docs review, and final security review all audit the
_finished_ state rather than a moving target. A single whole-branch review
caps it. This matches how v0.14 Race Ready — the largest prior release, 16
tasks — was built, and it gives one clean "roadmap closed" moment.

## Scope

Three tracks, one release.

### Track 1 — Polish

Closes the "design & UX" and "polish backlog" lines the v0.9.6 replan left
open. All cheap, all high-value, none earns its own release.

- **Empty states** on the pages v0.19 did not restructure — `plan`,
  `activity/[id]`, `activity/log`, `health`, `import`. Reuse the `EmptyState`
  primitive v0.19 added. Each renders an honest "nothing here yet + the next
  step" card, never a blank panel.
- **Loading skeletons** on the same pages. Only the five v0.19 routes ship a
  `loading.tsx` today; add layout-stable placeholders for the rest so there is
  no pop-in and the skeleton matches the final layout.
- **Chart consistency** — one visual grammar across every chart. `charts.ts`
  is the shared pure-math helper; the call sites are `stream-chart`,
  `wellness-trends`, `weekly-load-bars`, the dashboard sparklines,
  `get-training-load-summary`, and the coach `artifact-card`. **Scoped as
  token + axis/legend unification**, not a chart-engine rewrite: one shared
  set of axis/band/tooltip/color tokens and a shared axis+legend helper, so
  every chart reads as one system. The full "one grammar" ideal can balloon;
  this cap is intentional. Charts stay hand-rolled SVG — recharts stays
  unused.
- **Default journal entries** — pre-toggle frequent behaviours in
  `journal-form.tsx` so only exceptions get marked. **Hard constraint:** this
  must not re-introduce the v0.7 fabrication the score-integrity release
  fixed. Pre-toggles apply only to _behavioural_ flags the athlete has
  established as their norm; they must never pre-fill or auto-submit the
  energy/soreness/stress sliders. Unanswered subjective sliders still write
  nothing.
- **Performance-log filters** — the roadmap's "wire up the month/sport
  controls" item. `log/page.tsx` already drives `view/month/range/sport`
  through one shared href-builder (v0.19). **Verify** the controls work
  end-to-end; if confirmed, check the roadmap box. Only real work if a gap
  surfaces.

### Track 2 — Ops / Self-Hosted Citizen

The v0.17 operations track. Makes Recover behave like the rest of a homelab
expects.

- **Prometheus `/metrics` + richer health.** A `/api/health` route already
  exists; extend it and add a scrapeable `/api/metrics` in Prometheus text
  format exposing: sync staleness (per provider), `sync_jobs` failure counts,
  backup age, and push-delivery health. Owner-gated or unauthenticated-but-
  minimal per the deployment model — decided at plan time; default to
  requiring the metrics scrape token.
- **Outbound webhooks.** New `webhook_subscriptions` table (**migration
  0020**) plus a dispatch step wired into `scheduler.ts`. Events:
  readiness-computed, band-changed, backup-completed. A configured URL
  receives a signed POST (HMAC over the body, secret per subscription) on each
  event; deliveries retry and failures are visible. Targets Home Assistant,
  ntfy, or anything listening.
- **Sync-jobs admin UI.** The `syncJobs` table already exists (`schema.ts`);
  `requestImmediateSync` already exists in the scheduler. Extend the
  owner-only `/admin` page (added in v0.18.0 for security events) with a
  sync-jobs panel: queue, running, failed, with retry and manual-kick
  controls.
- **GDPR export → round-trippable.** `src/app/api/export/route.ts` exists but
  is partial (wellness, activities, daily metrics, chat threads only). Extend
  it to cover **all** of a user's data: journal, biomarkers, coach memories,
  chat _messages_ (not just threads), connections, settings/prefs, races,
  training plans, week plans, adjustments, tokens metadata. Add a matching
  **import** path. Definition of done: **export → wipe → import round-trips
  losslessly** on a scratch database. Connection _tokens_ export as
  presence/provenance only — never decrypted secrets.
- **Native `arm64` release runners.** `release.yml` builds `linux/amd64` only
  today (arm64 was dropped in v0.8 to escape ~50-minute QEMU builds). Restore
  multi-arch with a native `ubuntu-24.04-arm` runner + manifest-merge, so the
  arm image returns without the QEMU cost.
- **Deploy-guide refresh.** Bring the Vercel + Neon deployment guide current
  against Next 16 and the current schema.

### Track 3 — Hardening

The remainder of v0.18's 1.0-hardening list, minus the security slice already
shipped in v0.18.0 and minus the 2FA/passkeys explicitly not wanted. Runs last
so each item audits the finished branch.

- **Accessibility sweep.** `score-ring.tsx` and `hero-readiness.tsx` first
  (ScoreRing aria labels are the named backlog item), then a pass across nav,
  forms, and interactive controls for contrast, focus order, and button/roles.
  The v0.19 "accessibility as-you-go" commitment means this is a _check and
  close_, not a from-scratch retrofit.
- **Session-management UI** _(only — no 2FA, no passkeys)_. Better Auth's
  `sessions` table already exists. Add a settings surface to list active
  sessions/devices and revoke them.
- **Upgrade guarantees.** Test migrations against a real dump on a scratch DB
  (extend the `restore-drill.sh` pattern), document a rollback procedure, and
  record a backup-compatibility matrix.
- **Performance pass.** Define and meet a dashboard cold-load budget; audit
  queries for N+1s and missing indexes on the hot paths.
- **API/MCP stability freeze.** Freeze tool names and schemas in
  `src/lib/tools/registry.ts` (`allTools`, ~54 tools) and publish a
  deprecation policy. Placed last so nothing else in the sweep moves a tool
  schema after the freeze.
- **Docs reviewed end-to-end** — self-hosting, connectors, troubleshooting
  current against the shipped state.
- **Final security review** — a full pass over the branch before the tag,
  building on v0.18.0's audit rather than repeating it.

### Explicitly deferred to the new roadmap

- **Stronger Together** (v0.16 social/sharing) — a real subsystem; its own
  brainstorm + spec on the new roadmap.

### Parked (unchanged)

- **Cycle-Aware Readiness** — returns when a real athlete on an instance needs
  it.
- **Fitbit / Google Health direct** — returns if demand appears.

## Cross-cutting constraints

- **The round-trip export is the single most dangerous item in the sweep.**
  Per the standing live-DB gotcha, every DB-gated test in this repo runs
  against the **real** `recover-db-1` — there is no separate test database.
  The export → wipe → import drill therefore runs **only against an ephemeral
  scratch Postgres** (same pattern `scripts/restore-drill.sh` already uses),
  **never** the live DB. The per-user export endpoint stays strictly
  read-only. Any test that exercises `wipe` must be scoped to fake
  `test-*`-id rows and can never issue an unscoped delete.
- **One migration (0020)** for `webhook_subscriptions`; every other item is
  additive or no-schema. Additive-only, no collision (last applied is 0019).
- **CI test-skip guard carries over.** Any new Vitest file that imports
  `@/lib/db` must gate with `describe.skipIf(!hasDb)`, or CI crashes instead
  of skipping (CI does not set `DATABASE_URL`).
- **Execution model.** Subagent-driven development; all implementer and
  per-task reviewer subagents on **Sonnet**. Final whole-branch review on
  **Fable** (standing preference for the capstone review). One PR, one
  whole-branch review.
- **Release process.** Version bump + CHANGELOG + roadmap ticks happen
  in-branch before merge; merge to main; verify main CI green; only then tag
  `v0.20.0` **last** per `RELEASING.md`, then `gh release create`. CI runs
  `format:check` over the whole repo — prettier every committed doc (including
  this spec and the plan) before committing, or main goes red.

## Testing

- Polish: component/render tests for empty states and skeletons on the four
  pages; a snapshot of the unified chart tokens; a journal-form test asserting
  pre-toggles never touch the subjective sliders (guards the v0.7 regression).
- Ops: a `/metrics` shape test; a webhook-dispatch test with a fake receiver
  asserting the HMAC signature and retry behaviour; the export→wipe→import
  round-trip test against a scratch DB asserting the restored data is
  semantically identical to the export; a
  sync-jobs admin action test (retry/kick) scoped to a fake owner.
- Hardening: axe/aria assertions on ScoreRing and the hero; a
  session-list/revoke test; a migration-against-dump drill; a frozen-tool
  snapshot test that fails if any `allTools` name or schema changes after the
  freeze.
- Full suite green (currently ~787+), `tsc`/`lint`/`build`/`format:check`
  clean, migration 0020 applied to the live DB via the review worktree.

## Done when

Every un-parked open checkbox on the roadmap through v0.19 is checked or
explicitly moved to the new roadmap; the ops track lands (metrics scrape,
webhook fires to a listener, a lossless export round-trip, arm64 image
restored); the hardening track closes (a11y checked, sessions revocable,
upgrade drill documented, tools frozen, docs and security reviewed); and
`v0.20.0` is tagged as the final release of this roadmap.
