# v0.5d — Training Plan Generation Design

> Sub-project D of v0.5 (flagship feature). Depends on v0.5a (Artifacts for
> plan visualization), v0.5b (Weekly Review for adherence tracking), and v0.5c
> (Calendar for planned workouts + availability).

## Problem

No consumer product combines intervals.icu data depth with AI plan generation.
Existing options: TrainingPeaks (expensive, human coach), intervals.icu plans
(template-based, not adaptive), ChatGPT (no data access). Recover's unique
position: it has the athlete's real CTL, power/pace curves, recovery metrics,
AND an LLM with full tool access.

## Architecture

```
User tells coach about race goal
  → coach calls remember_fact(category: "race", ...)
  → coach calls generate_training_plan(raceType, raceDate, constraints)
    → tool gathers: current CTL, curves, calendar, preferences
    → tool builds periodization prompt → LLM generates blocks
    → tool validates structure → stores in DB
    → tool returns summary + calls render_chart (CTL projection)
  → coach presents plan with inline artifacts
  → weekly: Weekly Review compares actual vs target
  → coach adapts: suggests plan modifications via update_training_plan
```

## Schema

```sql
CREATE TABLE training_plans (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id text NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  title text NOT NULL,
  race_type text NOT NULL,          -- 'marathon', 'ironman', '70.3', 'gran_fondo', 'century', '5k', '10k', 'ultra'
  race_date date NOT NULL,
  start_date date NOT NULL,
  weeks_total smallint NOT NULL,
  current_week smallint NOT NULL DEFAULT 1,
  target_ctl real,                  -- projected CTL at race day
  starting_ctl real,                -- CTL when plan was created
  status text NOT NULL DEFAULT 'active', -- active | completed | archived
  constraints jsonb,                -- { daysPerWeek, hoursPerWeek, sports[], avoidDays[] }
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE training_blocks (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  plan_id uuid NOT NULL REFERENCES training_plans(id) ON DELETE CASCADE,
  week_number smallint NOT NULL,
  phase text NOT NULL,              -- 'base' | 'build' | 'peak' | 'taper' | 'recovery'
  target_load_total real,           -- target weekly TSS/load
  target_sessions smallint,
  workouts jsonb NOT NULL,          -- [{day: 0-6, sport, type, durationMins, intensity, description}]
  actual_load real,                 -- filled by weekly review
  actual_sessions smallint,
  adherence_pct real,               -- actual/target * 100
  notes text,                       -- coach commentary from weekly review
  UNIQUE(plan_id, week_number)
);

CREATE INDEX training_plans_user_status_idx ON training_plans(user_id, status);
```

## Periodization Logic

The LLM generates plans, but we constrain it with sport-science guardrails:

### Input Context for Plan Generation

```
Current fitness:
- CTL: {ctl} (represents ~{weeks_to_build} weeks of consistent training)
- Key power metrics: FTP {ftp}W, 5min {5min}W, 1min {1min}W
- Key pace metrics: 5k {5k_pace}/km, threshold {threshold}/km
- Weekly load capacity (last 4 weeks avg): {avg_weekly_load}
- Available days: {days_per_week}
- Available hours: {hours_per_week}
- Sports: {sports list}
- Recovery profile: avg readiness {avg_readiness}, HRV baseline {hrv}

Race target:
- Type: {race_type}
- Date: {race_date} ({weeks_until} weeks away)
- Priority: A-race

Periodization rules (MUST follow):
1. Never increase weekly load >10% week-over-week
2. Every 3rd or 4th week is a recovery week (load = 60% of prior week)
3. Taper: final 2 weeks reduce load 40% then 60%
4. Peak week: 10-14 days before race, last high-intensity session
5. Base phase: 70% Z1-Z2 volume, 2x strength, build aerobic base
6. Build phase: introduce race-specific intensity, maintain volume
7. Peak phase: reduce volume, maintain intensity, race-pace sessions
8. For triathlon: brick sessions (bike→run) every 10-14 days in build/peak
```

### Validation Rules (enforced after LLM generation)

- Total weeks matches `raceDate - startDate`
- Recovery weeks present every 3-4 weeks
- Load progression ≤10% per week (except recovery weeks)
- Taper present in final 2 weeks
- Each workout has required fields (day, sport, type, duration, intensity)
- No workouts on user's `avoidDays`

If validation fails: retry generation once with error feedback, then return
partial plan with a note to the coach.

## Tools

### `generate_training_plan`

```ts
parameters: z.object({
  raceType: z.enum(["marathon", "half_marathon", "10k", "5k", "ultra",
    "ironman", "70.3", "olympic_tri", "sprint_tri",
    "gran_fondo", "century", "crit", "general_fitness"]),
  raceDate: z.string().describe("YYYY-MM-DD"),
  title: z.string().optional().describe("Plan name, e.g. 'Berlin Marathon 2026'"),
  daysPerWeek: z.number().int().min(3).max(7).default(5),
  hoursPerWeek: z.number().min(3).max(25).default(8),
  sports: z.array(z.string()).optional()
    .describe("Override sports list. Defaults to athlete profile."),
})
scope: "write:plan"  // new scope — requires explicit grant
```

Returns: plan summary + artifact (table of weeks with phase/target load) +
CTL projection chart.

### `get_training_plan`

```ts
parameters: z.object({
  weekNumber: z.number().int().optional()
    .describe("Specific week to detail. Omit for overview."),
})
scope: "read"
```

Returns: active plan overview, or specific week's workouts if weekNumber given.

### `update_training_plan`

```ts
parameters: z.object({
  weekNumber: z.number().int(),
  action: z.enum(["reduce_load", "increase_load", "swap_rest_day", "skip_week", "extend"]),
  reason: z.string().describe("Why the adjustment is being made."),
})
scope: "write:plan"
```

Used by coach to adapt the plan based on weekly review feedback.

## Weekly Review Integration

When Weekly Review (v0.5b) runs and a plan is active:

```ts
// In weekly-review.ts
const plan = await getActivePlan(userId);
if (plan) {
  const block = await getCurrentBlock(plan.id);
  block.actual_load = thisWeekLoad;
  block.actual_sessions = thisWeekSessions;
  block.adherence_pct = (thisWeekLoad / block.target_load_total) * 100;
  // Include in review prompt: "Plan adherence: 85% (target 450 TSS, actual 382)"
}
```

## Artifact Output

When a plan is generated, the tool calls `render_chart` twice:

1. **Table artifact:** Week overview (phase, target load, sessions per sport)
2. **Line chart artifact:** Projected CTL ramp from start to race day

## Scope: `write:plan`

New token scope. Users creating MCP tokens can now grant:
- `read` — all read tools (default)
- `write:wellness` — log wellness entries
- `write:plan` — generate/update training plans

This prevents an MCP client from accidentally modifying plans without explicit
permission.

## DoD

- [ ] Coach generates a periodized plan from a race goal conversation
- [ ] Plan stored in DB as structured blocks (training_plans + training_blocks)
- [ ] Inline artifacts: week table + CTL projection chart
- [ ] Validation enforces periodization guardrails (load progression, recovery weeks, taper)
- [ ] Weekly Review reports plan adherence when a plan is active
- [ ] Coach can adjust plan via `update_training_plan`
- [ ] Multi-sport: triathlon plans include swim/bike/run + brick sessions
- [ ] Manual-only users can still get plans (based on CTL from activities, no calendar)

## Non-Goals (v0.5)

- Push plan workouts to intervals.icu calendar (future write-back)
- Push plan workouts to Google Calendar (future, needs `calendar.events` scope)
- Workout execution guidance (Garmin/watch integration — out of scope)
- AI-generated workout files (.fit/.zwo) — future if demand
