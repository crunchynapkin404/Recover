# v0.19 — Design Refresh: Design

**Date:** 2026-07-20
**Status:** In review

## Goal

A Superdesign pass rethought five screens — dashboard, coach, log, journal,
and settings — around **progressive disclosure**: collapsed-by-default
sections instead of everything rendered flat at once. This release adopts
that structure. It is a **restyle, not a rebuild** — the underlying data,
server queries, and features are unchanged. Done when: the dashboard, coach,
log, journal, and settings pages match the export's structural pattern
(collapsed-by-default sections, restyled chart wrappers), the "Settings
information architecture" backlog item is checked off, and those five pages
carry honest empty states and layout-stable loading skeletons.

Design source: `docs/flow-export-1784540566598/` (Superdesign project
`feee3bd4-a46d-4c81-93eb-16107ffebbcf`), six mockup HTML files.

## Why this is v0.19

v0.19 is the last 0.x before v1.0.0. It was originally slotted as "1.0
Hardening" and later reframed by the roadmap as the Design Refresh, with
hardening folded into v0.18. This is the release that makes the five most
data-dense screens legible before the app stops being pre-1.0. Several of
its patterns directly close items already sitting in the roadmap's "Ongoing
— design & UX" track (settings IA, empty states, skeletons).

## Principle

**The mockups' values are placeholders, not data.** The honesty pillar
governs this release the same as every other. The dashboard mockup shows
"BRI Score 4.2", "1,847 kJ", "Dolomites 2026"; the coach mockup shows an
archive badge reading "3"; the settings mockup shows "Claude 3.5" and
"v3.4.2". **None of these become real numbers.** Every restructured section
wires to the _same real values the current components already compute_, or
shows an honest empty / `calibrating` state — never `?? 0`, never an
invented default. Any mockup element with no real feature behind it is
dropped, not stubbed (see "Explicitly not carried over").

This means the release adds no new data, no new metric, and no new coach
capability. If a section can't be filled from data that already exists, it
renders an empty state, not a fabricated one.

## Scope

Five pages, one release, plus a shared foundation built first. The numbered
items below are the implementation tasks, in order (a seventh task, the final
whole-branch review, is covered under Delivery).

1. **Shared foundation** — a `Collapsible` UI primitive used by all five
   pages, and the empty-state + loading-skeleton conventions (the conventions
   are then applied within each page's own task, not separately).
2. **Dashboard** — hero readiness ring as the single focal metric; compact
   stat row; Recovery Metrics and Recent Sessions as collapsed accordions.
3. **Settings** — one accordion per domain; only Profile open by default.
4. **Log** — Today/Week/Month time-range toggle + month strip; PMC panel
   collapsible; wellness-trends charts preserved as a collapsed section.
5. **Journal** — stepped check-in; correlations promoted above the form;
   5-day calendar strip with the real streak.
6. **Coach** — collapsible Chat History and Quick Context panels; quick-reply
   chips. Header/composer stubs dropped.

### Confirmed decisions (made with the user)

- **Login page: out of scope.** The export includes a sixth mockup
  (`06-recover-premium-login-optimized.html`), but the roadmap's v0.19
  checklist names only the five surfaces above. The login mockup is a
  flow-tool artifact (the export needed a complete navigable flow); login is
  not restyled in this release.
- **Empty states + skeletons: bundled in.** Both are roadmap "Ongoing —
  design & UX" items meant to be picked up alongside any release. Since v0.19
  already rewrites the markup of these exact five pages, it also gives all
  five honest empty states and layout-stable skeletons. This closes those two
  items _for these five pages_ (other pages remain in the ongoing track).
- **Delivery: one branch, subagent-driven.** Matches the v0.14/v0.15
  precedent: one worktree, one task per surface plus the shared foundations,
  per-task manual visual check, a single final whole-branch review, one PR.
- **Log wellness trends: preserved as a collapsible section.** See the Log
  section below for the conflict this resolves.

## Shared foundation: the `Collapsible` primitive

**File:** `src/components/ui/collapsible.tsx` (new).

**Backing:** `@base-ui/react` (already a dependency). Follows the repo's
Base UI convention — the `render` prop, never `asChild` (per `AGENTS.md` and
the shadcn-wraps-base-ui note in the stack docs).

**Why a component and not the mockups' raw `<details>/<summary>`:** the
mockups use plain uncontrolled `<details>`. That works for the dashboard,
settings, and log accordions, which open and close independently. It does
**not** work for the journal's stepped check-in, where finishing step 1 must
_programmatically_ collapse step 1 and open step 2 — behavior an uncontrolled
`<details>` can't express. One shared primitive that supports both modes also
gives the app a single disclosure grammar instead of five ad-hoc ones,
matching the project's own "chart consistency" instinct.

**Behavior:**

- **Default: independent multi-open.** Verified against all six mockups —
  none of them ever force a sibling section closed when another opens. This
  is a set of independent collapsibles, not a strict one-at-a-time accordion.
- **Controlled and uncontrolled** use both supported. Dashboard / settings /
  log use it uncontrolled (each section owns its own open state, default
  closed except where noted). Journal uses it controlled (parent drives which
  step is open).
- **Shared summary row:** glass background, a chevron that rotates on open
  (`group-open:rotate-180` equivalent), and an optional badge slot for the
  count/status pills the mockups show ("3", "5 sessions", "Connected").
- **Motion:** height/opacity transition matching the mockups'
  `accordion-content` easing. Respects `prefers-reduced-motion`, which
  `globals.css` already enforces globally (`animation/transition: none`).

**Testing:** unit tests for open/close, controlled vs uncontrolled, the
badge slot, and that reduced-motion collapses instantly. This is the main
new testable unit in the release.

## Shared foundation: empty states and skeletons

- **`EmptyState` helper** (small, shared): an icon plus one honest line of
  copy, in the glass card idiom. Used wherever a restructured section can
  have no data — no activities logged, no wellness history, no plan, no
  correlations yet. Copy is honest and specific ("No activities synced yet",
  not a generic "Nothing here").
- **Skeletons:** the existing `src/components/ui/skeleton.tsx` primitive,
  placed so each restructured section reserves its final layout while
  loading — no pop-in, no layout shift. Applied to the dashboard hero and
  accordions, the log session list and charts, and the journal correlations.

Both are applied **as each page is restructured**, not as a separate task, so
every touched page ships complete.

## Page: Dashboard

**File:** `src/app/page.tsx` (currently 873 lines — the largest page in the
app) and `src/components/dashboard/*`.

**Mockup:** `01-recover-premium-home-dashboard-fixed-navigation.html`.

**Current state:** ~14 sections rendered in a flat vertical stack — hero ring,
`CalibrationProgress`, `StrainBudget`, `MorningBrief`, `CoachInsight`,
`RaceCountdownCard`, `VitalsGrid`, `SleepCard`, `SleepStagesCard`,
`SleepQualityCard`, `BodyBatteryCurve`, `WeeklySummary`, `MilestonesCard`.

**New structure (per mockup):**

1. **Hero:** the animated Readiness ring as the page's _single_ focal metric,
   keeping the existing `ring-animate` draw and `calibrating` semantics (no
   number invented when readiness is calibrating).
2. **Compact stat row:** a three-metric row of **Recovery, Sleep, and
   Strain** — the three top-line values the dashboard already computes
   (`recoveryScore`, `sleepScore`, `strainFraction`), each already carrying
   `calibrating` treatment. Each shows its real value or a calibrating
   state — never `?? 0`. The mockup's literal "BRI Score 4.2" is dropped:
   **there is no BRI metric in the app** (verified — the string appears
   nowhere in the codebase), and none is introduced. No new metric is added.
3. **"Recovery Metrics" accordion (collapsed by default):** HRV / RHR / sleep
   rows — today's `VitalsGrid` and the sleep cards move inside.
4. **Weekly Load** compact strip (real weekly load / target from v0.10's
   engine; hidden or calibrating when absent).
5. **Next Training** card: the real next planned session from the week-plan
   engine. Hidden when there is no active plan. The mockup's "Dolomites 2026 /
   120 TSS" text is placeholder content.
6. **"Recent Sessions" accordion (collapsed by default):** recent activities.

**Component extraction (approved):** the flat inline sections (hero, stat
row, section wrappers) are extracted into focused components under
`src/components/dashboard/`, rather than wrapping 873 lines of inline JSX in
accordions in place. Justification: the file is already too large to work in
comfortably, and the restructure touches all of it — this is the "improve the
code you're working in" case, not unrelated refactoring. Existing dashboard
card components are reused unchanged inside the new structure.

**Mockup:** `05-settings-optimized-navigation.html`.

**New structure:** one `Collapsible` per domain, only **Profile open by

- **Integrations** — intervals.icu, Strava, and the wearable connector cards

All existing settings components move into the matching section **unchanged**.
This checks off the roadmap's "Settings information architecture" backlog

omitted** — Recover stays dark-only until a real light theme is scoped.

## Page: Log

**File:** `src/app/log/page.tsx` (532 lines) and `src/components/log/*`.

**Mockup:** `03-update-navigation-bar-with-all-pages.html` (despite its
filename, this mockup is the Log / Performance Analytics page).

**The conflict this resolves.** The current Log page's toggle
(`LogTabs`, `?tab=`) switches between two _content types_: **Training** (PMC
chart, weekly stats, weekly load bars, sport-filtered session history, daily
load) and **Wellness** (`WellnessTrends` — HRV/RHR/sleep plotted against the
athlete's personal baselines). The mockup's new toggle is a _time-range_ axis
(**Today / Week / Month**) and shows only training/session content — the
wellness-trends charts appear nowhere in it. Taking "replace the toggle"
literally would orphan a whole view of real, honest baseline-band data.

**Resolution (user's decision):**

- The **Today/Week/Month time-range toggle becomes the page's primary axis**,
  URL-param driven, consistent with the existing `?range=`/`?days=`/`?sport=`
  pattern (every state change is a `Link`, as it is today). A month strip
  accompanies it per the mockup.
- The **PMC "Performance Trends" panel becomes collapsed-by-default** (a
  `Collapsible`). The chart itself is unchanged — this restyles the _wrapper_,
  which the roadmap explicitly notes is a first step on "Chart consistency,"
  not a resolution of it.
- The **`WellnessTrends` charts are preserved as a collapsed "Wellness
  Trends" section** on the same page. Nothing is orphaned.
- Sport-filter chips and day-grouped session history stay.

The `?tab=` param is retired in favor of the time-range param; the wellness
content it used to gate now lives in the always-present collapsible section.

## Page: Journal

**File:** `src/app/journal/page.tsx` (92 lines) and
`src/components/journal/*`.

**Mockup:** `04-journal-wellness-log-navigation-fix.html`.

**New structure:**

- **5-day calendar strip** at the top with the **real** consecutive-run
  streak indicator (the v0.9.4 milestone streak — a real run, never the old
  `Math.min(count, 30)` fake).
- **Correlation insights promoted above the form** (today's
  `correlation-insights.tsx` renders below it).
- **Stepped check-in** using the _controlled_ `Collapsible`: one step open at
  a time (Subjective feeling → wellness sliders → vitals). Completing a step
  collapses it to a checkmark summary row and opens the next.

**Honesty rule preserved unchanged:** the v0.7 honest-input contract still
holds — an untouched slider writes nothing, not an invented default, and
unanswered state is announced to screen readers. The stepping is presentation
only; it must not auto-fill or auto-submit any subjective value.

## Explicitly not carried over

From the roadmap's own list, plus what the mockups contain that must not ship:

- The floating nav's **"Login" / "Exit" tab** — a Superdesign flow-navigation
  artifact linking between draft pages, not a real logout control. The live
  `BottomNav` keeps its existing six items.
- The settings **light/dark toggle** — decorative; dark-only stays.
- Coach header **search / bookmark / archive** icons and composer **image**
  button — visual stubs, dropped.
- All **placeholder data** in every mockup (BRI 4.2, 1,847 kJ, Dolomites
  2026, archive badge "3", Claude 3.5, v3.4.2, the fabricated session names).
  Real values or honest empty states only.

## Non-goals

- No new data sources, metrics, coach tools, or database migrations. This
  release touches presentation only.
- No resolution of the "Chart consistency" backlog item (one visual grammar
  across all charts) — v0.19 restyles chart _wrappers_ on log/dashboard but
  leaves the underlying chart grammar for a later dedicated pass.
- No light theme.
- No changes to the login page, the `/plan`, `/activity`, `/health`, or
  `/import` pages.
- No accessibility _sweep_ (that was the v0.18 hardening item) — but new and
  restructured markup ships with labels/contrast/focus handled as-you-go, so
  it doesn't regress.

## Delivery & verification

- **One worktree, one branch** (`v0.19-design-refresh`), subagent-driven,
  matching v0.14/v0.15. Task order: (1) shared foundations →
  (2) dashboard → (3) settings → (4) log → (5) journal → (6) coach →
  (7) final whole-branch review. Empty states and skeletons are done within
  each page's task, not separately.
- **Per-task manual visual check.** The repo has **no Playwright or
  visual-regression tooling**, and these are server components rendering real
  data — so the guard against "renders wrong but tests pass" is a manual look
  at each running page (via the `run` skill) before the next task starts.
- **Green gates:** `vitest run`, `tsc --noEmit`, `eslint`, `next build`, and
  `prettier --check` must all pass. Most page logic is unchanged, so the
  suite should stay green; the new tested unit is the `Collapsible` primitive.
- **Release:** version bump + CHANGELOG + roadmap tick in-branch → PR → merge
  → verify main CI green → **tag `v0.19.0` last**, per `RELEASING.md`. The tag
  is the final step, never before merge.

## Risks

- **Component extraction on the dashboard** is a larger diff than pure
  restyling. Mitigated by reusing existing card components unchanged and
  keeping extraction mechanical (move inline JSX to a file, no behavior
  change).
- **No visual regression net.** Mitigated by the per-task manual check and by
  the fact that data flow is unchanged — a broken render is a markup bug, not
  a data bug, and is visible on first load.
- **`?tab=` retirement on Log** could break old bookmarks/links. Low impact
  (internal app, invite-only), and the wellness content is still reachable —
  it just moved from a tab to an always-present collapsible.
