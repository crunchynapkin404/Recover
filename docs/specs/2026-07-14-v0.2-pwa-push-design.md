# v0.2 — Phone & daily loop (PWA + morning push + manual resync)

Status: approved 2026-07-14 · Roadmap phase: v0.2 (docs/ROADMAP.md)

## Goal

Make Recover feel like an app you check every morning: installable on a phone,
a push notification when the morning readiness score is ready, and a one-tap
resync for the moments intervals.icu lags behind its upstream sources.

**Definition of done:** installed on the owner's phone; the morning
notification arrives unattended; a manual resync from the dashboard pulls
fresh data on demand.

## Decisions (with reasons)

- **Minimal hand-rolled service worker**, not Serwist (deviation from the
  original PLAN.md P6): v0.2 needs installability + push, not precaching.
  A ~60-line `public/sw.js` has no build-plugin risk against this Next
  version. Serwist remains an option later if offline mode is wanted.
- **Push fires after the morning sync computes today's readiness** (not at a
  fixed clock time): content is always fresh, no timezone configuration.
  Guards make it at-most-once per day, morning-window only.
- **VAPID keys auto-generated and stored in the database** (private key
  AES-256-GCM-encrypted with the existing `ENCRYPTION_KEY`): zero new
  required env vars for self-hosters.
- **Resync UX = header chip + pull-to-refresh** (user choice): chip works
  everywhere; the pull gesture activates only in the installed app, at
  scroll-top, with `overscroll-behavior-y: contain` so it never conflicts
  with the browser's native pull-to-reload.

## Data model (one migration)

- `push_subscriptions`: id uuid pk, user_id fk cascade, endpoint text unique,
  p256dh text, auth text, user_agent text, created_at. A user may have
  several (multi-device). Rows deleted when the push service returns 404/410.
- `notification_prefs`: id uuid pk, user_id fk unique, morning_push_enabled
  boolean default true, last_morning_push_date date. Row created lazily.
- `app_config`: key text pk, value text, updated_at. Holds
  `vapid_public_key` (plaintext) and `vapid_private_key` (encrypted via
  `src/lib/crypto.ts`). Generated on first access with the `web-push` lib.

## Components

### `src/lib/push.ts`

- `getVapidKeys()` — read-or-create from `app_config` (advisory-lock-free:
  insert with `onConflictDoNothing` then re-read, so concurrent first calls
  converge).
- `sendToUser(userId, payload)` — web-push to every subscription; on 404/410
  delete the row; never throws on per-subscription failure (logs).
- `buildMorningPayload(metrics, wellness)` — title
  `Readiness 66 · Amber`, body cites HRV / RHR / sleep, band-appropriate
  one-liner; `tag: "morning-readiness"`, `data.url: "/"`.
- `maybeSendMorningReadinessPush(userId)` — conditions, in order: prefs
  enabled; server-local hour 04:00–11:59; today's `daily_metrics.readiness`
  exists (calibrating → skip); `last_morning_push_date` ≠ today. On send,
  stamp the date first (at-most-once even if delivery partially fails).

### Sync hook

`runIntervalsSync`/`runStravaSync` callers stay untouched; the scheduler's
success path (`runSchedulerTick`) calls `maybeSendMorningReadinessPush(job.userId)`
after a job completes. Manual resyncs reuse the same path — the guards make
afternoon double-fires impossible.

### PWA shell

- `public/manifest.webmanifest` — name/short_name Recover, `display:
standalone`, `background_color`/`theme_color: #0a0a0a`, icons 192 + 512 +
  maskable (rendered from `public/logo.svg`), start_url `/`.
- `public/icons/` — PNGs incl. `apple-touch-icon` 180.
- `src/app/layout.tsx` — manifest link + `appleWebApp` metadata via the Next
  metadata API (consult `node_modules/next/dist/docs` first per AGENTS.md).
- `public/sw.js` — `push` → `showNotification`; `notificationclick` →
  focus-or-open `data.url`; `fetch` (navigations only) → network with a tiny
  offline fallback response. No precache.
- `src/components/pwa/sw-register.tsx` — client component registering the SW.
- Install affordances: `beforeinstallprompt`-captured button
  (Android/desktop) in settings; iOS Safari un-installed → "Share → Add to
  Home Screen" hint card. iOS web push requires 16.4+ and an installed PWA —
  the hint says so.

### Notifications settings card (`src/components/settings/`)

- Enable toggle: user-gesture → `Notification.requestPermission()` →
  `pushManager.subscribe({ applicationServerKey })` → `POST
/api/push/subscribe`. Disable → unsubscribe + `DELETE`.
- Morning-push on/off preference (server action, `notification_prefs`).
- **Send test notification** button → server sends a canned payload through
  the real pipeline (also the manual verification path).
- Inline warnings: insecure context (push needs HTTPS except localhost); iOS
  not installed.

### Manual resync

- `POST /api/sync/now` (session-authed): per-user rate limit (1 per 120 s,
  in-memory like the MCP token bucket); set the user's pending sync jobs'
  `run_after = now()` (insert incremental jobs if none exist for active
  connections); `await runSchedulerTick()`; return `{ lastSyncAt }`.
- Dashboard header chip `Synced 12m ago ⟳` (max `connections.last_sync_at`):
  tap → spinner → `router.refresh()` on completion; error toast on failure.
- Pull-to-refresh: dashboard scroll container, only when
  `display-mode: standalone` matches and `scrollTop === 0`; threshold ~70 px;
  calls the same endpoint; `overscroll-behavior-y: contain` on the shell.

## Error handling

- Push send failures: per-subscription; 404/410 prunes the row, other errors
  log and continue. The morning stamp is written before sending.
- `/api/sync/now` while a job is already running: tick claims nothing new;
  returns current `lastSyncAt` (the UI still refreshes — honest state).
- No subscriptions / permission denied: settings card reflects state; the
  morning hook is a no-op.

## Testing

Unit (vitest, web-push transport mocked):

- payload wording per band (green/amber/red), metric formatting;
- morning guard: outside window, already-sent today, calibrating, disabled
  prefs, no readiness row;
- subscription lifecycle: subscribe upsert, unsubscribe delete, 410 prune;
- sync-now rate limit (second call within 120 s rejected).

Integration (pg, existing pattern in `tests/`): subscribe → morning hook →
prefs stamped, transport called once; cross-user isolation (user A's sync
never pushes to user B).

Real device (owner): install on phone via tunnel, test-notification button,
next-morning push, pull-to-refresh + chip resync.

## Out of scope (v0.2)

Serwist/offline precache; per-user delivery-time preference (revisit only if
the data-driven timing feels wrong); notification kinds beyond the morning
readiness push (weekly review etc. is v0.4); Android/desktop install
banners beyond the standard prompt.
