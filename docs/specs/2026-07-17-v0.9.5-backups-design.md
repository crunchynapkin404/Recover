# v0.9.5 — Nightly Backups: Design

**Date:** 2026-07-17
**Status:** Approved
**Scope decision:** The roadmap's v0.9.5 "Infrastructure" bundled two
independent pieces. This release ships only the backups; absorbing the
standalone `intervals-icu-mcp` server moves to v0.9.6 (roadmap edit is part
of this release).

## Goal

A Recover instance backs itself up every night with zero configuration
beyond `docker compose up`, and one command proves a backup actually
restores. Done when: `scripts/restore-drill.sh` restores the latest nightly
dump into a clean scratch database unattended and its sanity checks pass.

## Decisions (made during brainstorm)

| Decision     | Choice                                      | Why                                                                                                                                         |
| ------------ | ------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| Destination  | Local named volume only (`recover-backups`) | Simplest; protects against corruption/bad migrations. Disk loss is the Proxmox layer's job. No offsite, no dump encryption.                 |
| Retention    | Newest 14 dumps, count-based                | Two weeks of restore points; dumps are small.                                                                                               |
| Verification | Scripted drill, run manually                | One command spins scratch Postgres, restores, sanity-checks, tears down.                                                                    |
| Mechanism    | Backup sidecar container (approach A)       | Version-matched `pg_dump` via `postgres:16-alpine` + busybox crond; all config versioned in the repo; no host state; no app-image coupling. |

Rejected: in-app scheduler job (app image has no `pg_dump`; couples app to
DB major version); host cron (unversioned host state, lost on rebuild).

## Architecture

One new default-on service in `docker-compose.yml` (deliberately **not**
profile-gated — a self-hoster who forgets a profile flag has no backups):

```yaml
backup:
  image: postgres:16-alpine
  restart: unless-stopped
  environment:
    PGHOST: db
    PGUSER: recover
    PGPASSWORD: ${POSTGRES_PASSWORD:-recover}
    PGDATABASE: recover
    BACKUP_KEEP: ${BACKUP_KEEP:-14}
  volumes:
    - recover-backups:/backups
    - ./scripts/backup.sh:/backup.sh:ro
  # crond in foreground; the crontab line is installed at container start
  command: sh -c 'echo "30 3 * * * sh /backup.sh" | crontab - && crond -f -l 8'
  depends_on:
    db:
      condition: service_healthy
```

Plus the named volume `recover-backups` in the top-level `volumes:` block.

Schedule: **03:30 nightly**, clear of the 04:00 weekly-review slot and the
05:00 intervals.icu sync. `pg_dump` is online; overlap would be harmless,
this just keeps the logs clean.

## Components

### `scripts/backup.sh`

POSIX sh (busybox-compatible — no bashisms). Behavior:

1. `pg_dump -Fc` (custom format: compressed, `pg_restore`-able,
   version-tolerant) to `/backups/recover-YYYYMMDD-HHMMSS.dump.tmp`.
2. On success, atomically `mv` to the final `.dump` name — a partial dump
   can never masquerade as a backup.
3. **Rotation only after a successful dump** (a failing dump must never
   delete old backups): list `recover-*.dump` sorted by name (timestamps
   sort lexicographically), delete all but the newest `$BACKUP_KEEP`.
4. Any failure: remove the `.tmp`, log to stdout, exit non-zero. Output is
   visible via `docker compose logs backup`.

### `scripts/restore-drill.sh`

Run from the repo root on the host. Unattended — no prompts. Behavior:

1. Start a disposable `postgres:16-alpine` container (unique name, no
   published ports — everything runs via `docker exec`) with the
   `recover-backups` volume mounted read-only; the host never needs the
   volume path.
2. Inside it, locate the newest `recover-*.dump` and wait for Postgres
   readiness (`pg_isready` loop with a timeout).
3. `pg_restore` the dump into the scratch server via `docker exec`.
4. Sanity checks, each fatal if it fails:
   - expected core tables exist (`users`, `wellness_daily`, `activities`,
     `daily_metrics`),
   - `users` has ≥ 1 row,
   - `wellness_daily` has > 0 rows,
   - print the newest `wellness_daily.date` (human freshness check — not
     asserted, printed).
5. Exit 0 with a one-line PASS summary, or exit 1 on the first failure.
6. `trap`-guaranteed teardown: the scratch container is removed on every
   exit path, including Ctrl-C and failures.

The drill script **is** the backup verification. There is no scheduled
auto-verify in this release (revisit if a drill ever fails in anger).

## Error handling summary

- Partial dumps: impossible to mistake for backups (tmp-then-rename).
- Failing dumps: never trigger rotation; old backups survive; failure
  visible in container logs and non-zero crond job exit.
- Drill: fails loudly (exit 1, first failing check named); scratch
  container always cleaned up.
- DB down at 03:30: `pg_dump` fails, logged, retried next night. No
  retry-loop complexity — nightly cadence is the retry.

## Testing

Shell + compose only; no vitest surface. Verification is exercising the
real thing, in the plan:

1. Bring up the stack with the new service; trigger `backup.sh` once
   manually (`docker compose exec backup sh /backup.sh`) and confirm a
   `.dump` appears in the volume.
2. Run `scripts/restore-drill.sh`; confirm PASS.
3. Rotation test: create >14 dummy `recover-*.dump` files in the volume,
   run `backup.sh`, confirm only the newest 14 remain and the dummies aged
   out first.
4. Failure test: run `backup.sh` with a wrong `PGPASSWORD`; confirm
   non-zero exit, no `.tmp` left behind, and no old dumps deleted.

## Out of scope

- Offsite/S3 sync and dump encryption (no offsite in this release).
- Scheduled auto-verify of backups.
- Backing up anything other than Postgres (the DB is the only state; the
  app container is stateless).
- The `intervals-icu-mcp` absorption (moved to v0.9.6).
