# v0.7 — Score Integrity Design

> Stop the app from knowing things it doesn't know. Two fixes sharing one
> theme: the journal fabricates subjective input, and illness silently poisons
> the baselines the readiness score is measured against.

## Problem

### The journal invents answers

`journal-form.tsx` initializes `energy = 7`, `soreness = 4`, `stress = 4` and
submits all three as hidden inputs on every save. Ticking one behavior tag and
hitting save writes `energy1_10 = 7` — a number the athlete never gave — stored
indistinguishably from a real answer.

`mood` on the same form is correct: it submits `""` when unselected, the
action's zod `preprocess` turns that into `undefined`, and `upsertWellness`
skips it. Three fields out of four lie.

This is the same defect class as the placeholder body-battery curve: invented
data presented as measured. It's worse here because it's invisible — it lives in
the database rather than on screen, and the journal replays it back to the user
as fact.

### Illness poisons the baselines

`BASELINE_WINDOW_DAYS = 60`. Five days of flu put five crushed HRV values into
the trailing baseline for the next two months. Once the athlete has _recovered_,
their baseline mean is artificially low, so a normal HRV reads as a positive
z-score and the score is falsely green — for eight weeks, silently, with no way
to say "those days weren't me".

The readiness engine is otherwise careful: personal baselines, honest
`calibrating` state, renormalized weights for missing components. The
contamination undoes that care.

## Scope

1. The journal stops writing subjective values the athlete didn't provide.
2. Athletes can flag abnormal days; flagged days are excluded from baselines.

Not in scope: validating the readiness weights against reported feel (needs
trustworthy ground truth this release starts collecting — see Hand-off), cycle
phase, and everything currently in v0.8+.

## Part 1 — Honest Subjective Input

### Write path: already correct

Part 1 needs **no** changes to `upsertWellness`, `wellnessSchema`, or
`log_wellness`. The merge semantics (`if (input.energy1_10 != null)
values.energy1_10 = …`) and the zod `preprocess` already do the right thing with
an absent value. **Only the form lies.** Fixing the form fixes the pipeline.

(Part 2 does extend both to carry `dayFlags` — see its Write Path section.)

### Form state

`energy`/`soreness`/`stress` become `number | null`, where `null` means
unanswered:

```ts
const [energy, setEnergy] = useState<number | null>(todayEntry?.energy ?? null);
```

The hidden input submits `""` when null, mirroring how `mood` already works:

```tsx
<input type="hidden" name="energy" value={energy ?? ""} />
```

`switchDay` loads `entry.energy ?? null` — a past day with no stored value shows
as unanswered rather than defaulting.

### Marking "answered"

A range input's thumb must physically sit somewhere, so an unanswered slider
renders at a neutral resting position (7 / 4 / 4 — the current defaults) while
displaying `—` and carrying a greyed style.

**"Answered" is tracked on interaction, not on value change.** If the thumb
rests at 7 and the athlete taps exactly 7 meaning it, no `change` event fires —
a naive `onChange` check would silently discard their real answer:

```tsx
onPointerDown={() => setEnergy((v) => v ?? 7)}
onKeyDown={() => setEnergy((v) => v ?? 7)}
onChange={(e) => setEnergy(Number(e.target.value))}
```

`pointerdown` commits the resting value, then `change` (if the thumb moved)
overwrites it with the real one. Ordering is guaranteed: `pointerdown` precedes
`input`/`change`. `keydown` covers the arrow-key-at-boundary case, where the
value can't change but the intent is real.

### Accessibility

The unanswered state must not be a visual convention only. Each slider carries
an accessible name that includes its state (e.g. `aria-label="Energy: not
answered"` → `aria-label="Energy: 7 of 10"` once answered). The `—` is decorative;
the accessible name is authoritative.

### Historical data

**No migration. Nothing is deleted.** Pre-fix rows are indistinguishable
per-field: `(7, 4, 4)` is the untouched signature, but a partially-touched save
wrote a real energy alongside a fabricated soreness, and some athletes genuinely
felt 7/4/4. There is no rule that separates them without destroying real
answers.

The cost is accepted knowingly: the journal keeps displaying possibly-invented
numbers for pre-v0.7 days. The alternative deletes truth to hide a lie.

## Part 2 — Day Flags and Baseline Exclusion

### The concept

An athlete states a **fact about their day**; the engine decides what it means
for the math. They are never asked to operate the statistics.

```ts
export type DayFlag = "ill" | "travel" | "altitude";
```

**Every day flag excludes that day from baselines.** There is no per-flag rule
and no exclusion set — a flag _is_ the statement "this day doesn't represent my
normal physiology". Keeping the rule total is what makes it explainable.

### Flags are not behavior tags

|              | Behavior tags              | Day flags                               |
| ------------ | -------------------------- | --------------------------------------- |
| Examples     | ☕ Caffeine, 🍷 Alcohol    | 🤒 Ill, ✈️ Travel, 🏔️ Altitude          |
| Meaning      | a choice you want measured | a fact that invalidates the measurement |
| Baselines    | included                   | excluded                                |
| Correlations | analyzed                   | never seen                              |

Alcohol crushes HRV, but it stays a _behavior_: excluding it would delete the
very insight `computeTagCorrelations` exists to find. The dividing line is
whether you want the day studied or discounted.

Because flags live in their own column, `computeTagCorrelations` never sees
them — no "🤒 Ill: −22% readiness" shipped as an insight, which is both
trivially true and noise.

### The engine does not change

`computeReadiness` accepts `hrvBaseline: number[]` and has no idea where the
numbers came from. Exclusion is therefore a **filter where the baseline array is
assembled** (`metrics.ts:37`), not an engine concern:

```ts
const baseline = rows.filter(
  (r) =>
    r.date < day.date &&
    r.date >= baselineFloor &&
    !isBaselineExcluded(r.dayFlags)
);
```

`src/lib/readiness.ts` and every assertion in `readiness.test.ts` are untouched.
That the 169-line pure core needs no edit is the evidence the boundary is right.

### Flagged days are still scored

Exclusion governs **baseline membership only**. An ill day must still produce a
red score — that is the app working. It must merely not drag the 60-day
reference down and make the athlete read falsely green next month.

This is the single most likely thing to be implemented backwards, so it is
called out here and enforced by test.

### Retroactive flagging already works

`upsertWellness` already ends with `computeDailyMetrics(userId, input.date)`,
which recomputes that date forward using correct trailing baselines. Flagging
last Tuesday as ill today automatically repairs every score since. No new
machinery — the existing hook is simply fed better data.

### Over-flagging degrades honestly

Flag enough days and the surviving baseline falls below `MIN_BASELINE_DAYS = 14`,
so `computeReadiness` returns `calibrating` rather than a confident wrong
number. This falls out of the existing engine for free and needs no guard.

### Write path

Unlike Part 1, flags are a new field and must be threaded through:

- `wellnessSchema` gains `dayFlags`, mirroring how `tags` is parsed today
  (`z.preprocess` → `JSON.parse` → array), validated against the known flag
  values.
- `WellnessWriteInput` gains `dayFlags?: DayFlag[]`, and `upsertWellness` gains
  the matching `if (input.dayFlags != null) values.dayFlags = input.dayFlags;`
  line — same merge semantics as every other field.

Clearing every flag submits `[]`, which is written and means "a normal day".
Omitting the field entirely leaves the stored value untouched, per the existing
merge contract.

### Coach visibility

`get_wellness` returns `dayFlags`. The coach knowing an athlete was ill
materially changes its advice, and the field is already being read.

Writing flags via `log_wellness` is out of scope — no demonstrated need. Adding
`dayFlags` to `WellnessWriteInput` does not expose it on the tool; the tool
chooses what it passes.

## Schema

```sql
ALTER TABLE wellness_daily ADD COLUMN day_flags jsonb;
```

```ts
// schema.ts, on wellnessDaily, beside `tags`
dayFlags: jsonb("day_flags").$type<DayFlag[]>(),
```

`null` and `[]` both mean "a normal day" — no backfill.

### Types module

`src/lib/day-flags.ts`, following the `strava-description-fields.ts` precedent
from v0.6.2: **zero imports**, because `db/schema.ts` needs `DayFlag` and
`metrics.ts` imports `db/schema` — a type defined in either would cycle.

Exports: `DayFlag`, `ALL_DAY_FLAGS` (key + emoji + label, UI-ordered),
`sanitizeDayFlags(input: unknown): DayFlag[]`, and
`isBaselineExcluded(flags: DayFlag[] | null | undefined): boolean`.

### Untrusted input

Flags arrive from a browser and land in JSONB. `sanitizeDayFlags` keys off
`ALL_DAY_FLAGS` and keeps only known values, dropping everything else, so a
client cannot write arbitrary JSON into the column. Unknown values already
stored are ignored at read time rather than throwing.

## Journal UI

A new row below the behavior tags, visually distinct from them:

```
Anything unusual today?
  [ 🤒 Ill ]  [ ✈️ Travel ]  [ 🏔️ Altitude ]

  Flagged days still get a score — they're just left out of your baselines.
```

The helper line is required, not decorative: without it, "Ill" looks like a
behavior tag and the athlete cannot tell what flagging does.

## Testing

**`metrics.ts` (the substance):**

- A flagged day is absent from the baselines of every subsequent day.
- A flagged day still receives its own `readiness` and `band` — never null on
  account of its flag.
- Flagging enough days drops later days to `calibrating`.
- Unflagging a day recomputes it back — exclusion is not one-way.
- `null` and `[]` day flags both behave as a normal day.

**Form:**

- An untouched slider writes nothing (the column keeps its prior value or stays
  null).
- A slider deliberately set to its resting value writes that value — the
  `pointerdown` path, i.e. a genuine 7 is not discarded.
- Loading a day with a stored value shows it as answered; a day without shows
  unanswered.

**Engine:**

- Every existing `readiness.test.ts` assertion passes unchanged. This is the
  check that the engine really didn't move.

**Types:** `sanitizeDayFlags` drops unknown values; `isBaselineExcluded` is true
for any non-empty known flag set and false for `null`/`[]`.

## DoD

- [ ] Saving the journal without touching a slider writes no energy/soreness/stress
- [ ] A slider deliberately set to 7 writes 7
- [ ] Unanswered sliders read `—` and announce their state to a screen reader
- [ ] Day flags can be set and cleared from the journal
- [ ] A flagged day still gets a readiness score
- [ ] A flagged day does not appear in later days' baselines
- [ ] Flagging a past day recomputes every score after it
- [ ] Flagging most of the window returns `calibrating`, not a wrong number
- [ ] `readiness.ts` is unmodified and its tests pass unchanged
- [ ] `get_wellness` returns day flags
- [ ] No existing subjective data is deleted or altered

## Hand-off to Score Validation

This release exists partly to start collecting trustworthy ground truth. The
validation work (does readiness predict reported energy? should the weights be
personal?) needs it and cannot proceed without it.

Constraints that work inherits:

- **Only trust rows from v0.7.0 onward.** Earlier `energy1_10` /
  `soreness1_10` / `stress1_10` values may be fabricated defaults and are not
  separable. That spec defines the cutoff constant; defining one here would be
  unused code.
- **`mood` is nominal, not ordinal.** `happy | neutral | exhausted | injured |
tired` has no scalar axis — "injured" is orthogonal to the rest. It cannot
  serve as a regression target without redesign.
- **Ship behind an honesty gate**, following `MIN_BASELINE_DAYS` and
  `computeTagCorrelations`'s `MIN_EVENTS = 5`: say "not enough answered
  check-ins yet" rather than reporting a correlation from thin data.

## Roadmap Renumbering

This inserts ahead of the previously-planned v0.7. Body Intelligence → v0.8,
Journal Evolution → v0.9, Data-source freedom → v0.10, Wearable connectors →
v0.11.

The argument for going first: Body Intelligence's energy bank and biological age
both consume readiness and its baselines. Building them on an unvalidated,
contaminated foundation compounds the error into features that are harder to
walk back.

Part 1 overlaps v0.9 (ex-v0.8) Journal Evolution, which also touches the journal.
It ships here anyway: it is a correctness fix, and every week it waits is a week
of unusable ground truth.

## Non-Goals (v0.7)

- **Weight validation / personal weights** — needs the data this release starts
  collecting. See Hand-off.
- **Cycle phase** — independent; its own spec.
- **Deleting or repairing historical subjective data** — cannot be done without
  destroying real answers.
- **Per-flag baseline rules** — every flag excludes; a total rule is
  explainable, a per-flag matrix is not.
- **Auto-detecting illness** from HRV/RHR — circular: the score would infer the
  fact that corrects the score.
