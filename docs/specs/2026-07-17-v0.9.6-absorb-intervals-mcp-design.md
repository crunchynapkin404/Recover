# v0.9.6 — Absorb intervals-icu-mcp: Design

**Date:** 2026-07-17
**Status:** Approved

## Goal

Fold a curated slice of the standalone `intervals-icu-mcp` server's tools into
Recover's built-in MCP + coach registry, so the standalone Python server and
its two deploy repos can be retired. Done when: the curated tools work
through both Recover's MCP endpoint and the in-app coach, and the standalone
server is no longer needed for any workflow the owner uses.

## Why absorb (not proxy)

The standalone server is **single-athlete** — one intervals.icu API key from
env config. Recover is multi-user with per-user encrypted keys in the
`connections` table. Proxying to the Python server would require rewriting
its auth model anyway, so there is no shortcut: the value is in the tool
definitions and intervals.icu's response shaping, which we port to
TypeScript. The standalone stays as a **reference** (`/home/vscode/intervals-icu-mcp`)
during the build; nothing is copied without a test.

## Decisions (from brainstorm)

| Decision          | Choice                                                      |
| ----------------- | ----------------------------------------------------------- |
| Scope             | Curated ~24 tools (not full 59-tool parity)                 |
| Write safety      | New `write:icu` scope; in-app coach sees writes too         |
| Resources/prompts | Drop all except workout-syntax, ported as a tool            |
| Mechanism         | One file per tool + shared `icuFetch` helper (repo pattern) |
| Naming            | `icu_` prefix — "hits live intervals.icu API now"           |

## Architecture

### Shared helper — `src/lib/connectors/intervals.ts`

Add one low-level helper beside the 9 typed fetchers already there:

```ts
async function icuFetch(
  connection: Connection, // the user's intervals_icu connections row
  path: string, // e.g. `/athlete/{id}/events`
  opts?: { method?: string; query?: Record<string, unknown>; body?: unknown }
): Promise<unknown>; // parsed JSON, or throws IcuError with status
```

It centralizes the cross-cutting work: decrypt the API key (AES-256-GCM, via
`lib/crypto.ts`, same as existing fetchers), HTTP Basic auth
(`API_KEY:<key>`) against `https://intervals.icu/api/v1`, substitute
`connection.externalAthleteId` into `{id}` path segments, serialize query
params, and map non-2xx responses to a typed error. Existing typed fetchers
are NOT rewritten in this release — `icuFetch` is additive.

### One file per tool — `src/lib/tools/icu-*.ts`

Each curated tool is its own file following the exact `ToolDefinition<T>`
pattern of the existing 24 (see `src/lib/tools/get-athlete-profile.ts`):
a zod `parameters` schema, an `execute(args, ctx)` that (1) loads the user's
active `intervals_icu` connection, (2) calls `icuFetch`, (3) shapes the
result for an LLM (port the standalone's `response_builder.py` trimming per
tool — raw intervals.icu JSON is too noisy for a model), and a
`ToolDefinition` export carrying `name`, `description`, `parameters`,
`execute`, and `scope` (writes only).

Registration: add each to the `allTools` array in
`src/lib/tools/registry.ts`. No new machinery — both `buildAiSdkTools` (coach)
and the MCP route iterate `allTools`, so every tool lights up in both
surfaces automatically.

### Connection gate

Every tool first loads the user's `intervals_icu` connection; if absent or
`status !== "active"`, it returns `{ error: "No active intervals.icu
connection" }` — a usable message for the model, never a thrown exception.

## The curated tool set (24 new; registry 24 → 48)

`icu_` prefix throughout. Paths below are the real intervals.icu v1 endpoints
(verified against the standalone's `client.py`); `{id}` = athlete id.

| Tool                        | Method + path                                       |
| --------------------------- | --------------------------------------------------- |
| `icu_create_event`          | POST `/athlete/{id}/events`                         |
| `icu_update_event`          | PUT `/athlete/{id}/events/{eventId}`                |
| `icu_delete_event`          | DELETE `/athlete/{id}/events/{eventId}`             |
| `icu_bulk_create_events`    | POST `/athlete/{id}/events/bulk`                    |
| `icu_bulk_delete_events`    | POST `/athlete/{id}/events/bulk-delete`             |

| `icu_apply_training_plan`   | POST `/athlete/{id}/events/apply-plan`              |
| `icu_update_wellness`       | PUT `/athlete/{id}/wellness/{date}`                 |

"apply an intervals.icu **workout-library folder** to your intervals.icu

`generate_training_plan` / living-week system.

### Explicitly dropped (until missed)

Gear (create/update/delete/list/reminders), custom items, fit/gpx/file
downloads, sport-settings create/delete/apply, manual-activity bulk create,
`get_upcoming_workouts` (≈ existing `get_planned_workouts`), the full-text
`search-full` and `activities-around` reads, the athlete-level curve
endpoints (Recover already exposes `get_power_curve`/`get_pace_curve`), and
all 7 standalone prompts. Adding any later is a one-file change.

## Auth & scope wiring

- Add `"write:icu"` to the `Scope` union and the `parseScopes` guard in
  `src/lib/mcp/token-auth.ts`.
- Surface `write:icu` as a mintable scope in the token-creation UI
  (Settings → MCP API Tokens), alongside the existing write scopes.
- The 11 write tools set `scope: "write:icu"`; the 13 reads omit `scope`
  (defaults to `read`).
- The in-app coach runs under the user's own session and sees all tools
  regardless of scope (unchanged) — so it can place and move workouts on the
  real intervals.icu calendar, which is the payoff.

## Strava firewall — not implicated

These tools call the intervals.icu API directly. The Nov-2024 Strava API
agreement bars **Strava's own API** data from AI contexts; intervals.icu data
is unaffected and always has been (Recover's coach already reads
intervals.icu-synced data). No `ne(provider,'strava')` clause applies here —
that clause is for Recover's DB queries, not intervals.icu API calls.

## Testing

- **Per-tool unit tests** (`src/lib/tools/icu-*.test.ts`): mock `icuFetch`,
  assert the output shaping and the connection-missing error path. Follow the
  style of `src/lib/connectors/intervals.test.ts`.
- **Scope enforcement test**: a `read`-only token is rejected from a
  `write:icu` tool through the MCP auth path; a `write:icu` token is accepted.
- **`icuFetch` unit tests**: path-id substitution, query serialization, auth
  header, error mapping — with `fetch` mocked.
- **Manual live smoke** (documented, not CI): one create → get → delete event
  round-trip against the real intervals.icu API with the owner's key.
- No schema changes, no migration. Existing suite (438 tests) stays green.

## Implementation sequencing (for the plan, not a design change)

24 tools is large for one plan. Build in reviewable thematic batches:

1. `icuFetch` helper + `write:icu` scope + token UI (foundation; nothing
   user-visible yet, but unit-tested).
2. Event tools (create/update/delete/get/bulk/duplicate/apply-plan +
   get_calendar_events) — the highest-value cluster.
3. Activity/wellness/sport writes + their reads (update_activity,
   add/get messages, update_wellness, get/update sport-settings).
4. Read-only histograms + search + activity intervals + workout library.
5. `get_workout_syntax` tool + docs/retirement/release chores.

Each batch ends with a green suite and is independently reviewable.

## Out of scope

- Rewriting the 9 existing typed intervals.icu fetchers to use `icuFetch`.
- The dropped tool families above.
- MCP resources and prompts (only workout-syntax survives, as a tool).
- Any change to Recover's own training-plan / living-week system.
