# v0.22 — Wellness Fitness Metrics: Design

**Date:** 2026-07-22
**Status:** Approved

## Goal

intervals.icu already sends `vo2max`, `rampRate`, and per-sport `pMax`/`wPrime`
in the daily wellness payload we fetch and store every night — none of the
four are extracted into typed columns today, so they sit inert in
`wellness_daily.raw`. Bio-Age has had a real `vo2max` scoring factor since
v0.13 that has been permanently fed `null` because no provider populated it;
`health/page.tsx` even carries a comment claiming "no provider wired for
VO2max yet," which is factually wrong — the data has been there since day
one, just unextracted. `eftp` is the one sibling field that already made it
into a typed column and a merge-policy bucket, but never got a page-level
display. Done when: VO2max feeds Bio-Age's real score for any athlete whose
Garmin/intervals.icu data includes it, and eFTP/pMax/wPrime/rampRate are
visible together on the Log page next to the PMC chart they're computed
alongside.

## Decisions

| Decision | Choice |
| --- | --- |
| Schema change | Migration adds `vo2max`, `ramp_rate`, `p_max`, `w_prime` as nullable `real` columns on `wellness_daily`, mirroring the existing `eftp` column exactly |
| vo2max priority bucket | `PHYSIOLOGY` (`manual > whoop > oura > intervals_icu > apple_health`) — a measurement other wearables could plausibly report later, even though only intervals.icu populates it today |
| rampRate / pMax / wPrime priority bucket | `LOAD` (`intervals_icu` only) — same bucket as `eftp`/`ctl`/`atl`; these are outputs of intervals.icu's own training model, not measurements |
| Bio-Age wiring | `health/page.tsx` passes the athlete's latest non-null `vo2max` instead of the hardcoded `null`; no changes to `biological-age.ts` — the scoring factor already exists and already handles `null` |
| New UI surface | A stat row inside the Log page's existing "Performance Trends" collapsible (next to the PMC chart), not a new page or new collapsible section |
| Displayed stats | eFTP, pMax, wPrime as three small stats; rampRate as a sign-aware trend label next to the existing CTL number (e.g. "Ramping" / "Tapering" / "Steady") |
| Empty state | Row renders nothing (or an honest placeholder) for any stat that's null — no zero/fabricated defaults, consistent with the project's honesty principle |
| Coach/MCP exposure | Out of scope this round — matches the precedent set by `eftp`, which is fully wired end-to-end but has no page UI and no coach-tool exposure beyond `get_fitness_summary` |

## Architecture

### Connector — `src/lib/connectors/intervals.ts`

`fetchDailyWellness()`'s per-day mapping gains four fields read straight off
the same raw `row` object `eftp` already comes from:

- `vo2max: num(row.vo2max)`
- `rampRate: num(row.rampRate)`
- `pMax: num(row.sportInfo?.pMax)`
- `wPrime: num(row.sportInfo?.wPrime)`

`IntervalsWellnessDay` gains the matching four nullable fields alongside the
existing `eftp`.

### Schema — `src/lib/db/schema.ts`

New additive migration: `vo2max`, `rampRate`, `pMax`, `wPrime` as nullable
`real` columns on `wellnessDaily`, positioned next to `eftp`. No backfill —
historical raw JSON stays queryable directly if a future backfill is ever
wanted, but this migration only adds columns for new syncs going forward.

### Merge policy — `src/lib/wellness-merge.ts`

`WellnessPatch` gains `vo2max`, `rampRate`, `pMax`, `wPrime`. `FIELD_PRIORITY`
assigns `vo2max` to the existing `PHYSIOLOGY` array and the other three to
the existing `LOAD` array — no new priority arrays needed, this reuses the
two buckets `eftp`/`hrvMs` already established in v0.11.

### Sync — `src/lib/sync/intervals-sync.ts`

The existing `applyWellnessPatch(userId, day.date, { ... }, "intervals_icu",
day.raw)` call gains `vo2max: day.vo2max, rampRate: day.rampRate, pMax:
day.pMax, wPrime: day.wPrime` alongside its current fields. No new call
sites, no change to batching or `computeDailyMetrics`.

### UI — `src/app/health/page.tsx`

Replace the hardcoded `vo2max: null` (and its stale comment) with the same
"most recent non-null" lookup already used just above it for `restingHr` /
`hrvMs` / `bodyFatPct`. `BioAgeCard` and `biological-age.ts` need no changes.

### UI — `src/app/log/page.tsx`

Inside the "Performance Trends" `Collapsible` (currently PMC chart +
`RangeTabs`, only rendered when `wellness.some((w) => w.ctl != null)`), add a
stat row reading the latest non-null `eftp`/`pMax`/`wPrime` and rendering
them as three small labeled stats, plus a rampRate-derived trend label near
the existing CTL/ATL/TSB numbers. Each stat independently hides itself if
its value is null — a fresh connection with zero synced days still renders
the existing honest empty state for the whole section, unchanged.

## Batches (each ends green and reviewable)

1. **Data layer**: migration, connector extraction, `WellnessPatch`/
   `FIELD_PRIORITY` extension, `intervals-sync.ts` wiring, mapping + merge
   unit tests.
2. **UI**: Bio-Age real `vo2max` wiring on the health page; Log page stat
   row + rampRate trend label; component tests including the null/partial
   data case.
3. **Release chores**: version bump to 0.22.0, CHANGELOG entry, ROADMAP
   tick — same pattern as every prior release.

## Testing

- Connector: fixture-based test confirming `vo2max`/`rampRate`/`pMax`/
  `wPrime` extraction from a raw wellness-day fixture, null when absent.
- Merge policy: a `LOAD`-bucket rejection test for `rampRate`/`pMax`/
  `wPrime` from a non-`intervals_icu` source, mirroring the existing
  `eftp`/`ctl` tests; a `PHYSIOLOGY`-bucket acceptance test for `vo2max`
  from a hypothetical higher-priority source.
- Health page: Bio-Age receives a real number when the latest wellness row
  has one, `null` when none of the recent rows do.
- Log page: stat row renders all four values when present, hides
  individual nulls without breaking layout, and the whole section still
  falls back to the existing empty state when no `ctl` data exists at all.

## Out of scope

- Activity-level training analytics (zone-time distribution, power-duration
  model, efficiency/decoupling, `strain_score` reconciliation) — separate
  initiative, specced independently.
- `sleepQuality`, `spO2`, `tempRestingHR`/`tempWeight` data-quality flags,
  `atlLoad`/`ctlLoad` redundancy check — evaluated in the same audit but not
  part of this slice.
- Coach/MCP tool exposure for any of these four fields.
- Backfilling historical rows from stored raw JSON into the new columns.
- Any new chart or visualization — this ships as stat numbers/labels only.
