# v0.5a — Artifacts Engine Design

> Sub-project A of v0.5 (Training Intelligence). Ships independently; enables
> Weekly Review (5b) and Training Plans (5d) to render inline charts.

## Problem

The coach returns plain text. When discussing power curves, load trends, or
weekly comparisons, the user gets numbers in prose. An inline chart is worth a
hundred tokens — and structured data unlocks downstream features (plan tables,
weekly comparison bars, CTL projections).

## Architecture

```
Coach LLM → tool_call: render_chart({type, series, title, ...})
         → tool_result: {chartId, spec} (stored in toolCalls column)
         → assistant message can reference the chart naturally
         → client detects artifact in tool results → renders <ArtifactCard>
         → collapsed: 80px mini SVG preview with title
         → expanded: full chart (320px) with axes, legend, hover
```

MCP clients receive the raw structured JSON spec — they render however they
choose (or pass it to a UI).

## Principles

1. **No new tables** — artifacts live inside `chat_messages.toolCalls` as they
   already do for any tool result. The chart spec *is* the artifact.
2. **No chart library** — pure hand-rolled SVG, same approach as existing
   `StreamChart`, `PmcChart`, `WellnessTrends`. Zero bundle impact.
3. **No server-side rendering** — server returns typed JSON; client renders SVG.
   This keeps the tool fast (no canvas/puppeteer) and the output portable.
4. **Extensible** — new chart types are just new `case` branches in the client
   renderer. The schema uses a discriminated union on `type`.

## Tool Definition

```ts
// src/lib/tools/render-chart.ts

import { z } from "zod";

const seriesSchema = z.object({
  label: z.string(),
  data: z.array(z.object({
    x: z.union([z.string(), z.number()]),
    y: z.number(),
  })),
  color: z.string().optional().describe("Tailwind color name or hex. Defaults to theme palette."),
  style: z.enum(["solid", "dashed", "area"]).default("solid"),
});

export const parameters = z.object({
  type: z.enum(["line", "bar", "area", "table"]).describe(
    "Chart type: line (time series), bar (categorical comparison), area (filled band), table (structured rows)."
  ),
  title: z.string().describe("Short chart title shown above the visualization."),
  series: z.array(seriesSchema).min(1).describe("Data series to plot."),
  xLabel: z.string().optional().describe("X-axis label."),
  yLabel: z.string().optional().describe("Y-axis label."),
  annotations: z.array(z.object({
    x: z.union([z.string(), z.number()]),
    label: z.string(),
    color: z.string().optional(),
  })).optional().describe("Vertical markers (e.g., race day, today)."),
});

// execute returns the spec as-is — the client renders it.
async function execute(args, ctx) {
  return {
    artifact: true,
    chartId: crypto.randomUUID(),
    spec: args,
  };
}

export const renderChart: ToolDefinition<typeof parameters> = {
  name: "render_chart",
  description:
    "Render an inline chart or table in the conversation. The athlete sees a " +
    "visual preview that expands on click. Use for trends, comparisons, and " +
    "structured data — anything better shown than described.",
  parameters,
  execute,
};
```

**Scope:** `read` (default) — rendering a chart doesn't write anything.

## Chart Types

| Type | Visual | When to use |
|------|--------|-------------|
| `line` | SVG polylines with optional area fill | CTL/ATL over time, HRV trend, pace improvement |
| `bar` | Vertical bars, grouped if multiple series | Weekly load comparison, sessions per day |
| `area` | Filled region between zero and line (or between two series) | TSB band, energy balance |
| `table` | Structured rows/columns with optional highlight | Best efforts, plan week overview, workout schedule |

## Client Component

```tsx
// src/components/chat/artifact-card.tsx

interface ArtifactCardProps {
  spec: ChartSpec;
  expanded?: boolean;
}

// Collapsed: ~80px tall, shows title + mini SVG (simplified, no axes)
// Expanded: ~320px tall, full SVG with axes, labels, legend, hover tooltip
// Click toggles expanded state with smooth height transition
```

### Rendering rules
- **Colors:** Default palette cycles through `emerald-500`, `sky-500`,
  `amber-500`, `rose-500`, `violet-500`. Override via `series[].color`.
- **Responsive:** SVG viewBox scales to container width. Y-axis auto-scales
  from data min/max with 10% padding.
- **Collapsed mode:** Strip axes, labels, legend. Show only the shape (sparkline
  aesthetic). Add title as a small label top-left.
- **Hover (expanded):** Show crosshair + tooltip with exact values on desktop.
  Touch: tap-and-hold.

## Message Integration

In `src/components/chat/message.tsx`, when rendering tool results:

```ts
if (toolResult.artifact === true) {
  return <ArtifactCard spec={toolResult.spec} />;
}
```

No changes to the chat route or message storage — the tool result already
persists in `chat_messages.toolCalls`.

## System Prompt Addition

Add to the coach system prompt (after tools section):

```
## Artifacts

You have a `render_chart` tool. Use it when:
- Comparing numbers over time (trend lines are always clearer than lists)
- Showing weekly/daily load breakdown
- Presenting structured data like best efforts or plan blocks
- The athlete asks "show me" or "what does it look like"

Do NOT use it for:
- Single numbers (just state them)
- Simple yes/no answers
- When the data has fewer than 3 points (just say the values)
```

## Testing Strategy

1. **Unit:** `render-chart` tool returns correct envelope structure
2. **Unit:** `artifact-card` renders SVG for each chart type (line, bar, area, table)
3. **Integration:** `registry.test.ts` count assertion + param validation
4. **Visual (manual):** Seed a conversation with artifact tool results, verify
   collapsed/expanded states in the browser

## Non-Goals

- Interactive chart editing (zoom, pan) — v0.5 is display-only
- Export to PNG/PDF — future if demand
- Custom chart types beyond line/bar/area/table — extensible but not now
- Server-side SVG generation (for email previews) — not needed yet
