# v0.22 — Wellness Fitness Metrics Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Extract `vo2max`, `rampRate`, and per-sport `pMax`/`wPrime` from the intervals.icu daily wellness payload we already fetch nightly (currently unextracted, sitting inert in `wellness_daily.raw`) into typed columns, wire `vo2max` into Bio-Age's dormant scoring slot, and surface eFTP/pMax/wPrime/rampRate as a new stat row on the Log page.

**Architecture:** Mirrors the existing `eftp` field end-to-end at every layer — connector extraction → per-field merge policy → sync wiring → typed column — then two independent UI wiring changes (health page, log page). No new architecture; four fields added to five existing files plus one new small presentational component.

**Tech Stack:** Next.js 16 (App Router, RSC), Drizzle ORM + Postgres, Vitest, TypeScript.

## Global Constraints

- All wellness writes go through `applyWellnessPatch`/`mergeWellnessPatch` (`src/lib/wellness-merge.ts`) — never a raw upsert. Every new field needs a `FIELD_PRIORITY` entry or the merge silently drops it (`rank()` returns `Infinity` for undeclared fields).
- Training-load-model outputs (`rampRate`, `pMax`, `wPrime`, matching `eftp`/`ctl`/`atl`) are `intervals_icu`-only in `FIELD_PRIORITY` — a wearable or manual value would be fabrication.
- `vo2max` goes in the `PHYSIOLOGY` priority bucket (`manual > whoop > oura > intervals_icu > apple_health`), not `LOAD` — it is a measurement, not a training-model output.
- No fabricated defaults: every new UI element hides itself (renders nothing) when its backing value is `null`, per this project's honesty principle. No `0`, no placeholder dash standing in for missing data.
- Migrations are additive only, generated via `npm run db:generate` (drizzle-kit diffs `src/lib/db/schema.ts` against `drizzle/meta/`, no live DB connection needed) — never hand-write migration SQL.
- `npm test`, `npm run lint`, `npm run typecheck`, `npm run build`, `npm run format:check` must all be green before merge (`docs/RELEASING.md`).

Create `src/components/log/fitness-stats-row.tsx`:

```tsx
export interface FitnessStat {
  label: string;
  value: string;
}

interface Props {
  stats: FitnessStat[];
}

const GRID_COLS: Record<number, string> = {
  1: "grid-cols-1",
  2: "grid-cols-2",
  3: "grid-cols-3",
  4: "grid-cols-4",
};

/**
 * Coarse CTL ramp-rate trend label. A small dead zone around zero absorbs
 * day-to-day noise instead of flapping between Ramping/Tapering.
 */
export function rampTrendLabel(rampRate: number | null): string | null {
  if (rampRate == null) return null;
  if (rampRate > 1) return "Ramping ↑";
  if (rampRate < -1) return "Tapering ↓";
  return "Steady";
}

/**
 * eFTP / max power / W' / ramp-rate trend, next to the PMC chart that
 * computes them. Caller supplies only the stats with real values — there
 * is no zero or placeholder for a field the athlete's data doesn't have.
 */
export function FitnessStatsRow({ stats }: Props) {
  if (stats.length === 0) return null;
  return (
    <div className={`grid gap-2 ${GRID_COLS[stats.length]}`}>
      {stats.map((s) => (
        <div key={s.label} className="flex flex-col">
          <span className="text-[10px] font-bold uppercase tracking-wider text-white/50">
            {s.label}
          </span>
          <span className="mt-1 text-sm font-bold tabular-nums text-white">
            {s.value}
          </span>
        </div>
      ))}
    </div>
  );
}
```
