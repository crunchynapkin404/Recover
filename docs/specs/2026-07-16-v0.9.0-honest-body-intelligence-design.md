# v0.9.0 — Honest Body Intelligence Design

> v0.7 stopped the app from knowing things it doesn't know — in the database.
> The dashboard never got the same treatment. Two of its cards still present
> invented numbers as measurements. This release makes them real, or deletes
> them.

## Problem

v0.7's own spec named the defect and then shipped without fixing it:

> This is the same defect class as the placeholder body-battery curve: invented
> data presented as measured.

That placeholder is still on the dashboard, and it has company.

### The body battery is a drawing

`body-battery.tsx` has a hardcoded SVG path as its default:

```ts
const path =
  curvePath ??
  "M0 40 Q50 30 80 45 L120 120 L160 140 Q200 130 250 110 L300 80 L400 90";
```

No caller ever passes `curvePath`. Every athlete sees the same fictional day —
the same dip at the same hour — regardless of what they did. The percentage
beside it is invented too (`page.tsx:467`):

```ts
current={Math.round(((sleepHours ?? 7) / 9) * 100 * (readiness / 100 || 0.5))}
```

`sleepHours ?? 7` assumes seven hours of sleep for an athlete who logged none.
`readiness / 100 || 0.5` silently becomes 0.5 when readiness is 0 or null —
the `||` swallows a legitimate zero. The formula has no basis: it is two
unrelated numbers multiplied to land in 0–100.

### The sleep card states facts it has never been told

`page.tsx:453-459` passes constants:

```tsx
stages={[
  { label: "Awake", pct: 8,  color: "rgba(239,68,68,0.8)" },
  { label: "Deep",  pct: 20, color: "#6366f1" },
  { label: "Core",  pct: 25, color: "#3b82f6" },
  { label: "REM",   pct: 47, color: "#38bdf8" },
]}
bedtimeAdvice="22:30 – 23:00"
```

Every athlete gets 47% REM every night, forever. The bedtime is a string
literal rendered under the words "Optimal bedtime tonight". Efficiency
(`page.tsx:452`) is `sleepHours / 8` — a duration ratio labelled as
efficiency, which is time asleep over time in bed.

### The sleep score is not the sleep score

The worst of the set, because a real value exists and is ignored.
`page.tsx:450` computes the card's score as:

```ts
score={Math.round((sleepHours / 9) * 100)}
```

That is duration over an arbitrary nine hours. `wellnessDaily.sleepScore` —
the provider's actual 0–100 score, present on **221 of the athlete's 368
days** — is read nowhere on the dashboard. An athlete who slept 9 hours badly
sees "100 Score".

The same invented number appears a second time in the vitals grid
(`page.tsx:409-419`), on a tile labelled "Sleep Score" with the unit "/100",
alongside `Efficiency: ${sleepHours / 8}`. Two surfaces, one fiction, one
unused real column.

### What the data actually supports

Verified against the live database. Figures below are the **real athlete's
368 wellness days**, every one carrying a raw intervals.icu payload. (The
instance also holds a 91-day demo account; summing the two gives 459, which
is not a number about anyone. A spec arguing against overstated data should
not overstate its own.) The wellness row exposes 46 keys. Populated:

| Field           | Days with data | Verdict                       |
| --------------- | -------------- | ----------------------------- |
| `sleepSecs`     | 222 / 368      | usable                        |
| `sleepScore`    | 221 / 368      | usable                        |
| `sleepQuality`  | 221 / 368      | subjective 1–4, not stages    |
| `steps`         | 236 / 368      | daily total, not intraday     |
| `avgSleepingHR` | 33 / 368       | too sparse                    |
| `vo2max`        | 79 / 368       | too sparse (v0.10 bio age)    |
| `systolic`      | 0 / 368        | absent (v0.10 blood pressure) |
| `respiration`   | 0 / 368        | absent                        |

**There is no sleep-stage data. There is no bedtime, wake time, or
time-in-bed — no key in the payload carries them.** intervals.icu gives
duration and a score, nothing more.

So the stage breakdown and the efficiency figure are not "not yet wired up".
They are unbackable: no provider field exists to wire them to. They get
deleted, not fixed.

## The line this release draws

Two things look similar and are not:

- **A measurement claim** — "you got 47% REM", "your efficiency is 88%".
  Requires data. We have none. **Delete.**
- **A labelled model estimate** — "estimated energy, derived from your morning
  readiness and today's training load". Requires real inputs and honest
  labelling. We have both. **Build.**

The body battery survives as the second kind. It is never presented as
measured energy; it is a model, named as one in the UI, with its constants in
one documented place.

## Scope

1. Body battery: a real curve from real inputs, or nothing.
2. Sleep debt: cumulative deficit from real `sleepSecs`.
3. Bedtime target: derived from debt and a wake time **the athlete gives us**.
4. Sleep score: read the real `sleepScore` column, on both surfaces.
5. Delete the stage breakdown and both efficiency figures.

Not in scope: biological age, health records, blood pressure (v0.10 — the data
isn't there, see table above); intraday HR (no provider); adjusting bedtime for
tomorrow's planned session (needs v0.9.1's adaptive plans — see Hand-off).

## Part 1 — Body Battery

### Model

New file `src/lib/body-battery.ts`. Pure functions, no I/O — same contract as
`readiness.ts`.

The battery starts each day at the morning readiness score and declines. It
does not recover during the day, and this is a deliberate modelling choice
rather than an oversight: recovery detection needs intraday HR, which no
connected provider gives us. Recovery happens overnight, and it is expressed
as tomorrow's readiness — which is exactly where the next day's curve starts.
Modelling a daytime rebound would mean inventing one.

```
charge(00:00) = readiness            // from daily_metrics; null → calibrating
charge(t)     = charge(00:00) − awakeDrain(t) − activityDrain(t)   clamped 0..100
```

**Awake drain** — a linear metabolic cost spread evenly from `wakeMinutes` to
`bedMinutes`, zero before waking:

```
AWAKE_DRAIN_TOTAL = 25   // points across a full waking day
```

**Activity drain** — proportional to training load, spread across the
activity's real duration:

```
DRAIN_PER_LOAD = 0.35    // a 100-load session costs 35 points
```

**The waking window when the athlete hasn't set a wake time:**

```
DEFAULT_WAKE_MINUTES = 420   // 07:00
DEFAULT_BED_MINUTES  = 1380  // 23:00
```

This looks like the fabrication this release is deleting, and the distinction
is worth being precise about. These two are **model shape parameters**: they
decide how 25 points of drain are distributed across a day, and the curve is
dominated by activity drain regardless. Nothing is asserted to the athlete
about when they sleep. A _bedtime recommendation_ built on a guessed wake time
would be different in kind — it puts a specific claim about this athlete's life
on screen, under the words "Optimal bedtime tonight". That is why Part 2's
`wakeTime` has no default and this does.

When `bodyPrefs.wakeTime` is set, it replaces `DEFAULT_WAKE_MINUTES`, and
`bedMinutes` becomes the computed bedtime target from Part 2.

These four constants are the entire model. They live at the top of
`body-battery.ts` with a comment explaining they are calibration choices, not
measurements.

**Provenance:** activities are filtered `ne(provider, 'strava')`, consistent
with every other analytics query (Strava's Nov 2024 API agreement). The live DB
has 121 intervals.icu and 59 manual activities with load — enough.

**Honest degradation:** `readiness == null` (calibrating, or no wellness that
day) → the function returns `null` and the card renders "Not enough data yet",
matching the readiness engine's `calibrating` band. It never falls back to a
default charge.

### Interface

```ts
export interface BatteryPoint {
  minutes: number; // minutes past local midnight, 0..1440
  charge: number; // 0..100
}

export interface BodyBatteryInput {
  readiness: number | null;
  wakeMinutes: number; // minutes past midnight
  bedMinutes: number;
  activities: Array<{
    startMinutes: number;
    durationMin: number;
    load: number;
  }>;
  /** Clip the curve here (now, for today). 1440 for a past day. */
  nowMinutes: number;
}

export interface BodyBatteryResult {
  current: number | null; // null = not enough data
  points: BatteryPoint[]; // empty when current is null
}

export function computeBodyBattery(input: BodyBatteryInput): BodyBatteryResult;
```

Sampled every 15 minutes (97 points across a full day, 0..1440 inclusive) —
enough for a smooth path, small enough to pass through a server component
without ceremony. A consequence worth knowing: `current` is the last sampled
point, so it can lag wall-clock by up to 14 minutes. At the modelled drain
rates that is well under half a battery point, so it is accepted rather than
solved.

### UI

`body-battery.tsx` loses its placeholder default. `curvePath` stops being
optional; the component takes `points` and builds the path itself, and takes
`current: number | null`. When null it renders the empty state instead of a
chart. The header label becomes "Estimated Energy" with the subtitle "modelled
from readiness and training load" — the honesty lives in the label, not a
tooltip.

Path construction reuses the existing hand-rolled SVG approach (`lib/charts.ts`
is the precedent — recharts stays unused).

## Part 2 — Sleep Debt & Bedtime

### Preferences

Two values we cannot derive and must ask for. New table — `notification_prefs`
has already drifted into a junk drawer (`autoDescribeStrava` lives there), and
sleep preferences are not notifications:

```ts
export const bodyPrefs = pgTable("body_prefs", {
  id: uuid("id").primaryKey().defaultRandom(),
  userId: text("user_id")
    .notNull()
    .unique()
    .references(() => users.id, { onDelete: "cascade" }),
  // null = not set; no default, because a guessed wake time is a fabrication.
  wakeTime: text("wake_time"), // "HH:MM" local
  sleepNeedSecs: integer("sleep_need_secs").notNull().default(28800), // 8h
});
```

`wakeTime` has **no default**. A defaulted wake time would put an invented
bedtime back on the dashboard — the exact bug this release removes. Null wake
time means no bedtime target is shown; the card prompts the athlete to set one
and links to settings.

`sleepNeedSecs` defaults to 8h legitimately: it is a target the athlete can
change, not a claim about them.

Migration `0011_body_prefs.sql`, generated via `npm run db:generate`.

### Debt

New file `src/lib/sleep-debt.ts`, pure:

```ts
export const DEBT_WINDOW_DAYS = 14;
export const MIN_DEBT_DAYS = 7;
export const MAX_NIGHTLY_PAYBACK_SECS = 3600; // 1h

export interface SleepDebtResult {
  debtSecs: number | null; // null = not enough data
  nightsCounted: number;
  bedtime: string | null; // "HH:MM", null when wakeTime unset
}

export function computeSleepDebt(input: {
  nights: Array<{ sleepSecs: number | null }>; // most recent last, ≤ 14
  sleepNeedSecs: number;
  wakeTime: string | null;
}): SleepDebtResult;
```

Debt is `Σ max(0, need − sleepSecs)` over the window. Nights with
`sleepSecs == null` are **skipped, not counted as zero** — 147 of 459 days have
no sleep record, and treating those as perfect nights would understate debt.
Surplus does not offset: sleeping 10h on Sunday does not repay Wednesday, and
modelling it as though it does is the kind of tidy fiction this release exists
to remove.

Fewer than `MIN_DEBT_DAYS` nights with data in the window → `debtSecs: null`,
card shows "Not enough sleep data yet".

Day flags (v0.7) are deliberately ignored here: an illness day's sleep is real
sleep. Flags exclude days from _baselines_, and debt is not a baseline.

### Bedtime target

```
bedtime = wakeTime − (sleepNeedSecs + min(debtSecs, MAX_NIGHTLY_PAYBACK_SECS))
```

Payback is capped at one hour: six hours of debt cannot be repaid tonight, and
recommending a 01:00 bedtime shift would be advice no one follows. Returns
`null` when `wakeTime` is null.

### UI

`sleep-card.tsx`:

- **Remove** the `stages` prop and the stage bar entirely — no data source.
- **Remove** the `efficiency` prop — no time-in-bed, so no efficiency.
- **Add** `debtSecs: number | null`, rendered as "Sleep debt: 3h 20m over 14
  nights" or the empty state.
- `bedtimeAdvice` becomes computed, and renders the settings prompt when null.
- `score` becomes `number | null` and is fed `latest?.sleepScore` — the real
  column — rendering "—" when the provider gave none. It is **not** derived
  from duration.

`duration` stays: `sleepSecs` is real.

`vitals-grid` on `page.tsx:409-419`: the "Sleep Score" tile takes the same real
`latest?.sleepScore` (`"—"` when null), and its `avg7d` efficiency subtitle is
removed rather than replaced — there is nothing true to put there. The
`sparkPath` over `window7.map((w) => w.sleepSecs)` is real and stays.

## Test Plan

Pure functions get unit tests, mirroring `readiness.test.ts`:

`body-battery.test.ts`

- readiness null → `current: null`, `points: []`
- no activities → monotonic decline of exactly `AWAKE_DRAIN_TOTAL` across the
  waking window
- one 100-load activity → 35-point drop across its real duration
- charge clamps at 0 and never goes negative
- curve is monotonically non-increasing (the no-daytime-recovery invariant)
- `nowMinutes` clips the curve

`sleep-debt.test.ts`

- 6 nights of data → null (below `MIN_DEBT_DAYS`)
- nulls skipped, not zeroed: 7 nights of 8h + 7 nulls → debt 0, `nightsCounted: 7`
- surplus does not offset a deficit
- bedtime null when `wakeTime` null
- payback capped at 1h with 6h of debt (wake 07:00, need 8h → 22:00, not 16:00)
- bedtime wraps past midnight correctly (wake 05:00, need 8h, no debt → 21:00)

Component tests use `renderToString` in the default node env, following
`src/components/coach/artifact-card.test.tsx`. Both cards here are
presentational, so the `react-dom/client` + `act()` setup that
`tests/journal-form.test.tsx` needs for the v0.7 journal form would add
ceremony without adding reach. The sleep card renders no stage bar; the battery
card renders the empty state on null.

DB-gated tests need `set -a; . ./.env; set +a`.

## Definition of Done

- The sleep and energy cards invent no physiological value:

  ```bash
  grep -rn '"REM"\|22:30\|M0 40 Q50\|"Deep"\|"Core"\|sleepHours / 9\|sleepHours ?? 7\|sleepHours / 8' \
    src/ --include=*.ts --include=*.tsx | grep -v '\.test\.'
  ```

  returns nothing. Before this release it returned exactly ten lines —
  `body-battery.tsx:17` (curve), and `page.tsx` 413, 418, 450, 452, 455, 456,
  457, 459, 468 — one for each of the eight sites in the fabrication table.

  Two things this grep is **not**:

  - It is not a whole-dashboard guarantee. It searches for the _known_
    literals, so it cannot see a fabrication nobody has named yet. It says
    nothing about the hero Recovery/Strain rings, which still render `60` and
    `0.0` from `?? 0`-coalesced nulls for a manual-only athlete. That is real,
    pre-existing, out of this release's scope, and recorded in the CHANGELOG
    rather than hidden behind a green grep.
  - It must exclude test files. `body-battery.test.tsx` and
    `sleep-card.test.tsx` legitimately contain these literals inside negative
    assertions that pin them from returning — the test asserting the bug is
    gone must name the bug.

  (Don't loosen `"REM"` to bare `REM` either — it matches
  `INCREMENTAL_OVERLAP_DAYS` in both sync modules and the DoD becomes
  unsatisfiable.)

- `latest?.sleepScore` is read on both the sleep card and the vitals grid. The
  real column stops being ignored.

- A day with training shows a curve that drops when the athlete actually trained.
- An athlete with no wake time set sees a prompt, not a bedtime.
- A calibrating athlete sees "not enough data", not a number.

## Hand-off to v0.9.1

Bedtime does not yet account for tomorrow's session. That needs the adaptive
plan engine to know what tomorrow _is_ — v0.9.1 builds it, and should extend
`computeSleepDebt` with an intensity term at that point.

`DRAIN_PER_LOAD = 0.35` and `AWAKE_DRAIN_TOTAL = 25` are first-pass
calibrations. Once the curve is live against 121 real activities, they are worth
revisiting — the honest version of that is comparing modelled end-of-day charge
against next-morning readiness, which is a v0.9.2 correlation-engine question.
