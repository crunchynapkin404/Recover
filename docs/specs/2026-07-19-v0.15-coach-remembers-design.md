# v0.15 — The Coach Remembers: Design

**Date:** 2026-07-19
**Status:** In review

## Goal

Coach memory holds structured facts; it still can't recall what was actually
said, and every ride ends in silence. This release gives the coach a real
memory of past conversations and journals, and closes the loop after every
ride: the upload is detected automatically, the athlete is asked how it went,
and the coach writes a review that combines the data with the athlete's own
words. Done when: the coach quotes a real past conversation unprompted, a
synced ride produces a debrief prompt within ~15 minutes and a review that
cites the athlete's RPE, and a month-end report shows up without being asked.

## Why this is v0.15

The roadmap had **Cycle-Aware Readiness** here. Nobody on the instance
generates the data (no menstruating athlete, no Oura temperature stream in
use), so phase-aware baselines would be tuned against synthetic cycles —
fabrication with extra steps. It moves to the deferred section ("if demand
shows up", next to Fitbit), and the versions behind it shift down one:
The Coach Remembers → **v0.15**, Stronger Together → v0.16, Good Self-Hosted
Citizen → v0.17, 1.0 Hardening → v0.18. The roadmap edit ships with this
release.

## Principle

Recall must be literal. The coach cites what was actually written — dated,
quoted, attributed to its source — and says "I found nothing about that"
when search comes up empty. No paraphrase presented as memory, no retrieval
injected behind the athlete's back, and nothing recalled from ghost threads
(they were promised to vanish). The ride review holds the same line: the
athlete's subjective input is quoted, never invented, and a skipped debrief
produces a review that says the athlete gave no feedback — not "felt fine."

## Scope

Five features, one release:

1. **Recall over history** — `recall_history` coach tool over Postgres FTS
2. **Post-ride loop** — auto-detect upload → RPE prompt → coach ride review
3. **Monthly report** — the weekly review's big sibling
4. **Voice input** — mic in the chat composer via the Web Speech API
5. **Token transparency** — per-user LLM usage in settings

Approach decision (made with the user): recall is a **tool the coach calls**
(Approach A), not automatic per-message retrieval injection — zero token
cost on turns that don't need it, no second store to drift, and the
dishonesty failure mode of irrelevant snippets inviting confabulated
connections is structurally avoided. Embeddings stay out per the roadmap
("FTS first; embeddings only if FTS demonstrably isn't enough").

## Schema (migration 0018, additive)

- **`chat_messages.search`**: generated `tsvector` column,
  `to_tsvector('simple', content)`, GIN index. The `simple` config because
  the athlete's conversations are mixed Dutch/English — language-specific
  stemming would quietly mangle one of them; exact-token matching is weaker
  but never wrong.
- **`wellness_daily.search`**: same treatment over `notes`.
- **`chat_threads.kind`** enum grows `"debrief"` and `"monthly"` (existing:
  `chat` | `morning` | `weekly`).
- **`activities`**: `perceivedExertion` (real, 1–10, nullable), `feel`
  (`strong` | `normal` | `weak`, nullable), `debriefState`
  (`pending` | `answered` | `skipped` | `expired`, nullable — null means
  never eligible), `debriefThreadId` (nullable FK → chat_threads, on delete
  set null), `reviewedAt` (nullable timestamp — once-only guard for the
  review, same pattern as the v0.14 race debrief).
- **`llm_usage`**: `id`, `userId` (cascade), `createdAt`, `model` (text),
  `slot` (`quick` | `deep`), `purpose` (`chat` | `morning` | `weekly` |
  `monthly` | `ride_review` | `race_debrief` | `health_extract`),
  `inputTokens` (int), `outputTokens` (int). Index on `(userId, createdAt)`.
  (The Strava describer is deterministic formatting, not an LLM call — it
  doesn't appear here.)
- **`notification_prefs`**: `debriefPushEnabled` (boolean, default false —
  the user chose opt-in push for debriefs) and `rideDebriefsEnabled`
  (boolean, default true — the whole loop's kill switch).

All additive; no backfill. Existing rows get `debriefState = null` (never
eligible — no retroactive prompts for last month's rides).

## 1. Recall over history

**Indexing.** In-place generated tsvector columns (above) — no separate
search table to drift out of sync. Surfaces covered: all chat messages
(which includes weekly reviews, morning insights, debriefs, and monthly
reports — they are chat messages in kind-tagged threads) and journal notes.
Coach memories are already injected into the system prompt wholesale and
don't need search.

**The tool.** `recall_history(query, limit?)` (registry 53 → 54, read
scope):

- Query goes through `websearch_to_tsquery('simple', …)` — supports quoted
  phrases and `-exclusion` without ever throwing on user input.
- Searches the user's own rows only (join through threads for userId;
  journal notes filtered by userId), **excluding ephemeral threads** and
  excluding the current thread (the model already has it in context).
- Returns up to `limit` (default 5, max 10) hits ranked by `ts_rank`, each
  with: ISO date, source (`chat` | `journal` | `weekly` | `morning` |
  `debrief` | `monthly`), the thread title where applicable, and a
  `ts_headline` snippet (~40 words) of the matched text.
- Empty result returns an explicit "no matches" — the system prompt
  instructs the coach to say so rather than improvise.

**Prompting.** The coach system prompt gains a short instruction: when the
athlete refers to something from the past ("like I said", "my knee again",
"that plan we discussed"), search before answering, and cite recalled
material with its date ("on June 3rd you wrote…"). This is the "unprompted"
behavior from the roadmap's done-when — models call cheap read tools
eagerly, and if practice shows otherwise, automatic injection can be layered
on later without schema changes.

Journal note search also honors nothing extra: notes are already
provider-free athlete text. Strava activity _data_ never enters any of this
— recall returns conversation and journal text, not activity rows.

## 2. The post-ride loop

The centerpiece. Flow: ride done → auto sync → "how was the ride?" → RPE +
feel + notes → review combining data and athlete input.

**Detection — activity poll.** intervals.icu has no webhooks (sync is
pull-only), so a new lightweight job kind `activity_poll` runs on the
existing scheduler: every **15 minutes**, for users with an active
intervals.icu connection and `rideDebriefs` on, fetch the activity list
since the last poll cursor (one cheap API call; no wellness, no streams).
Poll sleeps between 23:00 and 06:00 server-local — the 05:00 full sync
covers the night. New activities found are ingested through the existing
activity upsert path (streams stay lazy, as on the activity detail page).
The poll also maps `icu_rpe` and `feel` from the API when the athlete
already set them in intervals.icu — they prefill the debrief form.

Strava: its webhook could push instantly, but Strava's API agreement bars
Strava data from AI models, so a Strava-only activity can never get an AI
review. Strava-sourced activities keep their existing daily sync and are
excluded from this loop entirely (`ne(provider, 'strava')`, as everywhere).

**Eligibility.** A newly ingested non-Strava activity with moving time
≥ 15 min becomes `debriefState = 'pending'` — but only one pending debrief
per user at a time. A second ride the same day stays unmarked and is
promoted to `pending` by a later scheduler tick once the first resolves
(the tick already handles expiry; promotion rides the same pass).
Activities ingested by the
05:00 sync (overnight uploads, backfills) are eligible only if they
started within the last 24 h — no debrief prompts for historical imports.

**The prompt.** On `pending`: an opt-in web push ("Ride synced — how did it
go?") via the existing push infra, and a **debrief card** on the dashboard
and the activity detail page: RPE slider 1–10, feel picker
(strong / normal / weak), free-text "anything to note — pain, mood,
conditions?". Honest-input rules apply: untouched controls write nothing
(the v0.7 slider pattern). Prefilled RPE/feel from intervals.icu is shown
as prefill, editable before submit.

**The review.** On submit (`answered`): RPE/feel/notes are stored on the
activity, then the coach generates the ride review with the activity's full
data (metrics, laps, intervals; streams summarized the way the activity
tools already do) plus the athlete's input. The review is instructed to
reconcile the two — RPE 9 on a zone-2 ride is worth flagging, RPE 4 on an
interval day is a good sign — and to quote the athlete's words, not
paraphrase them into findings. Pain or injury mentions get filed to coach
memory through the existing `remember_fact` machinery (the review
generation runs with the memory tools available). The review posts as an
assistant message in a `debrief`-kind thread for that activity
(`debriefThreadId`), push-notified if opted in, and the activity detail
page renders it. Post-and-mark is atomic (`reviewedAt`), the lesson of the
v0.14 race-debrief fix — no duplicate reviews on partial failure.

**Skip and expiry.** "Skip" (`skipped`) generates the review immediately
from data alone, opening with the fact that the athlete gave no feedback.
A card untouched at end of day expires (`expired`) on the next scheduler
tick and gets the same data-only review. Never a fabricated "felt fine."

**Relationship to the race debrief.** v0.14's post-race debrief stays as
is — it's about the race entity. A race-result activity that would get both
gets only the race debrief (`reviewedAt` set when the race debrief claims
the activity).

**Failure handling.** Poll errors log and retry next slot (no user-visible
error for a transient API blip; the sync-jobs status row records it). A
failed review generation leaves `reviewedAt` null and retries on the next
tick, capped at 3 attempts, then the thread gets a plain "review couldn't
be generated" note instead of silence.

## 3. Monthly report

`generateMonthlyReport(userId)` clones the weekly review's shape: due-slot
logic (`mostRecentMonthlySlot` — 1st of each month at the review hour,
computed due-since so a sleeping server still delivers late rather than
never — the v0.5 lesson), posts as an assistant message in a
`monthly`-kind thread. No push — a monthly report can wait for the next
visit.

Content, all from existing engines (no new math): monthly load vs the plan
(training-load + week-plan adherence), recovery trend (readiness bands
distribution, HRV/RHR direction), milestones hit (v0.9.4 machinery),
biomarker deltas when a new blood panel landed that month (v0.13), races
completed or upcoming (v0.14), and next month's outlook from the plan
skeleton. Aggregates are Strava-firewalled at the query level like the
weekly review's. Sections with no data are omitted, not padded — a manual
athlete with no biomarkers gets a shorter report, not an interpolated one.

## 4. Voice input

A mic button in the chat composer (`chat-interface.tsx`) using the Web
Speech API (`SpeechRecognition`/`webkitSpeechRecognition`): tap to dictate,
interim results stream into the input field, tap again to stop, the athlete
edits before sending — dictation fills the box, it never auto-sends.
Feature-detected; the button simply doesn't render where the API is absent.
Language follows the browser/OS locale (covers the Dutch/English mix).

Honesty note, surfaced as a one-line hint next to the control the first
time: depending on the browser, recognition may run on the vendor's servers
(Chrome sends audio to Google). Recover itself never sees or stores audio.
The roadmap's "on-device" phrasing was wrong and dies here; a self-hosted
Whisper path is possible later if that trade-off ever matters.

## 5. Token transparency

Every LLM call site records a row in `llm_usage` — the AI SDK exposes usage
on `generateText` results and in `streamText`'s `onFinish`. Call sites:
chat route, morning insight, weekly review, monthly report, ride review,
race debrief, health-record extraction. Recording failures are
swallowed (usage accounting must never break a chat reply); providers that
omit usage (some OpenAI-compatible endpoints) record nothing rather than
estimates.

Settings gets a **usage card**: current and previous month, token totals
split by model and purpose. Tokens, not euros — BYO endpoints make pricing
unknowable, and a made-up cost number is exactly the kind of decoration
this app deletes. No quotas, no alerts (single-digit-user instances; YAGNI).

## UI summary

- **Dashboard**: debrief card when one is pending.
- **Activity detail**: debrief card / stored RPE + feel / the ride review.
- **Coach**: mic button; recall citations appear naturally in replies;
  debrief and monthly threads listed with their kinds like weekly reviews.
- **Settings**: ride-debrief toggle, debrief push toggle, LLM usage card.

## Testing

House rules: pure engines get unit tests, DB paths get env-gated tests.

- **Pure**: `mostRecentMonthlySlot` (boundaries: Jan 1, month lengths, due
  vs not-due), debrief eligibility (duration floor, 24 h window, one
  pending per user, Strava exclusion), FTS query building (websearch
  syntax, limit clamp), report section assembly with missing data.
- **DB-gated**: tsvector round-trip (insert Dutch and English text, search
  both), recall isolation (user A never sees user B; ephemeral excluded),
  debrief state machine transitions (pending → answered/skipped/expired,
  once-only `reviewedAt`), llm_usage inserts, migration 0018.
- **Review honesty**: prompt-level tests asserting the data-only review
  states the absence of athlete feedback, quoted-not-paraphrased subjective
  input reaches the prompt.

## Out of scope

- Embeddings / pgvector (FTS must fail visibly first)
- Automatic retrieval injection (revisit only if the coach demonstrably
  under-uses the tool)
- AI reviews for Strava-sourced activities (API agreement)
- Cycle-Aware Readiness (deferred — no real athlete data on this instance)
- Cost-in-currency estimates, usage quotas
- Self-hosted speech-to-text
