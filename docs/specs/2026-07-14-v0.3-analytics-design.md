# v0.3 — Analytics depth (activity detail, PMC ranges, wellness trends)

Status: approved 2026-07-14 · Roadmap phase: v0.3 (docs/ROADMAP.md)

## Goal

Give a month of training history real depth: open any activity and see what
happened inside it; watch fitness build over selectable time ranges; see
HRV/RHR/sleep trending against personal baselines.

**Definition of done:** a month of history is explorable end-to-end — from
the dashboard or history list into any activity's streams and laps, and
through Training/Wellness trend charts at multiple ranges.

## Decisions (with reasons)

- **Analytics live on the existing Performance (log) page** as two sub-tabs
  (Training | Wellness): the bottom nav's five slots stay untouched, and
  there is one obvious place to look. No new top-level routes except
  `/activity/[id]`.
- **Activity detail = charts + laps, no GPS map** (user choice): no map
  library, no tiles; intervals.icu already renders maps well.
- **Streams fetch lazily on first view**, cached in the existing
  `activity_streams` table. No schema changes; no bulk backfill hammering
  the intervals.icu API.
- **Charts stay hand-rolled SVG** like the rest of the app (recharts is
  installed but unused — consistency and weight both favor the existing
  pattern). Stream data is downsampled to ≤ ~300 points before rendering.
- **Strava/manual activities show summary only** ("no detailed data") —
  Strava stream ingestion stays deferred (provenance rules; revisit v0.6).

## Components

### Connector additions (`src/lib/connectors/intervals.ts`)

- `fetchActivityStreams(apiKey, externalId, types)` → GET
  `/activity/{id}/streams?types=time,heartrate,watts,velocity_smooth,altitude`;
  normalize to `{ type, data: number[] }[]`. Missing streams are simply
  absent, not errors.
- `fetchActivityIntervals(apiKey, externalId)` → GET
  `/activity/{id}/intervals`; normalize to laps
  `{ index, label, durationS, distanceM, avgHr, avgPower }[]`.
- Same Basic-auth + ConnectorError handling as existing calls; fixture-based
  unit tests like the current connector tests.

### Streams service (`src/lib/activity-streams.ts`)

- `getOrFetchActivityDetail(userId, activityId)`:
  1. Load activity, verify `activity.userId === userId` (cross-user 404).
  2. Read `activity_streams` rows; if present → return cached.
  3. If `provider === "intervals_icu"` and a connection exists: decrypt key,
     fetch streams + intervals, upsert one row per stream type plus a
     `type: "intervals"` row (unique on activity+type), return.
  4. Manual/Strava or fetch failure → return summary-only marker
     `{ streams: null, reason: "unavailable" | "fetch_failed" }` — the page
     renders the note instead of charts. Failures are logged, never thrown
     to the page.

### Pure utilities (`src/lib/charts.ts`)

- `downsample(values: (number|null)[], target = 300)` — bucket-mean
  downsampling, null-safe.
- `baselineBand(lnMean, lnSd)` → `{ low: exp(m−s), high: exp(m+s) }` for
  HRV (ln-space storage); RHR band is linear (`mean ± sd`).
- `weeklyLoads(activities, weeks = 12)` — Monday-based weekly load sums.
- All unit-tested with hand-computed fixtures.

### Activity detail page (`src/app/activity/[id]/page.tsx`)

- Server component; `requireUser()`; calls `getOrFetchActivityDetail`.
- Summary header (existing glass style) + stream charts (HR red, power
  purple, pace cyan — pace derived from `velocity_smooth`) + laps table.
- Charts are server-rendered SVG components (`src/components/activity/`)
  — no client JS needed beyond the shell.
- Unknown/foreign id → `notFound()`.

### Performance page split (`src/app/log/page.tsx` + `src/components/log/`)

- Client tab switch (Training | Wellness), URL param `?tab=` so it's
  linkable; Training is default.
- **Training tab:** existing content, plus range selector (30/90/180/365 d)
  driving the PMC chart; TSB drawn on the chart (filled area around zero);
  weekly load bars (12 w); **history list**: activities grouped by day
  (last 31 days, "load more" via `?days=` param), rows link to
  `/activity/[id]`.
- **Wellness tab:** HRV chart (daily line, 7-day rolling average, baseline
  band), RHR chart (same, linear band), sleep chart (duration bars, score
  line, 8 h guide) — all from `wellness_daily` + `daily_metrics`, honoring
  the range selector.
- Dashboard recent-activities rows also become links to `/activity/[id]`.

## Error handling

- Streams fetch: connector errors → summary-only render + logged warning;
  the page never 500s because intervals.icu is down.
- Ranges with no data render the existing empty-state pattern, not broken
  axes.
- Cross-user activity access → 404 (tested).

## Testing

- Connector: fixtures for streams/intervals parsing incl. missing types and
  malformed rows.
- Utilities: downsample (short input, nulls, exact bucketing), baselineBand
  (ln round-trip), weeklyLoads (week boundaries).
- Service (pg integration, mocked connector): cache-miss fetches and
  stores; cache-hit skips connector; cross-user access rejected;
  manual-provider returns `unavailable`.
- Smoke: demo-seeded stack — history list renders, activity detail shows
  summary + "no detailed data" (manual provider), tabs/ranges switch.
  (Demo data is manual-provider, so chart rendering with real streams is
  verified with a connector-fixture-backed test, and live on the owner's
  instance after upgrade.)
