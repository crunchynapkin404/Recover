# v0.6 — Strava AI Descriptions Design

> Auto-generate data-dense activity descriptions from intervals.icu metrics and
> push them to Strava. The AI never reads Strava data — only writes back.

## Problem

After a ride or run, athletes want their Strava activity to show useful metrics
at a glance — load, efficiency, form, PRs. Currently they type these manually
or leave the description blank. Recover already has all the data from
intervals.icu; it just needs to format and push it.

## Architecture

```
Scheduler sync completes → new activity detected
  → gather intervals.icu metrics for that activity
  → format into description template
  → check: activity already described? (skip marker)
  → check: existing description? (append below separator)
  → PUT /api/v3/activities/{id} with new description
  → mark as described in DB
```

## Data Source: intervals.icu Only

Strava's API terms bar feeding Strava data to AI models. We comply strictly:

- **Input to description generator:** intervals.icu activity data only
  (`activities.raw` JSONB — which came from intervals.icu sync, not Strava)
- **Output target:** Strava activity description (write-only)
- **Never used as input:** Strava activity name, description, segments, kudos

Fields available from intervals.icu `activities.raw`:

- `icu_training_load` (TSS/load)
- `icu_intensity` (IF)
- `trimp`
- `power_hr_ratio` (Pw:Hr efficiency)
- `hr_decoupling` (aerobic decoupling %)
- `carbs_per_hour` (nutrition estimate)
- `icu_ftp` / `eFTP` at time of activity
- `icu_vo2max_estimate`
- Current CTL/ATL/TSB from `wellness_daily`
- All-time PR detection from `athlete_curves` cache

## Description Format

Template-based (deterministic, no LLM needed for the default format):

```
🏔️ {activity_title from intervals.icu}
🔋 Load: TL {load} | IF {intensity}% | TRIMP {trimp}
⚡ Efficiency: Pw:Hr {power_hr_ratio} | decoupling {hr_decoupling}%
🍔 Carbs: ~{carbs_per_hour} g/u
📈 Form: CTL {ctl} | TSB {tsb} | eFTP {ftp} W | VO2 {vo2max}
🚀 {pr_line — e.g. "594W/1min — all-time PR"}
```

Rules:

- Lines with null/unavailable data are omitted (no "N/A" placeholders)
- PR line only shown when a new PR was set in this activity
- Running activities show pace metrics instead of power
- Sport-specific emoji: 🚴 cycling, 🏃 running, 🏊 swimming, 🏔️ mixed/other

## Strava OAuth Upgrade

### Current scope: `activity:read_all`

### New scope: `activity:read_all,activity:write`

**Reconnect flow:**

- Settings page shows banner: "Upgrade Strava connection for AI descriptions"
- Click → re-initiates OAuth with the new scope
- Old token is replaced on callback
- If user doesn't reconnect, auto-describe is silently disabled (graceful)

### Schema change

Add to `connections` table or a new column:

```sql
ALTER TABLE connections ADD COLUMN strava_write_enabled boolean DEFAULT false;
```

Set to `true` only when the connection has `activity:write` scope (detected
during OAuth callback by checking granted scopes).

## Skip / Append Logic

### Skip marker

After writing a description, store in the activity's metadata:

- Option A: Add `\n\n---\n📊 Recover` as a footer marker
- Option B: Track described activity IDs in a `strava_descriptions` table

Use **Option A** (marker in the text) — simpler, no extra table, and the
marker is visually clean on Strava.

### Append mode

```ts
function buildDescription(existing: string | null, generated: string): string {
  if (!existing) return generated + MARKER;
  if (existing.includes(MARKER)) return existing; // already described
  return existing + "\n\n---\n" + generated + MARKER;
}

const MARKER = "\n📊 Recover";
```

## Implementation: `updateStravaActivity`

```ts
// src/lib/connectors/strava.ts — new export

export async function updateStravaActivity(params: {
  accessToken: string;
  activityId: string;
  description: string;
}): Promise<void> {
  const res = await fetch(
    `https://www.strava.com/api/v3/activities/${params.activityId}`,
    {
      method: "PUT",
      headers: {
        Authorization: `Bearer ${params.accessToken}`,
        "Content-Type": "application/json",
      },
      body: JSON.stringify({ description: params.description }),
    }
  );
  if (!res.ok) {
    throw new Error(`Strava PUT failed: ${res.status}`);
  }
}
```

## Auto-Describe in Sync Pipeline

After activity sync completes for a Strava-connected user with
`strava_write_enabled` + `autoDescribeStrava`:

```ts
// In the sync pipeline, after intervals.icu activities are stored:
for (const activity of newActivities) {
  if (!user.stravaWriteEnabled || !user.autoDescribeStrava) continue;
  if (!activity.raw) continue; // no intervals.icu data

  const description = formatActivityDescription(activity);
  const stravaId = activity.stravaExternalId; // cross-reference
  if (!stravaId) continue;

  const existing = await getStravaDescription(accessToken, stravaId);
  const final = buildDescription(existing, description);
  if (final === existing) continue; // already described

  await updateStravaActivity({
    accessToken,
    activityId: stravaId,
    description: final,
  });
}
```

**Cross-reference:** Activities synced from intervals.icu may have a
`strava_activity_id` field in their raw data (intervals.icu links them).
Use this to find the Strava activity ID to update.

## Coach Tool: `describe_strava_activity`

```ts
name: "describe_strava_activity";
parameters: z.object({
  activityId: z
    .string()
    .optional()
    .describe("Strava activity ID. Omit to describe the most recent activity."),
  style: z
    .string()
    .optional()
    .describe("Custom instructions for the description format."),
});
scope: "write:strava";
```

When called manually, the coach can customize the description or describe
older activities that missed auto-describe.

## Non-Goals (v0.6)

- LLM-generated custom prose descriptions (future — current format is deterministic template)
- Describing activities from other providers (only Strava write-back)
- Segment analysis or KOM mentions (no Strava data as input)
- Editing/deleting descriptions already pushed
