# v0.9.4 — Deeper Insights

**Date:** 2026-07-17
**Status:** approved design, pre-implementation

## Why

The journal's correlation card (v0.7) was the first honest
behavior→readiness insight: mean next-day readiness after each tagged day
against the athlete's overall mean, minimum five events, everything else
"not enough data yet". Three limits have surfaced since:

1. **It only sees what the athlete types.** Training behaviors that
   plausibly move readiness — a hard session, a double day, training late —
   are already in the `activities` table, but produce no insight unless the
   athlete manually tags them. The roadmap calls this out: derive tags from
   activity data instead of asking.
2. **Impact numbers carry false precision.** "🍷 Alcohol −11%" from six
   events reads as authoritative as one from sixty. There is no confidence
   interval, and the vs-overall-mean baseline biases every impact toward
   zero (tagged days are inside the baseline).
3. **The streaks are counts.** The dashboard renders
   `Math.min(window30.length, 30)` — 22 scattered logging days show as a
   "22-day streak". The journal's `streakDays` is likewise a count of
   journaled days in the last 7. Both are tracked as honesty debt in the
   roadmap.

v0.9.4 fixes all three with pure, computed-on-read engines. No new tables,
no migrations, no backfill jobs: everything derives from durable rows at
query time (~400 days of data per athlete — cheap), so nothing can drift
stale and every number is reproducible from the database alone. "A streak
survives a restart" comes for free.

## Decisions taken (with the athlete)

1. **Sober milestones, not gamification.** A quiet Milestones card on the
   dashboard and journal: real streaks, plan weeks completed, plans
   finished. No badges, XP, unlock animations, or notifications.
2. **Athlete-relative intensity thresholds.** "Hard session" means the
   athlete's own top-quartile load over the trailing window, not a fixed
   number. Below 20 training days of history the intensity tags stay
   silent — the same calibrating ethos as the readiness engine.
3. **Insights stay on `/journal`.** The existing correlation card grows
   richer rows (impact ± CI, weekday/weekend detail); no new page or tab.
4. **Pure functions, computed on read.** Auto-tags are never written to
   `wellness_daily.tags`; milestones are never persisted. The alternative
   (materialized `auto_tags` column + achievements table) was rejected:
   stored tags go stale when the athlete's load quartile moves, and a
   stored unlock the data no longer supports is a fabrication.

## Module layout

New `src/lib/insights/` module, following the `week-plan/` pattern of pure
engines plus one thin DB wrapper per consumer:

| file              | role                                               |
| ----------------- | -------------------------------------------------- |
| `auto-tags.ts`    | pure: activities in → per-day auto-tags out        |
| `stats.ts`        | pure: mean, Welch SE, t-critical 95% CI helpers    |
| `correlations.ts` | v2 engine: pure core + the one query wrapper       |
| `milestones.ts`   | pure: streak/milestone computation + query wrapper |

`src/lib/correlations.ts` (v1) is deleted; its single consumer
(`src/app/journal/page.tsx`) repoints to the new module.

## Auto-tags

`deriveAutoTags(days: ActivityDay[]): Map<date, AutoTag[]>` — pure. Input
is per-day aggregates of **non-Strava** activities (the analytics
exclusion `ne(provider,'strava')` applies at the query layer, same as
everywhere else) over the correlation window.

| tag                 | rule (server-local time, as the week engine)           |
| ------------------- | ------------------------------------------------------ |
| 🔥 Hard session     | day's total load ≥ P75 of training-day loads in window |
| 2️⃣ Double day       | ≥2 activities of ≥20 min each                          |
| 😴 Rest day         | no activities that day                                 |
| 🌅 Morning training | first activity starts before 12:00                     |
| 🌙 Late training    | last activity starts at or after 19:00                 |

- **Calibrating silence:** 🔥 requires ≥20 training days (load > 0) inside
  the window; below that the tag is never emitted. The other four are
  unambiguous and always active.
- P75 is computed once per analysis over the window's training-day loads
  (linear interpolation percentile) — deterministic and explainable.
- Morning/late training as tags means "time-of-day patterns" fall out of
  the ordinary correlation machinery ("🌙 Late training → −8% ± 5")
  instead of needing a separate analysis path.
- Auto-tags carry `auto: true` through to the UI; manual tags win on
  collision (dedupe by tag string after union).

## Correlations v2

`correlateTags(input): TagInsight[]` — pure core over
`{ tagsByDate (manual ∪ auto), readinessByDate }`, 90-day window kept from
v1.

- **Two-sample comparison** replaces vs-overall-mean: for each tag, mean
  next-day readiness of days carrying that tag vs days **without that
  tag** (per-tag baseline — with 😴 Rest day in play, most days carry
  some tag, so "untagged overall" would be nearly empty). Impact % =
  `(meanTagged − meanUntagged) / meanUntagged × 100`.
- **95% CI** on the difference: Welch standard error
  `√(s₁²/n₁ + s₂²/n₂)`, multiplied by the t-critical value for the
  Welch–Satterthwaite degrees of freedom (small-n honest; ≈1.96 for large
  n). Rendered as `−11% ± 6` (half-width in percentage points).
- **Row states:** a tag needs ≥5 events to appear at all (unchanged).
  A row whose CI crosses zero renders greyed as
  **"inconclusive · n events"** — visible, never overstated, never hidden.
  Rows are sorted: conclusive by |impact| desc, then inconclusive by
  events desc.
- **Weekday/weekend split:** each row expands to show the same two-sample
  computation restricted to weekday tag-events and weekend tag-events
  (baseline restricted the same way). A side renders only with ≥5 events,
  otherwise "not enough data". No other splits ship — time-of-day is
  covered by the 🌅/🌙 tags.
- Readiness-null days (calibrating) stay excluded, as in v1.

```ts
interface TagInsight {
  emoji: string;
  behavior: string;
  auto: boolean;
  impactPct: number; // signed
  ciHalfWidthPct: number; // 95% CI half-width, percentage points
  conclusive: boolean; // CI does not cross zero
  events: number;
  splits: {
    weekday: SplitInsight | null; // null = fewer than 5 events
    weekend: SplitInsight | null;
  };
}

interface SplitInsight {
  impactPct: number;
  ciHalfWidthPct: number;
  conclusive: boolean;
  events: number;
}
```

## Milestones & streaks

`computeMilestones(input): Milestones` — pure over
`{ journaledDates, planWeeks }`.

- **Logged day** = a `wellness_daily` row with any user-entered signal:
  `energy1_10`, `soreness1_10`, `stress1_10`, `mood`, `notes` non-null, or
  `tags` non-empty. Auto-synced rows (HRV/sleep only) do not count — a
  streak the athlete didn't build isn't theirs.
- **Logging streak:** current run = consecutive days ending today or
  yesterday (today not yet logged doesn't break yesterday's run); best
  run = longest ever. Computed over the full history (~400 rows, one
  indexed query).
- **Plan weeks completed:** count of `training_blocks` rows with
  `adherencePct ≥ 70` (written by the weekly rollover since v0.9.2).
- **Plans completed:** `training_plans` rows with status `completed`.

Surfaces:

- **`MilestonesCard`** (new component, dashboard + journal): current
  logging streak with best-ever, plan weeks completed, plans completed.
  Quiet glass-card styling like TodayCard; rows with zero value render a
  muted em-dash, not a celebration prompt.
- **Dashboard hero streak** (`src/app/page.tsx:596`) switches from
  `Math.min(window30.length, 30)` to the real current logging streak.
- **Journal `streakDays`** switches from the 7-day count to the real
  current run (same source, one computation).

## UI changes

- `CorrelationInsights` grows: `impact ± ci` with event count on each row,
  greyed inconclusive state, an `auto` chip on derived tags, and an
  expand/collapse weekday–weekend detail per row (server-rendered
  `<details>`; no client JS needed).
- `MilestonesCard` as above.
- No new routes, nav entries, or settings.

## Constraints

- `provider = 'strava'` activities are excluded from auto-tag derivation
  (Strava API AI/analytics clause — same rule as every analytics query).
- No LLM involvement anywhere in this release; every number is
  deterministic and reproducible.
- Coach/MCP tool exposure of insights is **out of scope** (candidate for a
  later release; the pure engines make it a thin wrapper when wanted).
- GDPR data export, default journal entries, and the other polish-backlog
  items stay out of scope.

## Testing

Pure engines get fixture-based unit tests, no DB:

- **auto-tags:** P75 boundary (exactly at threshold), <20 training days
  silence, double-day minimum duration, midnight/noon/19:00 boundaries,
  Strava rows never reach the engine (enforced in wrapper test instead).
- **stats:** CI against hand-computed known values (small n uses
  t-critical, not 1.96); degenerate cases (n=1, zero variance).
- **correlations v2:** conclusive vs inconclusive classification, sort
  order, split gating at exactly 5 events, manual∪auto dedupe.
- **milestones:** runs across gaps, today-unlogged/yesterday-logged edge,
  best-run tracking, synced-only rows not counting.

DB-gated integration tests (existing `describe.skipIf(!hasDb)` pattern):
one per query wrapper, seeding activities/wellness/plan rows and asserting
end-to-end output, including the Strava exclusion.

UI: server-HTML smoke via the existing pattern (mind the `<!-- -->`
splicing).

## Done when

- Auto-tags appear in the journal insights without any user input (roadmap
  criterion), each marked as derived.
- Every correlation row carries a 95% confidence interval; rows whose CI
  crosses zero say "inconclusive" instead of asserting an impact.
- The dashboard and journal streaks are real consecutive runs, derived
  from the database — restart-proof by construction (roadmap criterion).
- v1 correlations code is gone; no orphaned imports.
