# v0.11 — Wearable Connectors: Design

**Date:** 2026-07-18
**Status:** Approved

## Goal

intervals.icu stops being the only automatic pipe. Whoop and Oura bring the
data v0.9.0 had to delete cards for — staged sleep, bed/wake times, HRV and
resting HR measured on the wrist — with full per-user isolation and visible
provenance on every field. Done when: HRV and staged sleep flow in nightly
from a real Whoop or Oura account, two providers reporting the same morning
resolve by an explicit per-field priority instead of last-writer-wins, and
a fresh invite lands in a guided first run.

## Decisions

| Decision          | Choice                                                                                                                    |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------- |
| Conflict policy   | Per-FIELD source priority, recorded per field (`field_sources` jsonb)                                                     |
| Priority order    | `manual` > dedicated wearable (`whoop`/`oura`/`withings`) > `intervals_icu` > `apple_health`                              |
| Whoop auth        | OAuth2, instance-level app creds via env (`WHOOP_CLIENT_ID/SECRET`)                                                       |
| Oura auth         | Personal Access Token pasted in Settings (see deviation below)                                                            |
| Wellness writes   | ALL providers go through one `applyWellnessPatch` merge — no raw upserts                                                  |
| New wellness data | Sleep stages (deep/rem/light/awake), bed start/end, temperature deviation, respiratory rate, systolic/diastolic, body fat |
| Sync              | Same `sync_jobs` pipeline: nightly incremental, overlap-window healing                                                    |
| Schema            | Migration 0014, purely additive (enum values + nullable columns)                                                          |

### Deviation: Oura ships token-first, not OAuth-first

The roadmap wrote "Oura OAuth". Oura's v2 API officially supports Personal
Access Tokens, which a self-hoster creates in one minute on cloud.ouraring
.com — no developer-app registration, no redirect URL, no client secret.
That is pillar-3 boring and exactly the intervals.icu key flow users
already know. Whoop has no PAT equivalent, so Whoop carries the OAuth
framework; Oura OAuth can reuse it later if multi-user demand shows up
(one `authKind` switch, not a redesign).

## Architecture

### Conflict policy — `src/lib/wellness-merge.ts`

The v0.10-and-earlier intervals sync upserted whole wellness rows,
overwriting every field (including with `null`). With two providers
reporting the same morning that is data loss. All wellness writes now go
through one merge:

- `WellnessPatch`: the nullable-field subset a provider can report
  (hrvMs, restingHr, sleepSecs, sleepScore, sleep stages, bedStart/bedEnd,
  tempDeviationC, respiratoryRate, weightKg, bodyFatPct, systolic,
  diastolic, ctl, atl, eftp).
- `FIELD_PRIORITY`: explicit source order per field. Physiology
  (`hrvMs`, `restingHr`, sleep, temperature, respiration):
  `manual > whoop > oura > intervals_icu > apple_health`. Body
  composition + blood pressure: `manual > withings > oura > whoop >
intervals_icu > apple_health`. Training-load fields (`ctl`, `atl`,
  `eftp`): `intervals_icu` only — wearables never touch them.
- `mergeWellnessPatch(current, currentSources, patch, source)` (pure): a
  patch field wins iff it is non-null AND (the field is empty, or the
  incoming source's priority ≥ the recorded field source's). Writing
  records the source in `field_sources`. Null patch fields never erase
  data. Same-source rewrites always win (overlap-window healing).
- `applyWellnessPatch(userId, date, patch, source)` (db): upserts the row
  through the pure merge; callers batch `computeDailyMetrics` themselves.
- Legacy rows (`field_sources` null) treat every populated field as owned
  by the row's `source` column.

`wellness_daily.source` stays as the row-level "who created this day
first" and gains the new enum values; per-field truth lives in
`field_sources`.

### Connector framework

The framework is the set of conventions the three existing connectors
already share, now made explicit and reused:

- One module per provider in `src/lib/connectors/` exporting typed
  fetchers + a `mapXDay(s)` pure mapping to `WellnessPatch` per date.
- Tokens AES-encrypted on the `connections` row (enum gains `whoop`,
  `oura`, `withings`); OAuth providers use `encryptedRefreshToken` +
  `expiresAt` with refresh-on-expiry inside the sync runner (Strava
  pattern).
- One `src/lib/sync/<provider>-sync.ts` runner per provider: window
  resolution (backfill 90d / overlap 7d), fetch, `applyWellnessPatch`
  per day, one `computeDailyMetrics` at the end, connection status
  bookkeeping. Registered in the scheduler's `defaultProcessor` and
  `sync_jobs.provider` enum.
- Settings: OAuth providers get a connect-redirect route pair under
  `/api/connections/<provider>/` (state cookie, callback, inline first
  backfill); token providers get a paste-key card (intervals.icu
  pattern).

### Whoop — OAuth2 (api.prod.whoop.com/developer/v2)

- Scopes: `read:recovery read:sleep read:profile offline`.
- `GET /v2/recovery` → per-cycle `score.hrv_rmssd_milli` (ms),
  `score.resting_heart_rate`; assigned to the local date of the cycle's
  end (the morning the reading represents).
- `GET /v2/activity/sleep` → stage summary totals (light/REM/deep/awake
  milli), `start`/`end` as bed window; naps (`nap: true`) are ignored;
  score → sleepScore when present. Sleep total = light + REM + deep
  (in-bed awake time is not sleep).
- Tokens: 1h access + refresh token (rotating); refreshed via the shared
  runner when `expiresAt` is near.

### Oura — Personal Access Token (api.ouraring.com/v2)

- `GET /v2/usercollection/sleep` → staged sleep (`deep_sleep_duration`,
  `rem_sleep_duration`, `light_sleep_duration`, `awake_time`,
  `total_sleep_duration`, `bedtime_start`, `bedtime_end`), long sleep
  only (`type == "long_sleep"` wins over naps for a date).
- `GET /v2/usercollection/daily_sleep` → sleep score.
- `GET /v2/usercollection/daily_readiness` → `temperature_deviation`
  (v0.15 wants it banked from day one).
- HRV/RHR: Oura's sleep rows carry `average_hrv` and `lowest_heart_rate`
  — mapped to hrvMs / restingHr.
- Token pasted in Settings, validated with `GET /v2/usercollection/personal_info`.

### First-run experience

The guided source picker (connector / CSV / manual), the "day 5 of 14"
calibrating progress state, and first-fortnight prompts. Builds on the
connector cards above; ships as its own batch.

## Batches (each ends green and reviewable)

1. **Foundation**: migration 0014 (enums + columns + `field_sources`),
   `wellness-merge.ts` + tests, intervals-sync routed through the merge.
2. **Whoop**: connector + sync runner + scheduler + connect routes +
   settings card + mapping tests.
3. **Oura**: connector + sync runner + scheduler + settings card +
   mapping tests.
4. **Apple Health**: export-file upload + Health Auto Export webhook.
5. **Withings**: OAuth (reuses Whoop's framework), weight/body-comp/BP.
6. **First-run experience** + release chores (version 0.11.0, CHANGELOG,
   ROADMAP tick, README/SELF-HOSTING env docs).

## Testing

- Merge policy: pure-function matrix — empty field, lower/higher/equal
  priority, null never erases, same-source heal, legacy rows, load
  fields locked to intervals_icu.
- Provider mapping: fixture JSON (shaped like real v2 responses) →
  expected `WellnessPatch` per date; naps ignored; missing fields null.
- Sync runners: connection gating and status bookkeeping follow the
  intervals-sync tests' DB-gated pattern where present.
- Existing suite stays green throughout; no test depends on external env.

## Out of scope

- Sleep-stage UI cards (v0.12 — Sleep Intelligence earns them back).
- Fitbit / Google Health (conditional on demand).
- Oura OAuth (see deviation), Whoop webhooks, real-time sync.
- Any change to activity ingestion — wearables here feed wellness only.
