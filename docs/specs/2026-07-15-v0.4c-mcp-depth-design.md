# v0.4c — MCP depth (power/pace curves, best efforts, load summaries)

Status: approved 2026-07-15 · Roadmap phase: v0.4 (docs/ROADMAP.md) · Sub-spec C of 3 (A → B → C)

## Goal

Claude (via MCP or the in-app coach) can answer "what's my 20-minute power?"
, "any running PRs this month?", and "how has my load trended?" with real
numbers.

**Definition of done:** four new tools appear in the MCP tool list and work
end-to-end against the owner's instance: `get_power_curve`,
`get_pace_curve`, `get_best_efforts`, `get_training_load_summary` (weekly
buckets); the coach can call the same tools; a token with `read` scope can
call all four.

## Decisions (with reasons)

- **Fetch curves/best-efforts precomputed from intervals.icu** (approved
  over local computation): intervals.icu already maintains
  athlete-level power/pace curves and best efforts. Computing locally would
  require bulk stream backfill (explicitly rejected in v0.3) plus mean-max
  math for marginal benefit. Manual/Strava-only users simply get
  "unavailable from your data sources" — consistent with provenance rules.
- **Cache with a short TTL in `app_config`-style rows? No — new
  `athlete_curves` table** keyed (userId, kind, sport) with `fetchedAt`;
  TTL 6 h. Curves change slowly; a TTL cache keeps MCP snappy and
  respects intervals.icu rate limits. Stale-if-error: on fetch failure
  serve the cached copy (with `stale: true`) rather than erroring.
- **Load summary is pure SQL over `activities`** (no connector): weekly
  buckets reuse `weeklyLoads()` from v0.3 (`src/lib/charts.ts`), plus
  totals for duration/distance per week and the CTL/ATL endpoints from
  `wellness_daily`. Strava rows excluded as everywhere.
- **All four tools are `read` scope** and registered once in
  `registry.ts` (Principle 2: one registry, two consumers).

## Components

### Connector additions (`src/lib/connectors/intervals.ts`)

- `fetchAthletePowerCurves(apiKey, {days})` → GET
  `/athlete/{id}/power-curves{?days}`; normalize to
  `{ secs: number[], watts: number[] }` (+ `wattsPerKg` when present).
- `fetchAthletePaceCurves(apiKey, {days, sport})` → GET
  `/athlete/{id}/pace-curves` for Run; normalize to
  `{ distanceM: number[], secsPerKm: number[] }`.
- `fetchBestEfforts(apiKey, {days})` → best-efforts listing normalized to
  `{ label, sport, value, unit, activityExternalId, date }[]`.
- Same Basic-auth + ConnectorError mapping as existing calls;
  fixture-based tests (the exact response shapes get pinned by fixtures
  captured from the live API during implementation — field names above are
  the normalized output, which is the contract).

### Curves service (`src/lib/athlete-curves.ts`)

- `getCurves(userId, kind: "power" | "pace", opts: {days?: 30|90|365})`:
  cache row fresh (< 6 h) → return; else decrypt key, fetch, upsert,
  return. No connection / manual-only user → `{ available: false, reason }`.
- `getBestEfforts(userId, {days})` — same pattern, kind `best_efforts`.
- Schema: `athlete_curves` (id, userId FK cascade, kind text, params text
  — e.g. "days=90", data jsonb, fetchedAt; unique on (userId, kind,
  params)).

### Tools (`src/lib/tools/`)

- `get-power-curve.ts` — args `{days?: 30|90|365 = 90}`; returns curve
  points thinned to canonical durations (5s, 1m, 5m, 20m, 60m + full
  array capped at ~50 points) so LLM context stays small.
- `get-pace-curve.ts` — args `{days?}`; running pace curve, canonical
  distances (400m, 1k, 5k, 10k, half).
- `get-best-efforts.ts` — args `{days?: 30|90|365 = 90, sport?}`.
- `get-training-load-summary.ts` — args `{weeks?: 4|12|26 = 12}`; weekly
  load/duration/distance/sessions + current CTL/ATL/TSB. Pure db.
- All registered in `registry.ts`; MCP tool count assertions in existing
  tests updated (11 → 15 — sub-spec A's remember/forget land first).

## Error handling

- intervals.icu down: stale cache if present (`stale: true` in payload),
  else `{ available: false, reason: "fetch_failed" }` — tools never throw
  to the MCP layer.
- Rate-limited (429 → ConnectorError rate_limited): same stale-or-
  unavailable path; never retried in-request.
- Cross-user isolation: all queries filter by ctx.userId (as existing
  tools; covered by MCP security tests).

## Testing

- Connector: fixtures incl. missing fields and malformed rows; auth-error
  mapping.
- Curves service (pg, mocked connector): cache miss fetches + stores;
  fresh cache skips connector; TTL expiry refetches; fetch failure with
  cache → stale payload; without cache → unavailable; manual-only user →
  unavailable.
- Tools (registry): schema validation, canonical thinning
  (hand-computed fixture), load summary against seeded activities
  (Strava rows present but excluded).
- MCP integration (extend `tests/mcp-security.test.ts` counts +
  `read`-scope access for the four tools).
- Smoke: MCP initialize lists 15 tools; `get_training_load_summary`
  returns real buckets on the demo stack.

## Out of scope

Local mean-max computation from streams; per-activity curve views in the
UI (tools/MCP only for v0.4 — UI curves can come with v0.6 alongside
Strava streams); FTP/threshold estimation from curves (v0.6 body
intelligence); caching negotiation with intervals.icu beyond the 6 h TTL.
