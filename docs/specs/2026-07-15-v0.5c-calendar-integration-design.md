# v0.5c — Calendar Integration Design

> Sub-project C of v0.5. Two phases: intervals.icu planned workouts (immediate,
> no new OAuth) then Google Calendar (new OAuth flow).

## Problem

The coach doesn't know what's planned — it can only react to what already
happened. Athletes plan training in intervals.icu or schedule around life in
Google Calendar. Giving the coach visibility into both makes its suggestions
contextual: "You have a hard interval session tomorrow but your readiness is
amber — consider moving it to Thursday when you're free."

## Phase 1: intervals.icu Planned Workouts

### API

intervals.icu exposes planned workouts at:
```
GET /api/v1/athlete/{id}/events?oldest={date}&newest={date}&category=WORKOUT
```

Returns planned workout events with: name, description, category, start_date,
duration, load (target TSS), sport, workout structure (intervals).

### Implementation

**New fetcher** (append to `src/lib/connectors/intervals.ts`):

```ts
export interface IntervalsPlannedWorkout {
  id: string;
  name: string;
  sport: string;
  date: string;          // YYYY-MM-DD
  durationMins: number;
  targetLoad: number | null;
  description: string | null;
}

export async function fetchPlannedWorkouts(params: {
  apiKey: string;
  athleteId: string;
  startDate: string; // YYYY-MM-DD
  endDate: string;
}): Promise<IntervalsPlannedWorkout[]>
```

**Cache:** Use existing `athlete_curves` table:
- `kind: "planned_workouts"`
- `params: "start=2026-07-15&end=2026-07-22"`
- TTL: 1 hour (plans change more often than curves)

**Tool:**

```ts
// src/lib/tools/get-planned-workouts.ts
name: "get_planned_workouts"
parameters: z.object({
  days: z.number().int().min(1).max(14).default(7)
    .describe("Look-ahead window in days (1-14)."),
})
scope: "read"
```

Returns: array of planned workouts for the next N days, or
`{ available: false, reason: "no_connection" }` for manual-only users.

### System Prompt Addition

```
## Training Calendar

You have access to the athlete's planned workouts via `get_planned_workouts`.
When discussing what to do next or whether to adjust training:
- Check what's planned for today and tomorrow
- If readiness is red/amber and a hard session is planned, suggest moving it
- Reference the plan naturally: "You have hill repeats planned for Thursday"
```

## Phase 2: Google Calendar OAuth

### OAuth Flow

Follow the Strava pattern exactly:

1. **Initiate:** `GET /api/connections/google` → redirect to Google OAuth consent
   - Scopes: `https://www.googleapis.com/auth/calendar.readonly`
   - (Later add `.events` for write-back)
   - State cookie: httpOnly, 600s expiry, random nonce
2. **Callback:** `GET /api/connections/google/callback` → exchange code → store tokens
   - Provider: `google_calendar` in `connections` table
   - Store encrypted refresh token + access token
   - `externalAthleteId` → Google calendar ID (primary)

### Env Vars

```
GOOGLE_CLIENT_ID=
GOOGLE_CLIENT_SECRET=
```

### API Client

```ts
// src/lib/connectors/google-calendar.ts

export interface CalendarBusyBlock {
  start: string; // ISO datetime
  end: string;
  summary?: string; // "Team standup" — calendar event title
}

export async function fetchBusyTimes(params: {
  accessToken: string;
  startDate: string;
  endDate: string;
}): Promise<CalendarBusyBlock[]>
```

Uses Google Calendar FreeBusy API (`POST /freeBusy/query`) for privacy — only
returns busy/free, not full event details (unless user opts in).

### Token Refresh

Google access tokens expire after 1 hour. Add a `refreshGoogleToken(connection)`
helper that uses the encrypted refresh token to get a new access token. Call
before every API request (same pattern as Strava refresh).

### Tool

```ts
// src/lib/tools/get-calendar-availability.ts
name: "get_calendar_availability"
parameters: z.object({
  days: z.number().int().min(1).max(7).default(3)
    .describe("Look-ahead window (1-7 days)."),
})
scope: "read"
```

Returns: array of busy blocks, or `{ available: false, reason: "no_connection" }`.

### Write-Back (future, not v0.5c scope)

Once Training Plans (v0.5d) exist, the coach could push workouts to Google
Calendar. This requires the `calendar.events` scope and a confirmation UX
("Add this week's training to your calendar?"). Defer to v0.5d or later.
