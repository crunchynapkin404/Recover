# v0.13 — Deep Biology: Design

**Date:** 2026-07-18
**Status:** Approved

## Goal

Long-horizon health metrics. Deferred twice because the data wasn't there
(0/368 days of blood pressure in the live DB); v0.11's Withings connector
and this release's blood-test extraction fix the input side. Done when: a
blood test is parsed, reviewed, and appears as trends — and a missing
biomarker shows as missing, not interpolated.

## Principle (unchanged from the brand)

Nothing enters the DB unconfirmed. LLM-extracted biomarkers carry a
per-value confidence and go through a human review screen before they are
stored. Biological age reports an honest "insufficient inputs" state that
names what's missing rather than guessing. The coach references bloodwork
trends but never diagnoses.

## Schema (migration 0015, additive)

- **`biomarkers`** table: `id`, `userId`, `name` (canonical slug, e.g.
  `ldl_cholesterol`), `displayName`, `value` (real), `unit`, `measuredAt`
  (date), `source` (`blood_test` | `manual` | `withings`), `confidence`
  (real 0–1, null for manual), `category` (`lipids` | `metabolic` |
  `hematology` | `hormones` | `vitamins` | `organ` | `other`), `rawLabel`
  (the source's original label), `createdAt`. Unique on
  `(userId, name, measuredAt)` so re-import updates rather than duplicates.
- **`body_prefs.birth_year`** (integer, nullable): enables the biological-
  age estimate. Blood pressure already lands in `wellness_daily.systolic/
diastolic` (v0.11), so no new column for BP.

## Pure engines

### `src/lib/blood-pressure.ts`

- `classifyBp(systolic, diastolic)` → `{ category, band }` against the
  ACC/AHA guideline bands (Normal / Elevated / Stage 1 / Stage 2 /
  Crisis), taking the higher-severity of the two readings. Null on missing
  input.
- `bpTrend(readings)` → recent average + direction vs the prior window,
  gated on a minimum count. No reading → null (never a fabricated band).

### `src/lib/biological-age.ts`

- `biologicalAge(inputs)` where inputs are the honest, already-computed
  signals: RHR baseline trend, HRV median, sleep-consistency score,
  VO2max (if a provider sent it), body-fat %, and chronological age (from
  `birthYear`). Each available component maps to a small +/− year offset
  from a neutral baseline; the estimate is `chronologicalAge + Σ offsets`,
  clamped to a sane range.
- Returns `{ bioAge, deltaYears, components }` when there are at least
  `MIN_BIOAGE_COMPONENTS` inputs _and_ a birth year; otherwise
  `{ insufficient: true, missing: string[], have: string[] }`. Never
  invents an age from thin data. Pure and fully unit-tested — the offsets
  are the tested contract.

### `src/lib/health-records.ts`

- `normalizeBiomarker(rawLabel)` → canonical `{ name, displayName,
category }` via a curated synonym map (e.g. "LDL-C", "LDL Cholesterol"
  → `ldl_cholesterol`), else `other` with a slugified name.
- `validateExtraction(json)` → parse the LLM's array into typed,
  range-sane `ExtractedBiomarker[]` (drop non-numeric values, clamp
  confidence 0–1, require a unit or mark low-confidence). Pure — this is
  the tested boundary; the LLM call itself is thin.

## Health Records flow

1. **Upload** (`/health` page + action): paste lab text, or upload a PDF/
   image. PDFs/images go to the user's own LLM (`resolveProvider`) as a
   multimodal message with a strict extraction prompt asking for a JSON
   array of `{ rawLabel, value, unit, confidence }`; pasted text takes the
   same prompt as plain text.
2. **Validate**: `validateExtraction` + `normalizeBiomarker` shape the
   result server-side. Nothing is stored yet.
3. **Review**: the parsed rows render in an editable review table with the
   per-value confidence surfaced; the athlete corrects/deletes and picks
   the measurement date, then confirms.
4. **Store**: confirmed rows upsert into `biomarkers`. Only now do they
   exist.

If the user has no LLM configured, text paste still works via a
deterministic line parser (`LABEL  VALUE UNIT`) so the feature isn't
LLM-gated for the manual path.

## Coach visibility

- `get_biomarkers` tool (`scope: read`): returns the athlete's latest
  value per biomarker plus its trend, and the biological-age summary. Its
  description sharply bounds it: "surface trends and reference ranges;
  never diagnose, never recommend treatment — suggest seeing a clinician
  for anything abnormal." Registered in `allTools` (registry 48 → 49).

## UI

- **`/health`** page: upload/review flow, a biomarker list grouped by
  category with the latest value + sparkline trend, the blood-pressure
  card (bands + trend), and the biological-age card (or its insufficient-
  inputs state listing what to add). Linked from Settings and the desktop
  sidebar.
- **Dashboard**: a compact biological-age tile when computable; otherwise
  nothing (no invented number).

## Testing

- BP classification across every band boundary; trend direction + gate.
- Bio-age: each component offset, the min-components / no-birth-year
  insufficient states, clamp bounds.
- Health records: synonym normalization, extraction validation (drops bad
  rows, clamps confidence, slug fallback), deterministic text parser.
- `get_biomarkers` shaping + the connection/empty states.
- Existing suite stays green; the LLM call is mocked, never live in tests.

## Out of scope

- Diagnosis, treatment, or reference-range _interpretation_ beyond the
  guideline bands (BP) and the neutral bio-age offsets.
- Continuous glucose / lab-integration APIs (upload only).
- v0.14 forecasts; v0.15 cycle-aware readiness.
