# v0.18 (slice 1) — Security Hardening: Design

**Date:** 2026-07-20
**Status:** In review

## Goal

The first slice of the 1.0-hardening release: a **trustable security pass**
over the current codebase (post-v0.19). Ship the cheap, high-value web-security
fixes; run the exhaustive per-user isolation and input audit the roadmap's
"full pass before the tag" calls for; and add a light auth/token audit log.
Done when: the app sets real security headers, login is rate-limited, every
route/action/tool is confirmed scoped to its user (or fixed), and key auth and
token events are recorded — with the full test suite green.

## Why this is the shape it is

v0.18 as written in the roadmap is **seven independent workstreams** (auth
hardening, accessibility sweep, upgrade guarantees, performance, API/MCP
freeze, docs, security review). That is too much for one spec. This spec is
**slice 1 of v0.18: the security slice.** The a11y sweep, API/MCP freeze,
docs review, performance pass, and upgrade guarantees are each their own
later spec.

The scope of _this_ slice was set by an actual first-pass security review of
the codebase, not by the roadmap's speculative feature list. The review's
threat model is the deciding lens: **self-hosted, invite-only, no public
signup, ~10 friends, behind a Cloudflare tunnel, single owner.** That model
already blunts mass-credential and hostile-signup attacks, which is why the
roadmap's headline auth features (2FA, passkeys, full session-management UI)
are **explicitly deferred past 1.0** here — real work for a threat the
deployment model already addresses. What remains is the security work that is
either cheap-and-high-value or genuinely load-bearing for a 1.0 an owner would
point friends at.

## What the review confirmed is already solid (no action)

Stated so the spec is honest about where the bar already sits:

- **Route guard runs.** Optimistic `middleware` redirect on missing session
  cookie, plus server-side `requireUser()` on every page and server action.
  The old `proxy()`-vs-`middleware()` bug (guard never ran) is gone.
- **Cross-user isolation holds on the paths checked.** Queries are
  `userId`-scoped; the MCP layer threads the token's `userId` into every tool
  via `authInfo.extra.userId`; the export endpoint is `userId`-scoped.
- **MCP tokens**: SHA-256 hashed, compared with `timingSafeEqual`, scoped,
  rate-limited.
- **Crypto**: AES-256-GCM; `ENCRYPTION_KEY` validated at boot (fails loud,
  correct length).
- **No SQL string concatenation** (all Drizzle `sql\`\`` parameterized), no
secrets logged, invite-only enforced (`disableSignUp: true`), Strava
  compliance firewall intact.

The audit (§2) extends this from "spot-checked" to "systematically
confirmed."

## Scope

Three parts, one release:

1. **Cheap web-security fixes** — findings 1–7 from the review.
2. **Exhaustive isolation & input audit** — finding 8, the substance.
3. **Light auth/token audit log** — a modest, 1.0-appropriate record.

### Explicitly deferred past 1.0 (not in this spec)

- **2FA / TOTP and passkeys** — only the `nextCookies` Better Auth plugin is
  configured today. Adding the `twoFactor`/`passkey` plugins plus enrolment
  UI and recovery codes is real work for a threat the invite-only + tunnel
  model already blunts. Deferred.
- **Full session-management UI** (list/revoke devices across sessions) —
  nice-to-have; deferred. (Token revocation already exists for MCP tokens.)

### Explicitly not in this slice (separate later v0.18 specs)

Accessibility sweep, API/MCP schema freeze + deprecation policy, end-to-end
docs review, performance pass, upgrade guarantees.

## Part 1 — Cheap web-security fixes

### 1. HTTP security headers

**File:** `next.config.ts` (`async headers()` returning a header set for all
routes).

Ship a **pragmatic header set**, not a full strict CSP (design decision
below):

- `X-Frame-Options: DENY` and CSP `frame-ancestors 'none'` — clickjacking.
- `X-Content-Type-Options: nosniff` — MIME sniffing.
- `Referrer-Policy: strict-origin-when-cross-origin`.
- `Strict-Transport-Security: max-age=63072000; includeSubDomains` — HSTS.
- `Permissions-Policy` — deny the sensors/features the app doesn't use
  (camera, microphone excepted only where the app genuinely uses them — the
  coach mic is Web Speech API in-page; confirm before denying `microphone`).

**Design decision — pragmatic headers now, strict script-src CSP later.** A
full `Content-Security-Policy` with a strict `script-src` fights Next.js's
inline bootstrap scripts and needs per-request nonce plumbing. That is its own
project and a poor fit for this slice. The header set above already closes
clickjacking, MIME-sniffing, referrer leakage, and downgrade — the highest-value
gaps. A strict script-src CSP is recorded as a later hardening item, not
attempted here. `frame-ancestors 'none'` is included now because it does not
require nonces.

### 2. Login rate-limiting

**File:** `src/lib/auth.ts`.

Configure Better Auth's `rateLimit` explicitly (enabled, sensible
window/max) rather than relying on unstated defaults, so a leaked or weak
friend password can't be brute-forced. Storage: in-memory is acceptable for
this single-instance deployment; note the trade-off (counters reset on
restart) in a comment.

### 3. `BETTER_AUTH_SECRET` boot validation

**File:** `src/instrumentation.ts` (boot hook; already used for owner
self-seed).

Mirror what `crypto.ts` does for `ENCRYPTION_KEY`: at startup, assert
`BETTER_AUTH_SECRET` is present and of adequate length; throw loud if missing
or weak. Prevents Better Auth silently falling back to an ephemeral/degraded
secret (which would break sessions on restart and weaken cookie signing).

### 4. Apple-ingest `?token=` mitigation

**File:** `src/app/api/connections/apple-health/ingest/route.ts`.

**Design decision — mitigate, don't remove.** The Health Auto Export app
posts to a URL, so the shown-once ingest URL genuinely needs the query-param
token; removing it breaks a real integration. Keep the `X-Recover-Token`
header as the primary path (already supported) and, for the query-param path,
set `Referrer-Policy: no-referrer` and `Cache-Control: no-store` on the route
response so the token cannot leak downstream via referrer or a cache. The
token is already a dedicated, SHA-256-hashed, revocable ingest token — this
closes the leakage vector without a UX break. Document the header-preferred
guidance in the connector docs.

### 5. Upload cap on bytes actually read

**File:** `src/app/api/connections/apple-health/ingest/route.ts`.

The 10 MB cap currently trusts the `content-length` header, which a client
can omit or understate while streaming a larger body. Enforce the limit on
bytes actually consumed from the request stream — read with a running byte
count and abort (413) when it exceeds `MAX_BODY_BYTES`. Keep the fast
`content-length` pre-check as an early-out, but make the stream count
authoritative.

### 6. Dependency bump

**Files:** `package.json`, `package-lock.json`.

A **considered, tested** upgrade off the 7 moderate-severity versions
(postcss XSS transitively via next; better-auth behind on
drizzle-kit/next) — **never** the `npm audit fix --force` path, which
proposes a breaking downgrade to `next@9`. Bump next/postcss/better-auth to
current non-vulnerable releases, then verify tsc, lint, full suite, and
`next build` all pass. If a clean bump isn't reachable without a breaking
major, record the residual advisory and its non-exploitability here rather
than forcing it.

### 7. `middleware.ts` → `proxy.ts`

**Files:** rename `src/middleware.ts` to `src/proxy.ts`, keeping the exported
function name Next 16 expects (per `AGENTS.md`: this version uses `proxy.ts`,
not `middleware.ts`). Confirm the guard still fires (redirect on missing
session; matcher unchanged) after the rename.

## Part 2 — Exhaustive isolation & input audit

Finding 8, the substance of this slice. A **systematic** pass that turns
"spot-checked" into "confirmed," producing a pass/fail checklist and folding
any real fix (plus a regression test) into this release. Where a surface is
already clean, add a confirming test so 1.0 can assert the property rather
than assume it.

Surfaces to enumerate and confirm:

- **Every route handler** under `src/app/api/**` — confirmed to authenticate
  (session, MCP bearer, cron secret, or ingest token as appropriate) and to
  scope every data read/write to the authenticated principal.
- **Every server action** — confirmed to start from `requireUser()` and scope
  by `user.id`; no action trusts a client-supplied user/owner id.
- **Every MCP tool** — confirmed to read only the token's `userId` data (the
  `authInfo.extra.userId` thread), never a caller-supplied id.
- **Every OAuth callback** (strava, whoop, withings, google) — confirmed to
  validate the `state` parameter against what was issued (CSRF on the connect
  flow).
- **Webhook signature verification** (Whoop, Withings, Apple Health) —
  confirmed each verifies its provider signature / shared secret before
  acting on a payload.
- **LLM injection containment** — re-confirm the biomarker-extraction path
  (deep slot, no tools, deterministic fallback, review-before-save) can't be
  driven to act on injected instructions; confirm coach tool-calls can't reach
  another user's data.
- **File-parser safety** — the blood-test PDF/image upload path reviewed for
  parser abuse (size already capped per §5; confirm no unbounded work / no
  parser reaching the filesystem or network).

Output: a checklist committed alongside the code (each surface marked
confirmed-clean-with-test or fixed-with-test). This is where any _new_ real
vulnerability would surface; the spec's size flexes to whatever it finds.

## Part 3 — Light auth/token audit log

A modest record of security-relevant events — enough for a 1.0 to answer
"what happened to this account," without building a full SIEM.

**Schema (new table, migration number TBD — see Coordination):**

`audit_log`:

- `id` uuid pk
- `userId` text nullable, `references users.id` (nullable because a failed
  login has no authenticated user)
- `event` text enum: `login_success`, `login_fail`, `token_created`,
  `token_revoked`, `connection_added`, `connection_revoked`
- `ip` text nullable
- `userAgent` text nullable
- `metadata` jsonb nullable (e.g. token label, provider name — never a secret)
- `createdAt` timestamptz default now

**Write sites:** the login success/failure paths, MCP token create/revoke
actions, and connector connect/disconnect actions. A `recordAuditEvent(...)`
helper centralizes the write so no call site assembles rows ad hoc.

**View:** a simple owner-only "Recent security events" section in the existing
`/admin` page (owner-gated already), newest first, capped to a recent window.

**Honesty on failed logins:** a `login_fail` row records the attempt (email
attempted, ip) but the response to the user never reveals which field was
wrong — no account-enumeration regression.

## Testing

- **Headers**: a response from a representative route carries the six headers
  from §1 with the specified values.
- **Rate-limit**: repeated failed logins from one origin get throttled
  (429/blocked) within the configured window.
- **Boot validation**: instrumentation throws when `BETTER_AUTH_SECRET` is
  absent/too short (unit test around the validator).
- **Upload cap**: a request whose body exceeds 10 MB is rejected with 413 even
  when `content-length` is absent or understated.
- **Apple token route**: the response carries `Referrer-Policy: no-referrer`
  and `Cache-Control: no-store`.
- **Isolation**: for each gap the audit finds, a regression test proving the
  cross-user/cross-token read is denied; for confirmed-clean high-value
  surfaces, a confirming test.
- **Audit log**: each of the six events writes exactly one row via the helper;
  the admin view renders recent events; a `login_fail` row is written without
  the response leaking which field was wrong.
- Full suite green; tsc, lint, `next build`, `prettier --check` clean.

## Non-goals

- No 2FA, passkeys, or full session-management UI (deferred past 1.0).
- No strict script-src CSP (pragmatic header set only; strict CSP is a later
  item).
- No accessibility sweep, API/MCP freeze, docs review, performance pass, or
  upgrade guarantees — each is a separate later v0.18 spec.
- No new user-facing features. This slice only hardens what exists.

## Risks

- **The audit may surface a real vulnerability** that expands scope. That is
  the point of doing it; the spec flexes. A Critical finding is fixed in this
  slice regardless of size; a lower-severity one may be recorded and scheduled.
- **The dependency bump may not reach a clean tree** without a breaking major.
  Mitigation: bump what bumps cleanly, record and justify any residual
  advisory rather than forcing `next@9`.
- **Migration-number collision with v0.17** (see Coordination) — mitigated by
  confirming the free number at implementation time.
