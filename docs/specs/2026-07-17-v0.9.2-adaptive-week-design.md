# v0.9.2 — Adaptive Week (Smarter Coach)

**Date:** 2026-07-17
**Status:** approved design, pre-implementation

## Why

v0.5d generates a full periodized plan up front: every week's workouts are
written out to race day. Real life then diverges — a missed week, a red
readiness morning, a Wednesday that filled up with meetings — and the plan
does not move. The weekly review computes `adherencePct` after the fact and
increments `currentWeek`, but nothing ahead of the athlete ever changes.
The plan silently falls behind the athlete, or worse, the athlete behind
the plan.

The fix follows the model JOIN (join.cc) proved for cycling: **concrete
workouts should never exist more than one week ahead.** At the start of
each week the athlete states availability (hours per day); every morning
the day's workout adapts to measured readiness and the time actually
available. Recover has richer inputs than JOIN for every signal it uses:
a computed readiness score (HRV/RHR/sleep/TSB) instead of a mood
questionnaire, the v0.7 honest journal sliders, and the v0.5c Google
Calendar integration for availability prefill.

## Decisions taken (with the athlete)

1. **The v0.5d skeleton stays.** Phases and weekly target loads toward the
   race date remain the guideline; concrete workouts materialize only at
   week start. The long-horizon view (base/build/peak/taper, taper timing)
   stays visible.
2. **Availability = weekly intake, calendar-prefilled.** A short
   hours-per-day intake at week start, prefilled from the plan's constraints
   and Google Calendar busy blocks. Adjustable any day.
3. **Daily adaptation has full range.** Scale, shorten, move, swap, or drop
   today's workout, with redistribution across the remaining week — bounded
   by the week's effective target load. Automatic, explained afterwards;
   never a confirmation gate.

## Data model

Two new tables; no changes to `training_plans` / `training_blocks`.

### `week_plans` — the living week

| column                    | type        | notes                                            |
| ------------------------- | ----------- | ------------------------------------------------ |
| `id`                      | uuid PK     |                                                  |
| `userId`                  | text FK     | cascade delete, same pattern as other tables     |
| `planId`                  | uuid FK     | the active `training_plans` row                  |
| `weekStart`               | date        | Monday; `UNIQUE(userId, weekStart)`              |
| `skeletonWeek`            | smallint    | week number in the skeleton this week draws from |
| `days`                    | jsonb       | always 7 entries, see below                      |
| `status`                  | text enum   | `open` \| `closed`                               |
| `createdAt` / `updatedAt` | timestamptz |                                                  |

Each `days` entry:

```ts
{
  date: string;            // YYYY-MM-DD
  availableMins: number;   // 0 = rest day by availability
  workout: PlannedWorkout | null;
  status: "planned" | "completed" | "adapted" | "moved" | "missed" | "rest";
  activityId?: string;     // matched actual activity
  actualLoad?: number;
}
```

The week is always read and rewritten as a whole (materialization and the
morning adapter both operate on the full week), so days live as jsonb in
one row — atomic, consistent with `training_blocks.workouts` — rather than
seven child rows.

### `plan_adjustments` — the audit trail

| column       | type        | notes                                                                                          |
| ------------ | ----------- | ---------------------------------------------------------------------------------------------- |
| `id`         | uuid PK     |                                                                                                |
| `weekPlanId` | uuid FK     | cascade                                                                                        |
| `date`       | date        | the day the adjustment applies to                                                              |
| `trigger`    | text enum   | `low_readiness` \| `no_time` \| `missed_workout` \| `availability_change` \| `weekly_rollover` |
| `action`     | text enum   | `scaled` \| `moved` \| `swapped` \| `dropped` \| `redistributed`                               |
| `before`     | jsonb       | affected day slot(s) before                                                                    |
| `after`      | jsonb       | affected day slot(s) after                                                                     |
| `reason`     | text        | deterministic, human-readable sentence                                                         |
| `createdAt`  | timestamptz |                                                                                                |

This table is the "coach can explain what it changed and why" mechanism:
the explaining LLM and the UI quote from it; neither invents anything.

## Engines

Both engines are pure functions — no LLM, no DB access in the core
(state in → new week state + adjustments out) — in the style of
`sleep-debt.ts` and the v0.5d generator. All thresholds are named
constants.

### Engine 1 — `materializeWeek` (runs at weekly rollover)

**Input:** skeleton block (target load, phase, target sessions),
availability per day, readiness trend (7-day average vs baseline),
closed week's adherence.

**Rules:**

1. **Effective week load** = skeleton target, corrected:
   - previous week adherence < 70% → do not stack: target = previous
     week's _actual_ load × 1.10;
   - readiness trend suppressed — ≥4 of the last 7 days scored amber or
     worse → −15%;
   - always clamped to **±20% of the previous week's actual load**
     (ramp guard — a missed week reshapes what's ahead, gradually).
2. **Session layout:** the v0.5d phase-appropriate workout generator,
   reused, but distributed over the days _with_ availability — longest
   session on the roomiest day, quality sessions (intervals/tempo) never
   on consecutive days, `availableMins: 0` → `rest`.
3. **Availability wins.** If total session time does not fit the stated
   hours, the load drops to fit and a `weekly_rollover` adjustment records
   it ("6h available instead of 8h — week load lowered to …").

### Engine 2 — `adaptDay` (runs every morning after readiness)

**Input:** week state, today's readiness band, current availability,
yesterday's completion (activity matching on day + sport).

**Rules, in order:**

1. **Yesterday missed?** → status `missed`; a quality session moves to the
   next day with room (at most once — missed twice = dropped, its load
   redistributed over the remaining week, capped at +25% per day).
2. **Readiness red** → today's quality session becomes recovery or rest;
   an endurance session is shortened −30%.
3. **Readiness amber** → one step down: intervals → tempo → endurance;
   duration −15%.
4. **Availability shrank** below the workout's duration → shorten to fit,
   or swap with a lighter day when that covers the week better.
5. **Readiness `calibrating`** → availability-only adjustments. No
   readiness is never treated as a readiness (v0.7 principle).

Every rule that fires writes one `plan_adjustments` row. The morning brief
receives those rows and phrases them — with a template fallback when no
LLM key is configured, exactly like the weekly review today.

## Week lifecycle

- **Rollover** (inside the existing weekly-review job, same
  crash-safety order: stored review first as idempotency marker, plan
  writes after): close the open week (write `actualLoad` /
  `adherencePct` back to the skeleton block — unchanged behavior),
  then materialize the next week.
- **Intake:** rollover **always materializes the week immediately** from
  the prefill, so an ignored intake never stalls the plan. Prefill: hours
  pattern from plan `constraints` distributed like last week (first week:
  distributed evenly over `daysPerWeek` days), minus a reduction on days
  Google Calendar shows as busy (`fetchBusyTimes`, existing). The intake
  then stays open through the week's first day: confirming or editing it
  applies the stated hours as an `availability_change` and re-runs
  materialization for the days not yet completed.
- **Daily:** the morning pipeline (post-sync, after readiness) runs
  `adaptDay`. Availability edits during the day re-run it.

## UI

- **Dashboard — "Today" card:** today's workout (type, duration,
  intensity, description) plus, when the adapter intervened, a one-line
  explanation quoted from the adjustments log.
- **Dashboard — week strip:** 7 status-colored day dots
  (done / adapted / missed / rest / upcoming). This _is_ the continuous
  adherence surface — no separate adherence widget.
- **Plan page:** week detail (7 days written out + that week's
  adjustment history) above the existing skeleton view; the intake form
  appears here while an intake is open.
- **No plan → no-op.** No week plan exists, nothing new renders. Athletes
  without an active plan (manual-first, v0.8) notice nothing.

## Coach integration

Via the tool registry (one registry, two consumers — LLM coach and MCP
get these simultaneously):

- **New:** `get_week_plan` (current week state + adjustments log),
  `set_week_availability` ("no time Wednesday anymore" → adapter re-runs),
  `get_plan_drift` (actual vs skeleton load over past weeks; how far the
  effective targets have diverged).
- **Extended:** `update_training_plan` gains day-level
  `move_workout` / `swap_workout` actions (existing coarse week actions
  stay).
- **Context:** morning brief and weekly review receive adjustments/drift
  as prompt context — they explain, they never compute.

## Honesty guardrails

- `calibrating` readiness → no readiness-based adaptation, ever.
- No availability stated → pattern fallback, recorded as such in the
  adjustment reason — never presented as the athlete's own input.
- Every automatic change is logged with its deterministic reason before
  anything user-facing mentions it.
- No invented RPE prompts: measured readiness and the v0.7 sliders are
  the subjective inputs.

## Testing

- **Engines (the bulk):** hand-computed fixtures per rule — rule fires,
  rule does not fire just under the threshold, clamps hold (±20% ramp,
  +25% day cap, no consecutive quality days). Honesty cases: calibrating
  → availability-only; zero availability → rest, never an invented
  session.
- **Rollover/adapter integration:** `describe.skipIf(!hasDb)` suites —
  close week, write back adherence, materialize next, adjustments logged,
  idempotent on re-run.
- **Components:** week strip and today card via `renderToString`;
  "no week plan → no strip" (no empty claims).

## Migration & rollout

- One additive Drizzle migration (two tables).
- Existing active plans join automatically: their first living week
  materializes at the next weekly rollover. No backfill.
- No active plan → everything is a no-op.

## Scope

**In:** the two tables, both engines, rollover + morning integration,
intake UI, today card + week strip, plan-page week detail, three new
coach tools + `update_training_plan` extension, morning-brief/weekly-review
context.

**Out (explicitly):** push-notification flows for the intake (the morning
push and dashboard are the channels), maintenance mode (adaptive week without a
race plan — future, does not touch this design), post-workout RPE prompts,
multilingual explanation copy.

## Done when

- A week without training visibly and boundedly lowers the next week's
  target (ramp guard observable in the adjustments log).
- A red-readiness morning demonstrably turns today's intervals into
  recovery, with an explanation line on the dashboard.
- Telling the coach "no time Wednesday" reshuffles the week.
- Every automatic change can be read back in the adjustments log.
