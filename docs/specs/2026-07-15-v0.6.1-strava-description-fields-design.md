# v0.6.1 — Strava Description Field Selection Design

> Let users choose which metrics appear in their Strava descriptions. The v0.6
> template is already deterministic — this promotes the hardcoded field list to
> user config without changing the formatter's structure.

## Problem

v0.6 ships a fixed description template. Every opted-in user publishes the same
five groups with the same twelve metrics, whether or not they mean anything to
that user or their followers. TRIMP and carbs-per-hour are noise to a lot of
people; the header line repeats the activity title that Strava already renders
directly above the description.

The output is public and lands under the user's name. "I want fewer numbers on
my rides" currently has no answer short of turning auto-describe off entirely.

## Scope

Field **selection** only — which metrics appear. Not reordering, not custom
labels or emoji, not free-form templates. See Non-Goals for why.

## Architecture

The v0.6 formatter already omits null fields and drops any group whose fields
are all null (`joinSegments` returns `null` → the line is never pushed). Field
selection rides on that: a disabled field is gated to `null` before it reaches
`joinSegments`, and every existing guarantee holds unchanged.

```
formatActivityDescription(input, fields?)
  → per field: enabled && value != null ? render : null
  → joinSegments drops nulls, joins survivors
  → empty group → line omitted (existing behavior)
  → all groups empty → "" → caller skips the Strava write
```

No new branching. No change to `buildDescription`, the marker, or the
append/skip mechanics.

## Config Model

A flat record of enabled field keys, stored as JSONB.

```ts
export type DescriptionField =
  | "header" // sport emoji + activity title
  | "load" // TL
  | "intensity" // IF %
  | "trimp"
  | "powerHrRatio" // Pw:Hr
  | "decoupling"
  | "pace" // runs only
  | "carbs"
  | "ctl"
  | "tsb"
  | "eftp"
  | "vo2max"
  | "prs";

/** null/undefined = all fields enabled (v0.6 behavior, byte-identical). */
export type DescriptionFields = Partial<
  Record<DescriptionField, boolean>
> | null;
```

**Resolution rules — exactly two, no third case:**

1. `null` or `undefined` → **every** field enabled. This is v0.6 behavior,
   byte-identical, and it is what every existing row means.
2. An object → **explicit allowlist**: a field is enabled if and only if its key
   is present _and_ `true`. Missing keys are disabled; `{}` disables everything.

Rule 2 is an allowlist rather than a denylist specifically because the output is
public: when a later version adds a new `DescriptionField`, it must not
silently start appearing in the descriptions of users who already saved a
config. New fields default to off for configured users, and stay on for
unconfigured (`null`) ones — who by definition are still opted into "show me
everything".

Existing rows need no backfill. Unknown keys in stored JSONB are ignored rather
than throwing, so a downgrade or a removed field cannot break the write path.

### Global, not per-sport

One field set covers all sports. The existing `isRunSport()` branch keeps
choosing pace-vs-power, and null-omission already handles inapplicable metrics —
a swim has no `Pw:Hr`, so that field vanishes without any per-sport config.
Per-sport override sets are a possible later addition; the schema does not
preclude them.

### The marker is not configurable

`MARKER = "\n📊 Recover"` is the idempotency key. `buildDescription` uses
`existing.includes(MARKER)` to decide skip-vs-append; without it, every re-sync
appends another block. It is deliberately absent from `DescriptionField` and
must stay that way.

## Storage

```sql
ALTER TABLE notification_prefs
  ADD COLUMN strava_description_fields jsonb;
```

Semantically a stretch — this is not a notification preference — but
`autoDescribeStrava` already lives on `notificationPrefs` (`schema.ts`, the
`auto_describe_strava` column), and a dedicated table for one nullable column is
worse. Both Strava description settings stay together.

## Empty-Output Guard

If the enabled set produces no lines, `formatActivityDescription` returns `""`.
Publishing that means `buildDescription(null, "")` writes a lone `📊 Recover`
marker to the user's followers — and the marker then guarantees it is never
retried or corrected.

**Rule: when the generated description is empty, skip the Strava write
entirely.** Do not PUT, do not mark.

This is enforced in the describer's write path, not in the settings UI, because
the `describe_strava_activity` coach tool is a second route to the same write.
The UI may additionally warn, but the UI is not the enforcement point.

### Preview data

Renders against the user's most recent activity that has `raw` metrics,
resolved by a server action. Falls back to canned sample data when the user has
no synced activities with raw data (new users, manual-only users).

Fidelity matters here specifically because the output is public: a preview
against real data shows the user that their VO2 field is empty on every ride,
which sample data would hide.

## Data Flow Changes

There is exactly **one** write path: `describeActivityOnStrava`. Both consumers
already funnel through it — `runAutoDescribeStrava` (the post-sync hook) calls
it in a loop, and the `describe_strava_activity` coach tool calls it directly.

So the prefs lookup and the empty-output guard live in that one function.
Neither `scheduler.ts` nor `describe-strava-activity.ts` changes.

The tool's existing `style` parameter is unaffected and does not override the
field set.

**Why a separate types module:** `schema.ts` needs the `DescriptionFields` type
for the JSONB column, and `strava-describer.ts` already imports `schema`. Put
the type in the describer and that's an import cycle. `strava-description-fields.ts`
imports nothing, so both can depend on it.

No registry change — tool count stays at 21.

## Untrusted Input

The field set arrives from a browser and lands in JSONB. `sanitizeDescriptionFields`
runs in the server action before any write: it keys off `ALL_DESCRIPTION_FIELDS`
and keeps only known keys whose value is exactly `true`. Unknown keys are
dropped rather than stored, so a client cannot write arbitrary JSON into the
column.

## Testing

The formatter is pure, so most coverage is cheap unit tests:

- **Backward compatibility (golden):** existing v0.6 formatter tests pass
  unchanged when no field set is passed. `null` and `undefined` both mean
  all-on.
- **Allowlist semantics:** a key present and `true` → enabled; a key present and
  `false` → disabled; a key _absent from a non-null object_ → disabled. `{}` →
  everything disabled.
- Each field disabled → that metric absent from output.
- All fields in a group disabled → the group's line omitted entirely
  (e.g. TL + IF + TRIMP off → no `🔋 Load:` line).
- All fields disabled → returns `""`.
- Empty output → no `updateStravaActivity` call, no marker written (both the
  sync path and the coach-tool path).
- Unknown keys in stored JSONB → ignored, no throw.
- Preview action with zero synced activities → sample fallback.
- Preview action with an activity lacking `raw` → sample fallback.

## DoD

- [ ] Settings show a field checklist under the Strava section
- [ ] Preview renders against the user's most recent real activity
- [ ] Preview falls back to sample data when no activity has raw metrics
- [ ] Disabling every field in a group removes that group's line
- [ ] Disabling all fields skips the Strava write — no bare marker is ever published
- [ ] Users with no saved config get byte-identical v0.6 output
- [ ] Both the post-sync path and the coach tool honor the field set
- [ ] `📊 Recover` marker remains non-configurable

## Non-Goals (v0.6.1)

- **Reordering blocks / custom labels / custom emoji** — a "block builder" is a
  clean follow-up if the want materializes; the toggle config migrates into it
  without a rewrite. Nobody has asked yet.
- **Free-form template strings** (`{load}` placeholders) — would require a
  parser, validation, and conditional syntax to preserve null-omission.
  Without it, a null field yields dangling separators (`IF  | TRIMP 141`)
  published under the user's name, unretryable behind the skip marker. Rejected
  as strictly worse than the current output.
- **Per-sport field sets** — global set plus existing sport branch and
  null-omission already produce correct per-sport output.
- **Making the marker optional** — it is the idempotency key.
- **LLM-generated prose** — still a v0.6 non-goal; unchanged here.
